# ADR-0019 — the ONE way the iOS generated client authenticates: a custom `RequestBuilderFactory` injected into the generated config that routes EVERY generated request through the hand-written Core auth spine — one token source, no per-call header duplication

- **Status:** accepted   <!-- proposed | accepted | superseded | rejected -->
- **Date:** 2026-06-26
- **Supersedes:** —
- **Superseded by:** —
- **Applies to:** ios | mobile (the per-host generated-client contract) | cross-cutting
- **Refines / consumes:** **ADR-0013** D4 (the hand-written auth/session/header spine — `HeaderAdapter`,
  `actor SessionRefresher`, `TokenStore`, `DeviceIdProvider`, the no-Bearer-on-anon allow-list, the
  single-token-source invariant) + D5 (the swift5/urlsession codegen boundary, auth excluded). ADR-0013
  fixed *that auth is hand-written and the business client is generated* but **did not pin how a generated
  call reaches the spine** — this ADR pins it. Consumes **ADR-0014** (iOS-16 floor; `async`/`await`, `actor`
  all ≤16 — the mechanism is floor-compatible) and **ADR-0011** D4 (`ApiResult`/`ApiError` — the repo still
  returns it; this decision is about the transport *under* the repo). Not a parity-of-UX call — **ADR-0018
  (Gate-DP) does not apply** (this is infra, not a screen).
- **Ticket:** surfaced by the T-0303 Understand pass (the Phase-1 partner proving vertical) · **Consumers:**
  T-0303 (first authed generated call) and **every later authed generated wave** (T-0307/0309/0310 partner,
  T-0312/0313/0314 customer) — they all copy this seam, not re-invent it.

> **The decision in one line.** The iOS generated business client (`PartnerDashboardAPI` &c.) authenticates
> **only** by having a **custom `RequestBuilderFactory` installed into the generated global config**
> (`CleansiaPartnerApiAPI.requestBuilderFactory`) whose `RequestBuilder` routes every generated request
> through the **already-built Core auth spine** — the `HeaderAdapter` for the Bearer + device + time-zone
> headers and the `actor SessionRefresher` for the single-flight 401→refresh→retry. There is **no second
> token source** and **no per-call header injection** anywhere in the app-side client wrappers. It ships
> **no feature code** — it is the transport contract every authed generated wave consumes; the concrete
> wiring is built in T-0303.

## Context

**The generated client cannot authenticate itself — verified in the on-disk swift5 client**
(`src/cleansia_ios/CleansiaPartnerApi`, gitignored, generated by openapi-generator per ADR-0013 D5):

- The generated APIs are **static** and read a **global** config object. Every endpoint builds its URL from
  `CleansiaPartnerApiAPI.basePath` and obtains its builder from `CleansiaPartnerApiAPI.requestBuilderFactory`
  (`PartnerDashboardAPI.swift:32,46,188,202`; the config is `APIs.swift:11-17`).
- The base `RequestBuilder.init` applies **only** `CleansiaPartnerApiAPI.customHeaders` — a **static**
  `[String: String]` (`APIs.swift:13,38`). It does **not** inject a Bearer, does **not** stamp
  `X-Device-Id`/`X-Device-Label`/`X-Time-Zone`, and has **no** 401→refresh→retry.
- **Every dashboard endpoint is generated with `requiresAuthentication: false`** (`PartnerDashboardAPI.swift:48,
  90,132,168,204,246,339`). So with no adapter the dashboard calls go out **tokenless** and **401**. (This is
  the swift5 generator's default for a spec whose security is applied out-of-band — the same reason Android
  hand-writes its interceptors; the flag being `false` is *expected*, not a spec bug.)

**The spine that already does all of this exists — for the hand-written calls only.** ADR-0013 D4's
auth/session/header layer landed in T-0300 (`CleansiaCore/Auth`, 68 tests green): `HeaderAdapter` (the
injected `AnonymousAllowList`, the Bearer-iff-not-anon rule, `X-Device-Id` from the single `DeviceIdProvider`,
`X-Device-Label`, `X-Time-Zone`), the `actor SessionRefresher` (single-flight 401-refresh, refresh-token
replaced every refresh, `ForcedSignOut` + `clearAll()` on failure), the Keychain `TokenStore`, and the
two-session `AuthApiClient` (authed + separate no-auth refresh session). The **single-token-source invariant**
(ADR-0013 reviewer checks #3/#5/#6) holds for the hand-written `/api/Auth/*` calls. **The generated client is
simply not wired into it.** The gap is the bridge between the generated transport and that spine.

**Why this is ADR-worthy and not a one-line ticket note.** T-0303 is the *proving vertical*; the way it
authenticates a generated call is **copied by every later authed wave** (partner orders/pay/profile, the whole
customer app). If the first wave injects headers per-call, that pattern metastasises into ~10 tickets and the
single-token-source invariant is no longer enforceable by a grep. The decision picks the **one seam** so the
codebase converges. This is exactly the "one canonical way to do X" the pattern-evolution loop exists to fix
*before* it is folklore.

This is **one decision** — "how the iOS generated client authenticates" — because the seam choice and the
single-source invariant are inseparable: choosing the `requestBuilderFactory` seam *is* what preserves one
token source (all generated traffic funnels through the one spine), and rejecting per-call injection *is* what
keeps the single-flight refresh correct (per-call injection cannot coalesce concurrent 401s cleanly). The
*implementation* is the T-0303 wiring.

## Decision

> **Contract principle.** The generated business client authenticates **only** via a Core-spine-backed
> `RequestBuilderFactory` installed into the generated global config — **never** by a second token source and
> **never** by per-call header duplication in the app-side client wrapper.

### D1 — The seam: a custom `RequestBuilderFactory` → a `RequestBuilder` subclass that delegates to the spine

The composition root (`AppContainer`, ADR-0013 D3) installs, at app launch, a **custom
`RequestBuilderFactory`** into `CleansiaPartnerApiAPI.requestBuilderFactory` (and `basePath` from
`AppConfig.apiBaseURL`). The factory returns a `RequestBuilder<T>` subclass (a `CoreSpineRequestBuilder`)
that, for **every** generated request:

1. **Stamps headers via the Core `HeaderAdapter`** — overriding the generated builder's request-construction
   seam (`createURLRequest`, which is `open`) so the `HeaderAdapter` adds `X-Device-Id` (from the **one**
   `DeviceIdProvider`), `X-Device-Label`, `X-Time-Zone`, and the **Bearer iff the path is not on the injected
   `AnonymousAllowList` and a non-expired token is present** — the **same** `HeaderAdapter`/`AnonymousAllowList`
   the hand-written calls use. The generated `requiresAuthentication: false` flag is therefore **irrelevant** —
   authentication is decided by the allow-list, not the per-endpoint flag, exactly as Android decides it in
   `AuthInterceptor` regardless of any generated annotation.
2. **Runs the single-flight 401→refresh→retry via the Core `actor SessionRefresher`** — wrapping the generated
   `execute()` (which is `open`) so a 401 awaits the one in-flight refresh, retries once with the rotated token,
   and on a rejected/expired refresh emits `ForcedSignOut(.sessionExpired)` + `clearAll()`. Concurrent 401s
   across N generated calls coalesce into **one** refresh — the same `SessionRefresher` instance the
   hand-written path uses.

Because the builder delegates to the **already-injected** spine collaborators (one `HeaderAdapter`, one
`SessionRefresher`, one `TokenStore`, one `DeviceIdProvider`), the **token has exactly one source** and the
device-id exactly one source — the generated and hand-written traffic share the same spine, not two copies.

### D2 — What is forbidden (so the reviewer can enforce it by grep)

- **No second token source.** The app-side generated-client wrapper (`PartnerMobileApiClient` or equivalent)
  **must not** read `TokenStore` and set `Authorization` itself, **must not** write
  `CleansiaPartnerApiAPI.customHeaders` with a Bearer, and **must not** hold its own copy of the access token.
  The Bearer is set in **exactly one place**: the `HeaderAdapter`, reached through the factory.
- **No per-call header injection.** No `.addHeader("Authorization", …)` / device-header line at any generated
  call site or in the wrapper. Headers are stamped by the `HeaderAdapter` for **all** calls uniformly.
- **No bespoke 401 handling at the call site.** A call site **must not** catch a 401 and refresh itself — that
  would duplicate (and race) the `SessionRefresher`'s single-flight. The retry is the factory's, once, for all.
- **The generated client is still never hand-edited** (ADR-0013 reviewer check #1). The factory and the
  `RequestBuilder` subclass live in **hand-written `CleansiaCore`/app code** that *installs into* the
  generated config — they are not edits to generated files.

### D3 — Why the `requestBuilderFactory` seam (and not the alternatives)

- **vs. per-call header injection in the app client wrapper (the brief's rejected alternative).** It
  duplicates the header logic at every generated call site, drifts the moment one site is missed (a tokenless
  401 or a missing `X-Device-Id` that silently breaks remote-revoke), and **cannot** do single-flight refresh
  cleanly — each call site would have to coordinate a shared refresh, which is precisely what the
  `SessionRefresher` actor centralises. It also creates a **second** place the token is read, breaking the
  single-source invariant the whole spine is built to hold. Rejected.
- **vs. mutating the static `customHeaders` with the Bearer.** `customHeaders` is **static and global**; a
  Bearer written there is stamped on **every** request including the anon `/api/Auth/*` paths (some reject a
  Bearer), is not refreshed on a 401, and races across requests as the token rotates. It models neither the
  allow-list nor the refresh. Rejected.
- **vs. a `URLProtocol`/`URLSession`-delegate interceptor under the generated session.** It would work for
  header stamping but the generated builder owns its own `URLSession` (`createURLSession`, `open`) and a
  global `URLProtocol` would also catch the hand-written auth calls' sessions — entangling the no-auth refresh
  session that must **not** be intercepted (it would loop). The factory seam is **scoped to the generated
  client only** and reuses the spine's existing no-auth refresh session for the retry. Rejected as the wrong
  scope.
- **vs. regenerating with `requiresAuthentication: true` + the generated `bearerToken` hook.** The swift5
  generator's built-in auth hook injects only a Bearer — not the three device/time-zone headers, not the
  allow-list nuance, not the single-flight refresh — so it would still need the spine for everything else, and
  it couples auth to a codegen-config change (a spec/owner-regen dependency) for a behaviour the hand-written
  spine already owns. The factory seam needs **zero** codegen change. Rejected.

The chosen seam uses only the **`open`/overridable extension points the generator intentionally exposes**
(`requestBuilderFactory` is a `public static var`; `RequestBuilder`, `execute()`, `createURLRequest`,
`createURLSession` are all `open`) — so it survives regeneration untouched (it lives in hand-written code) and
adds **no** new dependency.

### D4 — Composition with the existing reviewer checks (this is additive, not a re-decision)

This decision **uses** ADR-0013 D4's spine; it does not change it. It makes the spine's invariants reach the
generated traffic:
- ADR-0013 **#3 (X-Device-Id single source)** — now also enforced for generated calls: the `HeaderAdapter`
  reached through the factory uses the **one** `DeviceIdProvider`; no generated wrapper resolves a device id.
- ADR-0013 **#4 (anon allow-list)** — now the allow-list governs generated calls too; the generated
  `requiresAuthentication` flag is **not** the authority.
- ADR-0013 **#5 (refresh token replaced every refresh)** / **#6 (single no-auth session + single-flight)** —
  the generated 401 path goes through the **same** `SessionRefresher`/no-auth session; no second refresh path.
- ADR-0013 **#2 (auth NOT generated)** — unchanged: `/api/Auth/*` stay hand-written; the factory authenticates
  the **business** client only.
- ADR-0013 **#1 (no hand-edited generated client)** — unchanged: the factory is hand-written code installed
  into the generated config, not an edit of generated files.
- ADR-0011 **D4 (one `ApiResult`)** — unchanged: the repo still wraps the generated call in `safeApiCall` and
  returns `ApiResult<T>`; this decision is about the transport beneath that.

### D5 — Scope guard

This ADR decides **how the iOS generated business client authenticates** (the transport seam + the
single-source/no-duplication rules). It does **not**: write the T-0303 wiring (that is the ticket); change the
hand-written auth spine (ADR-0013 D4 stands); change the codegen config or require a regen; touch the customer
host's allow-list (already host-specific in `HeaderAdapter`, ADR-0013 D4.4 / `patterns-mobile`); or decide any
screen/UX (Gate-DP N/A). A future need to authenticate a **non**-generated transport, or to add a header the
spine doesn't model, is revisited against ADR-0013 D4 (a header-parity-spec edit), not here.

## Alternatives considered

- **Per-call header injection in `PartnerMobileApiClient`.** Rejected (D3) — duplicates header logic across
  every generated call site, drifts silently (a missed site = tokenless 401 or a broken `X-Device-Id` →
  remote-revoke fails), **cannot** do single-flight refresh cleanly, and creates a **second** token source
  that breaks the single-source invariant. This is the brief's explicitly-considered-and-rejected option; the
  panel confirms the rejection.
- **Bearer in the static `customHeaders`.** Rejected (D3) — global/static, stamps anon paths, no refresh, races
  on rotation; models neither the allow-list nor the 401 path.
- **A global `URLProtocol` / session-delegate interceptor.** Rejected (D3) — wrong scope; would also catch the
  no-auth refresh session and risk a refresh loop. The factory seam is scoped to the generated client and
  reuses the spine's no-auth session.
- **Regenerate with `requiresAuthentication: true` + the generated bearer hook.** Rejected (D3) — injects only
  a Bearer (not the device/time-zone headers, allow-list, or single-flight refresh), still needs the spine for
  the rest, and couples auth to a codegen change. The factory needs zero regen.
- **Two parallel auth paths (generated has its own, hand-written has its own).** Rejected — it is the
  single-source invariant violation by construction; concurrent 401s on the two paths would refresh twice and
  could trip `RefreshTokenReused` self-revoke. One spine, one refresh, one token.

## Consequences

**Cheaper / safer:**
- **One token source, enforced by construction.** All generated and hand-written traffic funnels through the
  one spine — the single-source invariant (ADR-0013 #3/#5/#6) is preserved for the generated client with zero
  new token-reading code, and a reviewer can verify it by grep (D2).
- **The first authed wave sets the one pattern for every later wave.** T-0307/0309/0310/0312/0313/0314 install
  the same factory once per host and write **no** auth code — they just call the generated APIs and wrap in
  `safeApiCall`. The seam does not metastasise.
- **Survives regeneration untouched.** The factory + builder subclass are hand-written code installed into the
  generated config via `open`/overridable points; a spec regen (`manual_step: mobile-spec-regen`) replaces the
  generated files and the auth wiring is unaffected (ADR-0013 #1 holds).
- **Floor-compatible (ADR-0014).** `async`/`await`, `actor`, `URLSession` are all ≤ iOS 16; no iOS-17 API.

**More expensive (new obligations):**
- **The factory/builder subclass is hand-written and must be tested** — the single-flight, allow-list, and
  device-id behaviours that already have spine tests (T-0300) must be re-asserted **through the generated
  client** (the test contract below), because the *generated path* is the new surface.
- **A second factory is installed per host** (partner now, customer later) — each app's `AppContainer`
  installs its own factory into its own generated config (`CleansiaPartnerApiAPI` /
  `CleansiaCustomerApiAPI`), with its own host-specific `AnonymousAllowList`. Recorded so the customer wave
  copies, not re-derives.
- **A new reviewer check (#13-gen) is added** (below) — the gate that a generated call authenticates only via
  the factory, with no second token source and no per-call header duplication.

## How a reviewer verifies compliance

**The new check (composes with ADR-0013 #1–#10 + ADR-0014 #11–#13):**

**#13-gen — generated client authenticates ONLY via the Core-spine-backed `RequestBuilderFactory`.**
1. **The factory is installed.** The `AppContainer` sets `CleansiaPartnerApiAPI.requestBuilderFactory` (and the
   customer equivalent) to the custom Core-spine-backed factory at launch; the generated default
   (`URLSessionRequestBuilderFactory`) is **not** the one used at runtime.
2. **No second token source.** Grep the app-side generated-client wrapper + every generated call site: **no**
   read of `TokenStore`/access token, **no** `Authorization`/`Bearer` string, **no** write of a Bearer into
   `customHeaders`, **no** device-header (`X-Device-Id`/`X-Device-Label`/`X-Time-Zone`) set outside the
   `HeaderAdapter`. The Bearer and the device id are each set in **exactly one** place (the `HeaderAdapter` /
   the `DeviceIdProvider`), reached through the factory. A second source is a **blocking** finding (it breaks
   the single-source invariant and, for the device id, remote-revoke).
3. **No per-call header duplication / no per-call 401 handling.** No `.addHeader(…)` for auth/device headers at
   a call site or in the wrapper; no call site catches a 401 and refreshes itself. The 401→refresh→retry is the
   factory's single-flight, once, for all.
4. **The generated `requiresAuthentication: false` flag is NOT the authority.** Authentication is decided by the
   injected `AnonymousAllowList` in the `HeaderAdapter` (so a business endpoint generated with the flag `false`
   still gets a Bearer; an anon `/api/Auth/*` path still gets none).

**Test contract (T-0303 + every authed wave):**
- **TC-IOS-GEN-AUTH:** a generated business call (`PartnerDashboardAPI.dashboardGetStats`) goes out **with** the
  Bearer + `X-Device-Id`/`X-Device-Label`/`X-Time-Zone`, *despite* the generated `requiresAuthentication: false`
  — proving the factory, not the flag, governs.
- **TC-IOS-GEN-401:** N concurrent generated calls that 401 trigger **exactly one** refresh (the same
  `SessionRefresher` single-flight as TC-IOS-AUTH-401); queued callers retry with the rotated token without a
  second network refresh; a rejected refresh emits `ForcedSignOut(.sessionExpired)` + clears the caches.
- **TC-IOS-GEN-DEVICEID:** the `X-Device-Id` on a generated call equals the `Device/Register` deviceId for the
  same install (one source) — the generated path does not introduce a second id.

## Roles affected

The factory/builder is a thin **transport adapter** over the existing spine — it introduces **no new
responsibility**, it **routes generated requests into existing roles**. CRC (added with the T-0303 wiring, the
RDD home being the planned `agents/knowledge/roles/ios-*` cards from ADR-0013 that land with their code):

- **`ios-generated-client-auth-bridge`** (new, thin) — `CoreSpineRequestBuilder` + its `RequestBuilderFactory`:
  *responsibility:* route every **generated** request through the existing `HeaderAdapter` (headers + Bearer)
  and `actor SessionRefresher` (single-flight 401-refresh), installed into the generated global config.
  *Collaborators:* `HeaderAdapter`, `SessionRefresher`, the generated `RequestBuilder`/config.
  *Does NOT know:* the token value (that is `TokenStore`'s, reached only via the spine), the device id (that is
  `DeviceIdProvider`'s), any business payload, or how a screen renders. **If this role ever reads the token or
  resolves a device id directly, the responsibility is wrong — it must delegate to the spine** (the exact
  invariant this ADR protects).

The existing `ios-header-adapter`, `ios-session-refresher`, `ios-device-id-provider` CRCs (ADR-0013 §Roles
affected) are **unchanged** — they gain the generated client as an additional caller, not a new behaviour.

**Catalog edits (same change, per the pattern-evolution loop):** `agents/knowledge/patterns-mobile.md` iOS
section gains the **generated-client-auth-via-factory** rule (the one way; per-call injection and a second token
