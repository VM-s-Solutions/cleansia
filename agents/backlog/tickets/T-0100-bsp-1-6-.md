## Context

The authorization seam is **fail-OPEN** and the hosts diverge. ADR-0001 (ADR-AUTHZ, accepted
2026-06-01) is the frozen contract this ticket implements — specifically decisions **D1** (fail-closed
default + `AssertComplete`), **D2** (the complete, frozen permission map), and **D4** (one shared
per-host registration). It resolves audit findings **BSP-1, BSP-6, BSP-7, BSP-8**.

The proven holes, verified against the real code:

- **BSP-1 / BSP-6 — fail-OPEN default.** `PolicyBuilder.cs:204-205` is
  `Map.GetValueOrDefault(permission, PhysicalPolicy.Authenticated)`. Any `Policy.*` constant absent
  from `Map` silently degrades to **any logged-in user, any role, any host** with no compile-time or
  startup signal.
- **BSP-1 (proven instance) — the entire payroll family is unmapped.** None of the 21 pay-family
  constants declared in `Policy.cs:75-97` (`CanViewPagedInvoices`, `CanViewPeriodPays`,
  `CanCalculateOrderPay`, `CanGenerateInvoice`, `CanApproveInvoice`, `CanMarkInvoicePaid`,
  `CanCancelInvoice`, `CanClosePayPeriod`, `CanViewPayPeriods/PayPeriod`, `CanCreate/Update/Open/
  DeletePayPeriod`, `CanView/Create/Update/DeletePayConfig(s)`) appears in `PolicyBuilder.Map`. They
  all fall through to `Authenticated` → **a customer's JWT can read and mutate payroll.**
- **BSP-7 — per-host registration gaps + a self-inflicted duplicate.** Each of the 5 host
  `ServiceExtensions.cs` hand-copies its `AddUserAuthorization` body, and `CustomerOnly` is registered
  only on the two customer hosts; `AddJwt` also registers a *second* policy literally named
  `"AdminOnly"` via `RequireRole` — two registrations, last-wins, silent.
- **BSP-8** — documents the JWT key/audience trust boundary (D5); only the documentation/contract part
  is in scope here (no behavioral change ships in this ticket; refresh `RequiredProfile` per host is
  D5 §3 → a separate ticket).

This is ADR-0001's `T-AUTHZ-1+2` (fail-closed + AssertComplete + map fill) **merged with** the
`AddCleansiaAuthorization` extraction the TICKET-MAP assigns to BSP-1(+6) (D4.1 shared registration +
D4.2 startup assertions). Per ADR rollout ordering, **D1 (deny default + AssertComplete) and D2 (map
fill) MUST ship together** — shipping the assertion before the payroll rows are mapped would make
every host fail to **boot**.

## Acceptance criteria

- [ ] **AC1 (BSP-1 fail-closed default)** — Given a permission string with no entry in
  `PolicyBuilder.Map`, When `ToPhysicalPolicy()` is called, Then it returns `PhysicalPolicy.Deny`
  (not `Authenticated`). Evidence: replaces `PolicyBuilder.cs:204-205`; unit test
  `"SomeUnknownPermission".ToPhysicalPolicy() == PhysicalPolicy.Deny` (verification #3).
- [ ] **AC2 (Deny sentinel + registration)** — Given any host, When authorization is registered, Then
  a new `PhysicalPolicy.Deny = "Deny"` constant exists and is registered as
  `p => p.RequireAssertion(_ => false)`, so an unmapped permission yields 403. Evidence:
  `PhysicalPolicy.cs` gains `Deny`; shared registration includes it on every host.
- [ ] **AC3 (BSP-1 payroll map fill)** — Given the 21 pay-family `Policy.*` constants
  (`Policy.cs:75-97`), When `PolicyBuilder.Map` is built, Then each is mapped exactly per the D2 table:
  `CanViewPagedInvoices` + `CanViewPeriodPays` → **EmployeeOrAdmin [OWN-DATA]**;
  `CanViewPayPeriods` + `CanViewPayPeriod` → **EmployeeOrAdmin**; every `Pay*` mutation, all `CanView/
  Create/Update/DeletePayConfig(s)`, `CanCalculateOrderPay`, `CanGenerate/Approve/MarkInvoicePaid/
  CancelInvoice`, `CanClosePayPeriod`, and all `Create/Update/Open/DeletePayPeriod` → **AdminOnly**.
  Evidence: rows added to `PolicyBuilder.cs` `Map`; frozen-map snapshot test (#4) matches the D2 table.
- [ ] **AC4 (BSP-6 dispute message split — map rows only)** — Given the overloaded
  `CanRespondToDispute = Authenticated` (`PolicyBuilder.cs:76`, fail-open), When the map is corrected,
  Then a new `Policy.CanAddDisputeMessage` constant exists mapped to **CustomerOnly [OWN-DATA]** and
  `CanRespondToDispute` is mapped to **AdminOnly**. Evidence: new `Policy.*` constant + two map rows.
  (The endpoint move Partner→Admin and the `IsStaffMessage` handler derivation are **out of scope** —
  SEC-DSP-01, the next ticket in this serialization cluster.)
- [ ] **AC5 (AssertComplete)** — Given a developer adds a `Policy.*` constant and forgets to map it,
  When any host boots (and in CI), Then boot fails with an `InvalidOperationException` listing the
  missing permission(s). Evidence: `PolicyBuilder.AssertComplete()` implemented exactly per ADR D1.2
  (declared = reflected `Policy.*` consts; `missing = declared − Map.Keys − AnonymousAllowList`;
  orphan-key check; allow-list∩Map contradiction check) + the frozen `AnonymousAllowList` (7 entries
  per D1.2). Unit test mirrors it (verification #1).
- [ ] **AC6 (allow-list exhaustiveness)** — Given every host's `[AllowAnonymous]` actions carrying a
  `[Permission(...)]`, When the allow-list test (#1b) scans them, Then the collected permission set
  **equals** `AnonymousAllowList` (neither short → boot bricks on a legit anon route, nor long → a
  real route silently un-gated). Evidence: test in `Cleansia.Tests`.
- [ ] **AC7 (BSP-7 single shared registration)** — Given the 5 host `ServiceExtensions.cs`
  `AddUserAuthorization` bodies and the legacy `AddJwt "AdminOnly"` registration, When this ticket
  lands, Then they are replaced by ONE `AddCleansiaAuthorization(this IServiceCollection,
  IConfiguration)` in `Cleansia.Config` that sets the default policy, registers all physical policies
  (`Authenticated`, `CustomerOnly`, `EmployeeOrAdmin`, `AdminOnly`, `OwnerOrElevated`, `Deny`)
  **exactly once**, and every host calls it. The duplicate `AddJwt "AdminOnly"` is **deleted** (grep
  confirms zero bare `[Authorize(Policy="AdminOnly")]` consumers). Evidence: 5 host files call the
  shared extension; `AddJwt` registers no authorization policy. **NOTE:** the `OwnerOrElevated`
  resolver *redefinition* and the `GetUser` handler ownership check are **out of scope** — IDA-SEC-04
  (next-but-one in this cluster). This ticket registers `OwnerOrElevated` with its current behavior so
  the shared extension is complete; IDA-SEC-04 rebases the resolver onto it.
- [ ] **AC8 (BSP-7 registration presence + semantics assertion)** — Given a built host container, When
  the shared `IStartupFilter` runs (before first request), Then it asserts (a) every `PhysicalPolicy.*`
  except `Anonymous` resolves non-null, and (b) **behavior**: a constructed admin principal passes
  `AdminOnly` & fails `CustomerOnly`; a constructed customer fails `AdminOnly` & passes `CustomerOnly`;
  an employee passes `EmployeeOrAdmin` — else boot fails. Evidence: `AddCleansiaAuthorization`
  registers the filter; `AssertComplete` runs at filter construction, `AssertAllRegistered` runs in the
  filter's `Configure` where `ApplicationServices` is available (per D1.2 / D4.2 wiring). Per-host
  test (#2).
- [ ] **AC9 (CustomerOnly now resolves on every host)** — Given a `CustomerOnly`-gated endpoint routed
  onto the Admin or Partner host, When an admin calls it, Then it resolves and correctly **denies**
  (admin is not a customer) instead of throwing an unresolved-policy error. Evidence: covered by AC8's
  presence assertion + per-host test.
- [ ] **AC10 (no fail-open regression test)** — Given the test suite, When run, Then there is no path
  by which `ToPhysicalPolicy` returns `Authenticated` for an absent key. Evidence: AC1 test asserts
  `Deny`; the frozen-map snapshot (#4) pins every real mapping.

## Implementation notes

**Serialization cluster (TICKET-MAP):** this is the head of the `PolicyBuilder.cs` / `Policy.cs`
cluster — **BSP-1(+6) → IDA-SEC-04 → SEC-DSP-01 → SEC-DSP-02 → SEC-EMP-01** — which must run strictly
serial (each adds map rows / a `Policy.*` const the next must see). This ticket also **owns** the
shared `AddCleansiaAuthorization` extraction touching all 5 host `ServiceExtensions.cs` (per ADR-0001
D4) — that host-registration surface is serialized through this single ticket. Do **not** run any other
cluster member concurrently.

**Governing ADR:** ADR-0001 §D1, §D2 (frozen map table at `adr/0001-authorization-model.md` lines
202-309, Notes A/B/C), §D4. The map is the contract; changing a row's *semantics* later requires a
superseding ADR (additive rows update the snapshot in-PR).

**Files (file:line where known):**
- `src/Cleansia.Core.AppServices/Authentication/PolicyBuilder.cs:6-205` — the `Map` + the
  `ToPhysicalPolicy` default (the hole at `:204-205`); add `AssertComplete()` + `AnonymousAllowList`.
- `src/Cleansia.Core.AppServices/Authentication/PhysicalPolicy.cs:3-11` — add `Deny`.
- `src/Cleansia.Core.AppServices/Authentication/Policy.cs:99-106` — add `CanAddDisputeMessage`
  (pay-family already declared at `:75-97`).
- `src/Cleansia.Config/Services/ServiceExtensions.cs` — new shared `AddCleansiaAuthorization` +
  `IStartupFilter` (D4.1/D4.2).
- The 5 host registrations to consolidate/delete-duplicate: `src/Cleansia.Web.Admin/Extensions/
  ServiceExtensions.cs`, `…Web.Partner…`, `…Web.Customer…`, `…Web.Mobile.Partner…`,
  `…Web.Mobile.Customer…` — each currently has its own `AddUserAuthorization` + `AddJwt "AdminOnly"`.
- Tests in `Cleansia.Tests` (#1 completeness, #1b allow-list exhaustiveness, #2 per-host presence+
  semantics, #3 no-fail-open, #4 frozen-map snapshot).

**TEST-FIRST (per `agents/knowledge/testing.md`):** authorization is pure logic + a startup invariant —
