## Context
Finding **IA-1** (CRITICAL, `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:1014-1020`):
`CreateAdminUser` double-hashes the password, so **any admin created through this endpoint cannot
log in** — silently, with no error at creation time. They would need a password reset to ever
authenticate.

Root cause: the handler hashes the raw password explicitly at
`src/Cleansia.Core.AppServices/Features/AdminUsers/CreateAdminUser.cs:74`
(`var hashedPassword = command.Password.HashAndSaltPassword();`) and passes the already-hashed value
into `User.CreateWithPassword(...)`. The EF write-side value converter
`src/Cleansia.Infra.Database/Converters/PasswordConverter.cs:6` then calls `HashAndSaltPassword()`
**again** on persist (wired at `src/Cleansia.Infra.Database/EntityConfigurations/UserEntityConfiguration.cs:14-16`).
The stored value is therefore `hash(hash(password))`, which `VerifyPassword(rawPassword, stored)`
can never match.

The correct, already-shipping pattern passes the **raw** password to `User.CreateWithPassword` and
lets the converter hash exactly once — see `Features/Auth/Register.cs:77` and
`Features/Auth/RegisterEmployee.cs:71`. This ticket aligns `CreateAdminUser` with that contract.

This is a fix ticket: the hole is closed and a regression test proves it.

## Acceptance criteria
- [ ] **AC1 — Round-trip login works.** Given an admin is created via `CreateAdminUser.Command` with
  a raw password `P`, When the user entity is persisted through the EF `PasswordConverter`, Then
  `P.VerifyPassword(user.Password)` returns `true` (the stored value is hashed exactly once, not
  twice).
- [ ] **AC2 — Handler no longer pre-hashes.** Given the `CreateAdminUser.Handler`, When it builds the
  user, Then it passes `command.Password` (raw) into `User.CreateWithPassword(...)` and contains no
  call to `HashAndSaltPassword()` — consistent with `Register.cs:77` / `RegisterEmployee.cs:71`.
- [ ] **AC3 — Regression test (test-first).** A unit test is added under `Cleansia.Tests` for the
  `CreateAdminUser` feature that fails against the current double-hash code and passes after the fix:
  it asserts `rawPassword.VerifyPassword(createdUser.Password)` is `true` after a converter
  round-trip (the verifier is `PasswordExtensions.VerifyPassword`,
  `src/Cleansia.Core.Domain/Extensions/PasswordExtensions.cs:27`). The test also guards the negative:
  a wrong password verifies `false`.
- [ ] **AC4 — No other path regresses.** Given the existing `Register` / `RegisterEmployee` paths
  (which already pass raw passwords), When this change lands, Then their hashing behaviour is
  unchanged (single hash via converter) and existing auth tests stay green.

## Implementation notes
- **The one-line fix:** delete `var hashedPassword = command.Password.HashAndSaltPassword();` at
  `CreateAdminUser.cs:74` and pass `password: command.Password` (raw) into `User.CreateWithPassword`
  at `CreateAdminUser.cs:76-81`. The converter at `PasswordConverter.cs:6` performs the single hash.
- **Built TEST-FIRST** per `agents/knowledge/testing.md`: write the failing round-trip/verify test
  first (it must fail on the current `hash(hash(...))` code), then apply the handler fix to make it
  green. Pair: the regression test lands in the same merge as the fix.
- **Serialization cluster:** none. `CreateAdminUser.cs` is **not** part of any shared-file cluster in
  `agents/backlog/TICKET-MAP.md` (the `CreateOrder.cs` cluster is unrelated). Safe to run
  concurrently with other Wave-0 tickets that don't touch this file.
- **ADRs:** none in force for this change. ADR-0001 (authorization), ADR-0002 (outbox dispatch) and
  ADR-0003 (rate limiting) do not govern password hashing — `adrs: []`.
- **security_touching: true** — credential-handling path; Security gate is mandatory even though the
  diff is small (verify single-hash invariant, no plaintext logging introduced).
- **manual_steps: none** — no schema change, no DTO/endpoint contract change → no ef-migration, no
  nswag-regen.
