## Context

Audit finding **LG-SEC-01** (critical, S7 — non-atomic side-effect /
idempotency): a single-use promo code can be redeemed **past its per-user cap**
via a concurrency race. Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:2873`
and `AUDIT-2026-06-01-findings.json:1503`.

The per-user cap is enforced with a **check-then-act** read in
`PromoCodeService.ApplyAsync` — `CountForUserAndCodeAsync(userId, promoCode.Id)`
at `src/Cleansia.Core.AppServices/Services/PromoCodeService.cs:122-124`, compared
against `promoCode.MaxRedemptionsPerUser` — and again in `PreviewAsync`
(`:49-51`). Two concurrent Apply calls for the same `(user, code)` both read a
count below the cap, both pass, and both insert a `PromoCodeRedemption` row →
the cap is breached and the discount is granted more times than allowed (real
money side-effect). The denormalised global counter `CurrentRedemptionsCount`
(`PromoCode.cs:42`, bumped by `IncrementRedemptions`, `PromoCode.cs:149-153`) is
likewise read-then-incremented non-atomically against `GlobalMaxRedemptions`
(`ValidateAvailability`, `PromoCodeService.cs:170-174`), so the global cap has
the same race.

The existing schema has a **non-unique** `(PromoCodeId, UserId)` index for the
count query (`PromoCodeRedemptionEntityConfiguration.cs:54-55`) and a **unique**
`OrderId` index that only collapses double-fire *for the same order* — it does
not stop two distinct orders from over-redeeming a one-shot code.

This is a hardening of an implemented flow, not a gap. The fix moves cap
enforcement from app-layer reads to **DB-enforced atomicity**, per security law
**S7** (`agents/knowledge/security-rules.md:100`) and the tenant-scoped
unique-index rule **S8** (`security-rules.md:114` — unique indexes on
tenant-scoped tables are `(TenantId, X)`).

## Acceptance criteria

- [ ] **AC1 (per-user cap is DB-enforced)** — Given a promo code with
  `MaxRedemptionsPerUser = 1` already redeemed once by user U, When U attempts a
  second redemption (`ApplyAsync`) on a different order, Then a unique constraint
  on the redemption slot rejects the second insert and the call returns
  `PromoCodeApplyResult(false, …, PromoCodeError.PerUserLimitReached)` — never a
  successful second discount and never an unhandled `DbUpdateException`.
- [ ] **AC2 (concurrent per-user race closed)** — Given two `ApplyAsync` calls
  for the same `(user, code)` running concurrently against a `MaxRedemptionsPerUser = 1`
  code, When both pass the in-memory count check and both attempt to insert, Then
  exactly **one** redemption row exists afterward and the loser returns
  `PerUserLimitReached` (the `DbUpdateException` from the unique violation is
  caught and mapped). A concurrency test proves only one row lands.
- [ ] **AC3 (global cap is atomic)** — Given a code with
  `GlobalMaxRedemptions = N` and `CurrentRedemptionsCount = N`, When a redemption
  is attempted, Then the conditional increment
  (`UPDATE … SET CurrentRedemptionsCount = CurrentRedemptionsCount + 1 WHERE … AND CurrentRedemptionsCount < GlobalMaxRedemptions`)
  affects **0 rows** and the call returns
  `PromoCodeError.GlobalLimitReached` with no redemption row inserted; under two
  concurrent attempts at `N-1` only one succeeds.
- [ ] **AC4 (multi-use codes still work)** — Given a code with
  `MaxRedemptionsPerUser = M (M > 1)`, When user U redeems it M times on M
  distinct orders, Then all M succeed and the M+1th is rejected with
  `PerUserLimitReached` (the slot uniqueness keys on a per-slot ordinal, not a
  flat `(code, user)` pair, so legitimate repeat use is preserved).
- [ ] **AC5 (tenant-scoped unique index)** — The new unique index is
  `(TenantId, PromoCodeId, UserId, <slotOrdinal>)` per S8 (tenant-scoped table),
  added via an EF entity-config change in
  `PromoCodeRedemptionEntityConfiguration.cs`; the migration is **flagged as a
  `manual_step` (`ef-migration`)** and NOT run by any agent.
- [ ] **AC6 (idempotency on order unchanged)** — Given an order that already has
  a redemption row, When `ApplyAsync` is called again for that order, Then it
  still short-circuits to the existing result (`PromoCodeService.cs:92-96`) — the
  existing per-order idempotency behavior is preserved.
- [ ] **AC7 (tests prove the hole is closed)** — Per `T-0127` (TC-IDEMP-0), the
  test cases for AC1–AC4 land **with** this fix in the same merge, written
  test-first (red → green), each AC item mapping to a named case.

## Owner decision (2026-06-02) — race-fix approach (BINDING; supersedes the catch-DbUpdateException note)

**Context discovered during scoping:** `ApplyAsync` is called from **inside the `CreateOrder` handler**
(`CreateOrder.cs:391`), whose redemption row is added to the **same transaction** committed at the end by
`UnitOfWorkPipelineBehavior` (`:20`). Promo persistence there is deliberately **fail-soft** —
`CreateOrder.cs:384-403`: *"Failure logs but doesn't roll back — the customer already paid and the promo
just doesn't get tracked."* Therefore the ticket's original "wrap the insert + map the unique-index
`DbUpdateException` to `PerUserLimitReached`" approach is **WRONG here**: the insert isn't flushed until
the pipeline `CommitAsync`, *after* `ApplyAsync` returns — a unique violation at that point would fail the
whole commit and **roll back the paid order**, which is worse than the bug.

**Approach (owner-chosen): atomic conditional-UPDATE for BOTH caps; unique index = defense-in-depth only.**
- **Global cap:** replace read-then-`IncrementRedemptions` with an atomic conditional update
  `UPDATE PromoCodes SET CurrentRedemptionsCount = CurrentRedemptionsCount + 1 WHERE Id = @id AND
  (GlobalMaxRedemptions IS NULL OR CurrentRedemptionsCount < GlobalMaxRedemptions)`; **0 rows affected →
  `GlobalLimitReached`** (no row inserted). Implement via EF `ExecuteUpdateAsync`.
- **Per-user cap:** reserve the user's slot atomically (e.g. an `ExecuteUpdateAsync`/atomic-insert guard or
  a conditional counter) that **returns a result, not an exception** — a race loser gets a clean
  `PerUserLimitReached`. The `(TenantId, PromoCodeId, UserId, SlotOrdinal)` unique index from AC5 stays as
  a **backstop only**; it must NOT be the hot path that surfaces an exception at commit.
- **Deliberate UoW exception:** `ExecuteUpdateAsync` issues SQL immediately and **bypasses the UoW
  pipeline** (it is not change-tracked, auto-commits). That is intentional and REQUIRED for atomicity here
  — the dev must call it out and the reviewer must confirm it is a knowing, documented exception to the
  "never commit outside the pipeline" rule (not an accidental extra commit). It does NOT roll the order
  back: the conditional update either reserves the slot or returns "limit reached"; the order commit is
  untouched. The fail-soft contract is preserved (a `false` result just logs, as today).
- **SlotOrdinal** still added (AC4): one-shot codes (default `MaxRedemptionsPerUser = 1`) get hard
  single-use; `M > 1` codes remain valid. Derive the ordinal from the atomic reservation, NOT from a
  pre-read count (a pre-read ordinal would let two concurrent `M>1` redemptions collide on the same
  ordinal and falsely reject the loser — avoid that).
