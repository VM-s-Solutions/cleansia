## Context
Critical money-correctness blocker (S7 idempotency), ship-gate item from the 2026-06-01 audit.

`CreateMembershipSubscription.Handler` is non-idempotent on its confirmed-subscribe branch. In the
`PaymentMethodConfirmed == true` path
(`src/Cleansia.Core.AppServices/Features/Memberships/CreateMembershipSubscription.cs:79-94`) the
handler generates a **fresh** `Guid.NewGuid()` per call (`:84`) and passes it as the Stripe
idempotency key to `stripeClient.CreateSubscriptionAsync` (`:85-86`). The only guard against a
duplicate is `GetActiveForUserAsync(...) != null` at `:57`, evaluated *before* the Stripe call and
*before* commit — a textbook TOCTOU race. Two concurrent confirm calls (a double-tap on the mobile
PaymentSheet "confirm") both see "no active membership", both call Stripe with **different**
idempotency keys, and create **two** subscriptions + two `UserMembership` rows. The customer is
billed twice. `UserMembership.StripeSubscriptionId` is unique, but the two subs have different ids,
so the DB index does not catch it.

The `subscription.created` webhook already reconciles by `StripeSubscriptionId`, so the durable
collapse point must be a **client-supplied idempotency token** reused as the Stripe idempotency key,
making a retried/double-tapped confirm replay the same Stripe call instead of creating a second
subscription.

## Acceptance criteria
- [ ] **AC1 (the hole is closed — backend)** — Given a user with no active membership, When two
  `PaymentMethodConfirmed == true` `CreateMembershipSubscription.Command`s for that user carry the
  **same** client idempotency token and run concurrently, Then exactly **one** Stripe subscription is
  created and exactly **one** `UserMembership` row is persisted; the second request resolves to the
  same membership (or a deterministic `BusinessResult.Failure` / no-op) and does **not** call
  `CreateSubscriptionAsync` with a second distinct key.
- [ ] **AC2 (idempotency key is the client token, not `Guid.NewGuid()`)** — Given a confirmed
  subscribe request, When the handler calls `stripeClient.CreateSubscriptionAsync(..., idempotencyAttemptId, ...)`
  (`IStripeClient.cs:84`), Then the `idempotencyAttemptId` is **derived from the client-supplied
  token** (so a retry of the same logical attempt replays), not a per-call `Guid.NewGuid()`. Re-subscribe
  after a *real* cancellation still works because it is a new logical attempt with a new token.
- [ ] **AC3 (a test proves it — TDD)** — A unit test in `Cleansia.Tests` (paired ticket T-0127 /
  TC-IDEMP-0) drives the handler twice with the same token against a mocked `IStripeClient` and
  asserts the subscription side-effect fires **once** (the second call does not double the effect),
  and a sibling test asserts a *different* token produces a new subscription. Written **red first**
  per `agents/knowledge/testing.md` (must-cover #6, S7 idempotency). The handler is changed only after
  the failing test exists.
- [ ] **AC4 (mobile sends a stable token across retries)** — Given the customer-app subscribe flow,
  When `confirmSubscribe(planCode)` is retried (PaymentSheet returns Completed twice, a network retry,
  or a double-tap that survives the `_submitting` guard), Then the same client idempotency token is
  sent on every confirm attempt for that logical subscribe, so the backend collapses them. The token
  is generated once per subscribe attempt (e.g. at `startSubscribe` / Phase-1) and reused on Phase-2
  confirm. Files: `MembershipViewModel.confirmSubscribe` (`MembershipViewModel.kt:111`),
  `MembershipRepository.subscribePhase2` (`MembershipRepository.kt:66`).
- [ ] **AC5 (no regression in the unconfirmed Phase-1 path)** — Given `PaymentMethodConfirmed == false`,
  When the handler runs, Then it still returns the SetupIntent client secret + ephemeral key with no
  Stripe subscription created (existing behavior at `CreateMembershipSubscription.cs:107-114`
  preserved).

## Owner decision (2026-06-02) — idempotency-token source + local-row collapse (BINDING)
- **Client-supplied token on the Command.** Add `IdempotencyToken` (nullable `string?`) to
  `CreateMembershipSubscription.Command`. The MOBILE app generates it ONCE per subscribe attempt (at
  Phase-1 / `startSubscribe`, `MembershipViewModel.kt:83`) and resends the SAME value on every Phase-2
  confirm retry (`subscribePhase2`, `MembershipRepository.kt:66`). The backend uses it (derived) as the
  Stripe `idempotencyAttemptId`, replacing `Guid.NewGuid()` at `CreateMembershipSubscription.cs:84`.
- **Nullable + safe fallback.** Make the field nullable so the web consumer and any not-yet-updated
  caller still work: if `IdempotencyToken` is absent, the backend derives a deterministic key from stable
  inputs (e.g. `userId + planCode` for the current no-active-membership attempt) — defense in depth — but
  the client token is the authoritative collapse point. A genuine re-subscribe after cancellation gets a
  NEW token (new logical attempt) so it is not blocked (AC2).
- **Local-row collapse (the T-0110 lesson, applied):** with the same Stripe key, two concurrent confirms
  receive the SAME `subscription.SubscriptionId`; both attempt `UserMembership.Create(stripeSubscriptionId:
  same)` + `Add`, and the second hits the **existing unique index** on `UserMembership.StripeSubscriptionId`
  (`UserMembershipEntityConfiguration.cs:56-57`). UNLIKE T-0110, this handler is a **top-level command**
  (not inside the paid-order txn), so a failed second commit only fails THIS request — acceptable. But do
  NOT let a raw `DbUpdateException` surface: handle it cleanly (re-check active membership after the Stripe
  call / on unique-violation, map to `BusinessResult.Failure(MembershipAlreadyActive)` or resolve to the
  existing membership) so the loser gets a deterministic result, not a 500.

> **CORRECTION (round 2, 2026-06-02) — a post-Stripe re-check ALONE is insufficient.** Round-1
> implemented the collapse as only a `GetActiveForUserAsync` re-check after the Stripe call. The reviewer
> (CHANGES REQUESTED) correctly proved the residual race: if the loser re-checks BEFORE the winner's
> pipeline `CommitAsync` makes its row visible, the loser sees null, `Add`s a duplicate, and the loser's
> commit throws `DbUpdateException` on the unique index → 500. The re-check narrows but does NOT close the
> window (the round-1 unit tests passed only because they ran serialized, modeling confirm-after-commit,
> not true concurrency). **Required (round 2): close the window, not just narrow it.** Catch the
> unique-violation `DbUpdateException` at the boundary that owns the write — flush the membership insert
> inside the handler (its own `SaveChangesAsync` in a try/catch) so the violation is caught HERE (not at
> the pipeline), and on a `StripeSubscriptionId` unique violation resolve via the existing
> `GetByStripeSubscriptionIdAsync(subscription.SubscriptionId)` → return
> `MembershipAlreadyActive`/resolve. Keep the re-check as a fast path. A test must model the
> winner-not-yet-committed concurrent path (or the deferred true-parallel proof must be explicitly added
> to T-0127 with the residual window flagged — but the owner decision asks for the window CLOSED).
- **Contract change → nswag-regen (owner).** The `Command` gains a field → the generated mobile + web
  clients drift. Flag `manual_step: nswag-regen` to the owner; HOLD mobile/web wiring of the new field
  behind the owner's regen (mobile dev writes the Kotlin token plumbing against the intended contract; if
  the generated client isn't regenerated yet, note it). No EF migration (no schema change).
- **ADR-0002** (idempotent-consumer contract) governs: the second attempt must not double the effect.

## Review
**Round 1 — Security PASS; Reviewer CHANGES REQUESTED (2026-06-02).** The Stripe-side double-charge fix
was correct (client token → derived Stripe idempotency key, `Guid.NewGuid()` removed; mobile token
minted once at Phase-1 and replayed on every Phase-2 retry; Phase-1 unchanged; re-subscribe works with a
new token). Security PASS (token can't hijack another user's sub — handler still scopes to session
userId). BUT the reviewer correctly blocked: the local-row collapse relied ONLY on a post-Stripe
`GetActiveForUserAsync` re-check, leaving an open race — a loser re-checking BEFORE the winner's pipeline
`CommitAsync` would `Add` a duplicate and throw `DbUpdateException` on the `StripeSubscriptionId` unique
index → 500. The round-1 unit tests passed only because they ran serialized (confirm-after-commit), not
true concurrency. (Orchestrator independently validated the block against the real code before accepting.)

**Round 2 — Reviewer APPROVED (2026-06-02).** The window is closed at the write boundary: the handler now
flushes the insert in-handler via `userMembershipRepository.CommitAsync` inside
`catch (DbUpdateException) when (IsUniqueViolation(ex))` (Postgres 23505, detected provider-agnostically by
walking the inner-exception chain and duck-typing `SqlState` — AppServices has no Npgsql ref), resolving
the winner via the existing `GetByStripeSubscriptionIdAsync` and returning `MembershipAlreadyActive`. The
pipeline's final `CommitAsync` is a safe no-op (the row is `Unchanged` after the in-handler flush; same
scoped DbContext/UoW). A new test models the pre-winner-commit race (loser re-check null + flush raises a
simulated 23505 → resolves to `MembershipAlreadyActive`, not a throw), written test-first (RED on the
round-1 handler). DeriveStripeAttemptId / Command field / Phase-1 / mobile unchanged.
