## Context

Finding **LG-SEC-06** (major / S7 — `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:2908-2913`,
`AUDIT-2026-06-01-findings.json:1576`): admin manual grant/revoke of loyalty points is **fully
non-idempotent**, so a double-submit, proxy retry, or Stripe-style network retry **double-grants real
points**. Points are economically meaningful — they drive tier discounts via
`LoyaltyService.ResolveTierDiscountForOrderAsync` — so this is a financial side-effect doubling.

Verified in code:
- `LoyaltyService.GrantPointsManuallyAsync` (`src/Cleansia.Core.AppServices/Services/LoyaltyService.cs:181-211`)
  and `RevokePointsManuallyAsync` (`:213-247`) only run their idempotency guard when `orderId` is
  non-null (`:199-207`, `:227-235`); the comment at `:194-198` even states manual grants are
  "intentional duplicates by definition."
- The admin commands pass `orderId: null` (`GrantPointsManually.cs:58`, `RevokePointsManually.cs:57`),
  so the guard is **skipped by design** on the admin path.
- The endpoints `AdminLoyaltyController.cs:15-27` (grant) and `:29-41` (revoke) carry **no
  client-supplied idempotency key and no `[EnableRateLimiting]`** (S5).
- The ledger entity `LoyaltyTransaction` (`src/Cleansia.Core.Domain/Loyalty/LoyaltyTransaction.cs:31`)
  has `OrderId` but **no idempotency-key field**; its EF config only indexes `(OrderId, Source)`
  (`LoyaltyTransactionEntityConfiguration.cs:54`) — there is no unique key to collapse a retry.

Finding's long-term-correct fix: require a client-generated `requestId` on the grant/revoke command,
persist it on the `LoyaltyTransaction`, unique-index it so a retry collapses, and add a narrow
per-admin rate-limit window. This is the canonical **S7 idempotency** shape — see ADR-0002 (the
side-effect dispatch / idempotency contract, which adopts S7 and names the deterministic-key + unique
backstop pattern) and ADR-0003 (partitioned rate limiting) for the limiter window.

## Acceptance criteria

- [ ] **AC1** — Given a valid admin grant with a `requestId`, When the same command (same
  `requestId`, `UserId`, `Points`) is submitted **twice**, Then exactly **one** `LoyaltyTransaction`
  is appended and `LoyaltyAccount.LifetimePoints` increases by `Points` **once** (not twice). The
  second call returns success (same `Response`) without a second side effect. (Mirror for revoke:
  one negative ledger row, `LifetimePoints` reduced once.)
- [ ] **AC2** — Given two concurrent identical grant requests (same `requestId`), When both reach the
  database, Then the unique index on the new idempotency-key column rejects the second insert and the
  effect lands **exactly once** (the loser collapses, not errors to the admin).
- [ ] **AC3** — The `requestId` is a **required** field on `GrantPointsManually.Command` and
  `RevokePointsManually.Command`, validated (NotEmpty + bounded length) via the existing FluentValidation
  validators (which already cover `UserId`/`Points`/`Reason` — `GrantPointsManually.cs:20-44`), returning
  the standard `BusinessErrorMessage` code on a missing/oversized value.
- [ ] **AC4** — `LoyaltyTransaction` gains an idempotency-key property (persisted, max-length bounded),
  set on the manual grant/revoke path through `LoyaltyAccount.GrantPoints`/`RevokePoints`
  (`LoyaltyAccount.cs:54,78`) / `LoyaltyTransaction.Create` (`LoyaltyTransaction.cs:42`), with a **unique
  index** in `LoyaltyTransactionEntityConfiguration.cs` (filtered so existing rows with a null key are
  unaffected, preserving the order-driven path and back-compat).
- [ ] **AC5** — Both endpoints (`AdminLoyaltyController.cs:15,29`) carry `[EnableRateLimiting]` with a
  narrow per-admin window per ADR-0003 (S5), so brute retries are throttled at the edge.
- [ ] **AC6** — A unit/integration test in `Cleansia.Tests` proves AC1+AC2 ("safe to run twice" — the
  second invocation does not double points) and the validator test proves AC3. Tests are written
  **test-first** (red → green; visible in the diff/commit order per `agents/knowledge/testing.md`).
  The paired suite is **T-0127 (TC-IDEMP-0)** — same merge.

## MANUAL_STEP (owner) — schema delta (ef-migration, owner-only)

This change adds **one nullable column + one partial (filtered) tenant-scoped unique index** to
`LoyaltyTransactions`. It is **additive and nullable → S9-safe** (no backfill, no default needed).

**Schema delta:**
- New column: `"IdempotencyKey" varchar(80) NULL` on table `LoyaltyTransactions`
  (from `LoyaltyTransaction.IdempotencyKey` — `[MaxLength(80)] public string? IdempotencyKey`).
- New index: **partial unique, tenant-scoped** —
  `CREATE UNIQUE INDEX "IX_LoyaltyTransactions_TenantId_IdempotencyKey"
   ON "LoyaltyTransactions" ("TenantId", "IdempotencyKey")
   WHERE "IdempotencyKey" IS NOT NULL;`
  (EF emits this from `HasIndex(t => new { t.TenantId, t.IdempotencyKey }).IsUnique()
  .HasFilter("\"IdempotencyKey\" IS NOT NULL")`.)
- Tenant-scoped: **YES** — leading `TenantId` per S8. The token is client-generated and is unique
  **per tenant**, not globally.
- Existing `(OrderId, Source)` index is unchanged; existing NULL-key rows (order-driven / referral,
  every tenant) are unaffected by the filter.

**Single-tenant / back-compat caveat (TenantId == NULL):** Postgres treats NULLs in a UNIQUE index as
DISTINCT by default, so in single-tenant deployments (`TenantId` NULL) two NULL-tenant rows with the
same key are NOT rejected by the index — the *concurrent-race* DB backstop is relaxed there (the
serial-replay fast-path read still collapses). This matches the exact tradeoff every other
tenant-scoped unique index in the repo already makes; no one-off `NULLS NOT DISTINCT` was added for
consistency. If single-tenant concurrency hardening is ever needed, the upgrade is to add
`.AreNullsDistinct(false)` (EF/Npgsql → `NULLS NOT DISTINCT`, Postgres 15+) to this index — call it
out then, not now.
