## Context
Finding **LG-SEC-05** (`audits/AUDIT-2026-06-01-slice-reports.md:2901-2906`,
`audits/AUDIT-2026-06-01-findings.json:1560-1574`; pulled forward from Wave 2 into the Wave-0
anonymous-tenant bundle per `plan-corrections.md:56` as the more-severe sibling of BSP-9).

`MembershipPlan` implements `ITenantEntity` (`src/Cleansia.Core.Domain/Memberships/MembershipPlan.cs:24`),
so the EF Core global query filter applies. The `GetPlans` read is served on an `[AllowAnonymous]`
route (`src/Cleansia.Web.Customer/Controllers/MembershipController.cs:58-65`, and the identical mobile
host `src/Cleansia.Web.Mobile.Customer/Controllers/MembershipController.cs:58-65`). With no JWT there
is no `tenant_id` claim, so `TenantProvider.GetCurrentTenantId()` returns null
(`src/Cleansia.Infra.Database/TenantProvider.cs:12-20`) and the filter collapses to `TenantId == null`.

Concrete impact: in any multi-tenant deployment the anonymous marketing page returns **only**
null-tenant plans (wrong/empty), while the authenticated subscribe flow sees the correct tenant's
plans — "plans missing on the public page". Inverse footgun: a future `TenantId == null` "shared"
plan row becomes visible to **every** tenant's anonymous page. Per S3, anonymous routes must not serve
tenant-scoped data except behind a deliberate resolution mechanism. `LoyaltyTierConfig`
(`src/Cleansia.Core.Domain/Loyalty/LoyaltyTierConfig.cs:10`) is `ITenantEntity` with the same shape
and is in scope if it is reachable from any anonymous read path.

This is a **GAP** finding: it needs an explicit product/architecture decision (per ADR-0001's
anonymous-route discipline), not just a code patch. The two options:
1. Resolve the tenant from the request host/subdomain **before** the query on the anonymous plans
   route (`TenantProvider` already exposes `SetTenantOverride`/`ClearTenantOverride`,
   `TenantProvider.cs:22-30` — the seam exists), or
2. Treat `MembershipPlan`/`LoyaltyTierConfig` as true platform config (drop `ITenantEntity`) and
   document why.

## Acceptance criteria
- [ ] **AC1 (decision recorded)** — Given the two options above, When the team picks one, Then the
      chosen model is documented in ADR-0001 (or a one-line ADR addendum it references) so plan
      visibility no longer depends on whether a JWT happens to be present. Architect decides before
      implementation (see Implementation notes).
- [ ] **AC2 (anonymous read returns the right tenant's plans)** — Given a multi-tenant deployment with
      tenant-scoped `MembershipPlan` rows, When an anonymous client calls `GET .../Membership/GetPlans`
      on the Customer host (and the Mobile.Customer host), Then it returns the plans for the resolved
      tenant — not the `TenantId == null` slice — and the result matches what the authenticated
      subscribe flow sees for the same tenant.
- [ ] **AC3 (no cross-tenant over-broad read)** — Given a `TenantId == null` "shared" plan row exists,
      When an anonymous client of a specific tenant calls `GetPlans`, Then that null-tenant row is NOT
      silently served to every tenant (the inverse footgun is closed by the chosen model).
- [ ] **AC4 (loyalty parity)** — Given `LoyaltyTierConfig` is reachable from any anonymous read path,
      When that path is exercised anonymously, Then the same tenant-resolution rule applies (or the
      ticket records that no anonymous loyalty read path exists and the entity stays untouched).
- [ ] **AC5 (test proves the hole is closed)** — Given the host-test harness from T-0100/T-AUTHZ-0,
      When the integration test boots the real host and calls `GetPlans` anonymously with two tenants'
      data seeded, Then the test asserts each tenant resolves to its own plans and the null-tenant
      footgun does not leak. The test is written **test-first** (red before green; visible in the
      diff/commit order and noted in the status log).

## Architect panel ruling (2026-06-02) — BINDING (AC1 satisfied; pending owner approval to implement)
Panel = author + anonymous-route challenger + forward-compat challenger + lead. All premises verified
against real code. **Decision: Option A — `MembershipPlan` becomes platform config (drop `ITenantEntity`).**
Recorded as **ADR-0001 Addendum A1** (in `adr/0001-authorization-model.md`).

- **Why A over B:** Currency/Language/Country are already platform config (exact precedent); no inbound
  host/subdomain tenant-resolution exists today (Option B would require *building* it for one anonymous read
  of a zero-row table); the app is effectively single-tenant; the cost folds into the owner's Initial regen.
  Decisive security point (challenger): on an unauthenticated route Option B's host→tenant resolution keys off
  the client-controlled `Host` header — removing the tenant dimension (A) is strictly safer than header-trusting
  resolution (S3).
- **A closes all three bugs:** plans-missing-on-public-page; the null-tenant-leak footgun (the dimension is gone);
  **and a third bug the panel surfaced** — `GetByCodeAsync` carries no tenant logic, so an anonymous-subscribe →
  tenant-overridden Stripe webhook could mismatch the plan under per-tenant scoping → **AC6 (write-side parity)**.
- **Scope ruling:** the four sibling anon catalogs (`Service`/`Package`/`Extra`/`ServiceCity`) are the same class
  but are **BSP-9's** scope, NOT folded into T-0113 (avoids the double-fix collision the ticket itself warns of);
  they inherit Addendum A1 by cross-reference. `LoyaltyTierConfig` untouched (no anonymous read path — AC4).
- **manual_steps:** now **`[ef-migration]`** (Option A drops the `TenantId` column + reindex; folds into the
  owner's regenerated Initial — zero-row table, no incremental migration). No NSwag change.
- **Forward-compat (recorded so we're not trapped):** the return to per-tenant is symmetric (re-add
  `ITenantEntity` + inverse index migration) and the only costly piece (spoof-resistant resolution middleware) is
  unavoidable the day ANY anonymous catalog goes per-tenant — deferring it loses nothing.

**Implementation contract** (per Addendum A1): `MembershipPlan.cs:24` drop `ITenantEntity`;
`MembershipPlanEntityConfiguration.cs:55-56` index `(TenantId,Code)`→`(Code)`; no DbContext/handler/repo change;
ef-migration folds into Initial; ACs = AC2 read parity, AC3 no-footgun, AC6 write-side parity, AC5 host-boot test,
+ mandatory structural test (`MembershipPlan` is not `ITenantEntity`), AC4 LoyaltyTierConfig untouched.

## MANUAL_STEP (owner) — schema delta (db agent, T-0113)

- **Removed:** `builder.HasIndex(p => new { p.TenantId, p.Code }).IsUnique();` (composite unique `IX_MembershipPlans_TenantId_Code`).

- **DROP INDEX** `IX_MembershipPlans_TenantId_Code` (the old composite unique).
