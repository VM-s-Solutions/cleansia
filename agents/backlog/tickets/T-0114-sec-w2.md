## Context
Audit finding **SEC-W2** (major, S7 idempotency / double financial side-effect + data-integrity).
Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:3069-3075`,
`agents/backlog/audits/AUDIT-2026-06-01-findings.json:308-322`.

The web Checkout flow only creates the Stripe Session; the local `UserMembership` row is created
**exclusively** by the `customer.subscription.created` webhook in
`ProvisionFromCreatedEventAsync` (`StripeSubscriptionWebhookHandler.cs:102-167`). That provisioning
path calls `UserMembership.Create` (line 154) **without ever calling `GetActiveForUserAsync`** ‚Äî
unlike the request path which does guard (`CreateMembershipCheckoutSession.cs:54-59`). The domain
itself documents the "at most one active membership per user" invariant as enforced *in handler code,
not by a unique index* (`UserMembership.cs:12-15, 84-88`), and the EF config has **no** unique index
backstop ‚Äî only a non-unique composite `(UserId, Status)` index
(`UserMembershipEntityConfiguration.cs:59-61`).

A user who already has an active membership and reaches Stripe a second time (stale tab, Dashboard,
or two near-simultaneous checkouts ‚Äî the request-side guard only blocks the *session-creation* call,
not Stripe-side reality) gets a **second active row** with no guard. Benefit usage tracking, renewal
reminders, and the pricing pipeline all assume a single active row; duplicates double-grant benefits
and create reconciliation drift against Stripe. This is the "consumer asserts before acting"
contract from **ADR-0002** (D2 idempotent consumer) applied to the membership-provisioning side
effect, with a DB-level dedup backstop.

## Acceptance criteria
- [ ] **AC1** ‚Äî Given a user with an existing `Active` `UserMembership`, When a
  `customer.subscription.created` webhook arrives for that same user (new `subscriptionId`), Then
  `ProvisionFromCreatedEventAsync` calls `GetActiveForUserAsync(userId)` first, finds the active row,
  and does **not** call `UserMembership.Create`/`Add` ‚Äî it logs a reconcile/skip warning and returns
  without inserting a second active row.
- [ ] **AC2** ‚Äî Given a user with **no** active membership, When the `customer.subscription.created`
  webhook arrives, Then exactly one `UserMembership` is created (existing happy path unchanged) and
  the event is stamped as processed.
- [ ] **AC3** ‚Äî A **filtered unique index** on `(TenantId, UserId)` WHERE `Status = Active` exists on
  `UserMemberships` (added in `UserMembershipEntityConfiguration.cs`), so a second active row is
  rejected by the database (PG 23505) even if the application check is bypassed by a race. The
  index is *filtered* so `Cancelled`/expired + new `Active` is still permitted (per the
  `UserMembership.cs:12-15` comment that a naive full unique index would wrongly block cancelled+new).
- [ ] **AC4** (manual_step) ‚Äî An EF migration adding the filtered unique index is authored and flagged
  for the owner to apply; dependent work holds until the owner confirms. The PM does **not** run it.
- [ ] **AC5** ‚Äî Tests prove the hole is closed (TC-IDEMP-0 / `pairs_with: T-0127`, same merge):
  (a) a unit/handler test that a second `subscription.created` for an already-active user inserts no
  second active row; (b) a test that the first provisioning for a clean user still creates exactly
  one row; (c) a DB-constraint test that a direct attempt to persist a second `Active` row for the
  same `(TenantId, UserId)` raises a unique-violation.

**What changed (entity config):** added one PostgreSQL **partial (filtered) UNIQUE index** to
`UserMemberships`:

- Columns: `(TenantId, UserId)`
- `IsUnique()`
- `HasFilter("\"Status\" = 1")` ‚Äî i.e. `WHERE "Status" = 1`, where `1` is the int-stored value of
  `MembershipStatus.Active` (the enum is mapped as its underlying int ‚Äî there is **no**
  `.HasConversion<string>()` on `UserMembership.Status`, so the column is an integer and `Active = 1`).

**Why filtered + tenant-scoped:**
- Filtered to `Status = Active` so a `Cancelled`/expired membership **plus** a new `Active`
  subscription is still permitted (a full `(TenantId, UserId)` unique would wrongly block the
  legitimate re-subscribe-after-cancel case ‚Äî see `UserMembership.cs` doc).
- `(TenantId, UserId)` not `(UserId)` per **S8** (`UserMembership` is `ITenantEntity`).
- Postgres default = NULLS DISTINCT, so two **NULL-TenantId** active rows for the same user are
  NOT rejected by this index (single-tenant mode). There the app-level `GetActiveForUserAsync`
  assert + the existing `StripeSubscriptionId` unique index are the guards; this index hardens
  multi-tenant mode. Same tradeoff as `LoyaltyTransaction (TenantId, IdempotencyKey)` (T-0112) and
  `PromoCode`/`ReferralCode (TenantId, Code)`.

**Folds into the owner's next `Initial` regen.** The current pending Initial migration
(`src/Cleansia.Infra.Database/Migrations/20260602105850_Initial.*`) has not shipped; this filtered
unique index bundles into that same regen alongside the other pending Wave-0 deltas:
**T-0110 (`PromoCodeRedemption` SlotOrdinal), T-0112 (`LoyaltyTransaction` IdempotencyKey),
T-0113 (`MembershipPlan`)**.

**Expected emitted SQL (for review):**
```sql
CREATE UNIQUE INDEX "IX_UserMemberships_TenantId_UserId"
    ON "UserMemberships" ("TenantId", "UserId")
    WHERE "Status" = 1;
```
