## Context
Audit finding **SEC-W3** (major, security / S5 — unthrottled side-effecting endpoint).
Source: `agents/backlog/audits/AUDIT-2026-06-01-findings.json:324-337` and ADR-0003 **D5.2**
(`agents/backlog/adr/0003-partitioned-rate-limiting.md:321-339`).

The Stripe webhook action is `[AllowAnonymous]` with **no** `[EnableRateLimiting]` on all three
hosts that expose it (verified):
- `Cleansia.Web.Customer/Controllers/PaymentController.cs:43-47`
- `Cleansia.Web.Partner/Controllers/PaymentController.cs:13-17`
- `Cleansia.Web.Mobile.Customer/Controllers/PaymentController.cs:43-47`

Because the route is anonymous and unthrottled, an attacker who cannot forge a valid signature still
gets an unbounded free DB round-trip per request (read body → ConstructEvent attempt → repo lookup
before rejection), and the signature is computed up to three times per request
(`HandlePaymentNotification.cs:48,62` + the handler at `:124`) — a cheap unauthenticated DoS amplifier
on the most sensitive endpoint in the system. S5 (`agents/knowledge/security-rules.md §S5`) lists
side-effecting mutations as requiring a limit; webhooks qualify.

ADR-0003 (the frozen rate-limiter contract) deliberately does **not** fold this in (D5.2): it makes
the partitioning machine *support* an independent per-IP webhook policy without coupling. This ticket
adds that **separate, third named policy** (`"webhook"`), partitioned **per source IP**, applied via
`[EnableRateLimiting("webhook")]` on the webhook actions only — outside `"auth"`/`"interactive"` so a
429 to Stripe never consumes (or is consumed by) the auth/interactive allowance, and a webhook flood
cannot starve real login/checkout traffic. **SEC-W3 owns the webhook limit + the Stripe-IP allow-list;
ADR-0003 owns the shared partitioning shape it slots into** (the `ClientIp(ctx)` resolver and the D3
`UseForwardedHeaders` fix are reused verbatim — both delivered by T-0115/BSP-4, hence the dependency).

Note: this ticket scopes the **rate-limit window** only. The "deduplicate the triple `ConstructEvent`"
half of the SEC-W3 fix text is the separate SEC-W4 ticket (see Out of scope).

## Acceptance criteria
- [x] **AC1** — A new named rate-limit policy `"webhook"` is registered **once** in the shared
  `CleansiaStartupBase.ConfigureServices` via `options.AddPolicy("webhook", …)` (alongside the existing
  `"auth"`/`"interactive"` from T-0115), built as a `RateLimitPartition.GetFixedWindowLimiter`
  partitioned **per source IP** using the canonical `ClientIp(ctx)` resolver delivered by T-0115
  (ADR-0003 D2/D5.2). No host re-registers it (zero `AddPolicy("webhook"` outside `CleansiaStartupBase`).
- [x] **AC2** — Given two distinct source IPs hitting `POST /api/Payment/webhook` in the same window,
  When IP A exhausts the `"webhook"` permit limit and sends one more, Then A's next request is `429`
  while IP B's first request in the same window is **not** `429` (per-IP isolation, proven by a host
  integration test on the T-0115 `WebApplicationFactory` harness; forwarded-headers configured for the
  test network).
- [x] **AC3** — The `"webhook"` policy is **independent** of `"auth"`/`"interactive"`: flooding the
  webhook to `429` consumes **no** `"auth"`/`"interactive"` allowance and vice-versa (ADR-0003 D5.2,
  verification #11). A test exercises one policy to rejection and asserts the others remain unaffected.
- [x] **AC4** — `[EnableRateLimiting("webhook")]` is applied to the `Webhook()` action on **all three**
  hosts that expose it — `Web.Customer/Controllers/PaymentController.cs:43-47`,
  `Web.Partner/Controllers/PaymentController.cs:13-17`, `Web.Mobile.Customer/Controllers/PaymentController.cs:43-47`
  — and on no other action. The `[AllowAnonymous]` attribute is preserved (Stripe is unauthenticated).
- [x] **AC5** — A rejected webhook request returns `429` with a `Retry-After` header (inherits the
  shared `OnRejected`/`RejectionStatusCode` from T-0115; `QueueLimit = 0`). The window length and
  per-IP permit limit are config-overridable (e.g. `RateLimiting:Webhook:*`); the partition key (the
  source IP) is **never logged above `Debug`** and the D8 rejection metric is dimensioned by policy
  name `"webhook"` only (S6, ADR-0003 D6/D8).
- [x] **AC6** (Stripe-IP allow-list) — The webhook limit is keyed by source IP per the finding
  ("Stripe publishes its egress ranges"). The allow-list / trusted-Stripe-range handling is
  **documented and config-driven** (env-injected, no secret); whether unknown source IPs are rejected
  outright or merely throttled is decided in the ticket and recorded. No hard-coded IP ranges.
  **DECISION (orchestrator-set contract):** key the webhook limit **per source IP** and **throttle
  EVERYONE**; do **NOT** hard-reject unknown (non-Stripe-range) IPs. Rationale: Stripe rotates egress
  ranges; a stale allow-list that hard-rejects would silently **DROP REAL webhooks** (lost
  payments/subscriptions) — worse than the DoS it prevents. The signature check is the real auth; the
  rate limit is the DoS protection. Any Stripe published-range is treated as **config-driven
  documentation** (env-injected, no hard-coded ranges, no secret); an optional future hard-reject lives
  behind a config toggle, **default OFF / throttle-only**. Implemented in `WebhookPartitionKey(ctx)` =
  `webhook:ip:{ClientIp(ctx)}` — no IP ranges in code.
- [x] **AC7** — Tests prove the hole is closed (TEST-FIRST, same merge): the per-IP isolation case
  (AC2), the policy-independence case (AC3), and a `429`-carries-`Retry-After` case (AC5). They fail
  against current (no-`"webhook"`-policy / no-attribute) code and pass after the fix.

  ticket adds the rate-limit window only (the CPU-amplification half is addressed there).
- The `"auth"`/`"interactive"` partition design, `UseForwardedHeaders` + startup guard, the `ClientIp`
  resolver, the cardinality cap, the pipeline reorder, and the integration harness — all owned by
  **T-0115 / BSP-4 (ADR-0003)** and consumed here, not re-defined.
- **BSP-4d** attribute-coverage of other uncovered money endpoints (Membership/Dispute/Recurring/Device
  controllers) — separate Wave-0 follow-up; this ticket covers the **webhook** action only.
- Changing the webhook signature-verification logic or the `HandlePaymentNotification` command shape.
- Account-lockout / per-code throttle (BSP-4b) and distributed-limiter concerns (residual, ADR-0003).
