## Context

Finding **F11** (`audits/AUDIT-2026-06-01-findings.json` id `F11`, severity minor, type bug): the
MediatR pipeline **commits the unit of work even when validation fails**.

In `FluentValidationExtensions.cs:13-14` the behaviors are registered UoW-first, so
`UnitOfWorkPipelineBehavior` is the **outer** behavior and wraps `ValidationPipelineBehavior` (the
inner one). `UnitOfWorkPipelineBehavior.cs:19-20` runs `next()` (validation + handler) and then calls
`await unitOfWork.CommitAsync(...)` **unconditionally** for any request whose type name ends in
`Command` (`:27`). When validation rejects a command, `ValidationPipelineBehavior.cs:50-53` returns a
failure `BusinessResult` **without** calling `next()` — so the handler never runs, but the outer UoW
behavior still reaches `CommitAsync`. Today this is usually harmless (no handler ran → no tracked
changes), but any validator or earlier-stage code that touches the `DbContext` would **persist on a
rejected command**, and timer Functions drive these commands frequently with no transaction boundary
above them.

This is the same pipeline surface the side-effect dispatch contract sits on, so the fix is governed by
**ADR-0002 (Side-effect dispatch contract) — decision D4**, which resolves F11 as the *foundation* of
the post-commit dispatch reorder (Dispatch → Validation → UnitOfWork → Handler). F11 must land first
because F2/SEC-W1 (T-0118 family) builds the `PostCommitDispatchBehavior` on top of this reordered
pipeline.

**Serialization cluster:** this ticket is in the `UnitOfWorkPipelineBehavior.cs` + queue-call-sites
cluster (TICKET-MAP §Shared-file map): **F11 → F2/SEC-W1 → F4 → F3**, strict order. F11 is first; do
not run any other ticket from this cluster concurrently.

## Acceptance criteria

- [ ] **AC1** — Given the MediatR behavior registration in
  `Cleansia.Config/Validation/FluentValidationExtensions.cs`, When the pipeline is resolved, Then the
  order (outer → inner) is **Validation → UnitOfWork → Handler** (per ADR-0002 D4): `ValidationPipelineBehavior`
  is registered before `UnitOfWorkPipelineBehavior` so it runs as the outer behavior. (If T-0118's
  `PostCommitDispatchBehavior` is not in this ticket's scope, only the Validation/UoW swap is required
  here; the dispatch behavior is added by the paired F2 ticket.)
- [ ] **AC2** — Given a `*Command` whose validator **fails**, When the command is sent through the
  pipeline, Then `IUnitOfWork.CommitAsync` is **never called** and the caller receives the validation
  failure `BusinessResult`. (Proven by the F11 regression test — see AC5.)
- [ ] **AC3** — Given a `*Command` whose validator **passes** and whose handler succeeds, When the
  command is sent, Then `CommitAsync` is called **exactly once** after the handler runs (happy path
  unchanged; no regression in committing valid commands).
- [ ] **AC4** — Defense-in-depth (ADR-0002 D4, C7 predicate alignment): Given the reordered pipeline,
  When `UnitOfWorkPipelineBehavior` runs, Then it commits **only if** `response is BusinessResult {
  IsSuccess: true }`, so a future re-swap of the registration order cannot resurrect F11. The UoW
  behavior keeps skipping non-`Command` requests (existing `IsNotCommand` guard) and paged queries
  returning `PagedData<T>` (not `BusinessResult`) are still skipped.
- [ ] **AC5** — Test-first (`knowledge/testing.md`): a pipeline-order unit test resolves
  `IEnumerable<IPipelineBehavior<,>>` and asserts the concrete order (ADR-0002 verification check #4),
  **and** an F11 regression test (ADR-0002 verification check #10) drives a command whose validator
  fails and asserts `CommitAsync` was **not** invoked (mock `IUnitOfWork`). Both tests are written
  before the production change and visibly fail for the right reason first (red → green). These cases
  belong to the paired test ticket **T-0127**; this ticket and T-0127 merge together.

## Implementation notes

- **Files:**
  - `src/Cleansia.Config/Validation/FluentValidationExtensions.cs:13-14` — swap the two
    `AddTransient(typeof(IPipelineBehavior<,>), …)` registrations so `ValidationPipelineBehavior<,>`
    is registered **before** `UnitOfWorkPipelineBehavior<,>` (registration order = outer → inner for
    MediatR), per ADR-0002 D4.
  - `src/Cleansia.Core.AppServices/Behaviors/UnitOfWorkPipelineBehavior.cs:19-20` — gate the
    `CommitAsync` with the response-success predicate (`response is BusinessResult { IsSuccess: true }`),
    keeping the existing `IsNotCommand` short-circuit (`:14`, `:25-28`).
  - `src/Cleansia.Core.AppServices/Behaviors/ValidationPipelineBehavior.cs:50-53` — no change needed;
    it already returns the failure `BusinessResult` without calling `next()`. Relied on by AC2.
- **Governing ADR:** ADR-0002 D4 (`agents/backlog/adr/0002-outbox-dispatch-contract.md`). The ADR's
  "How a reviewer verifies compliance" lists check #4 (pipeline-order test) and #10 (F11 regression)
  as the gate for this change.
- **Serialization:** part of the `UnitOfWorkPipelineBehavior.cs` cluster (F11 → F2/SEC-W1 → F4 → F3).
  Must complete and merge before F2/SEC-W1 starts editing the same file to add the dispatch behavior.
- **TEST-FIRST** per `agents/knowledge/testing.md` — pipeline wiring + commit-gating is logic, so the
  two tests (AC5) are written first and fail before the swap/predicate land. The tests live in
  `Cleansia.Tests`; pair ticket **T-0127** owns them and merges with this fix (TDD same-merge).
- `security_touching: false` — no auth/ownership surface; this is a transaction-boundary correctness fix.
