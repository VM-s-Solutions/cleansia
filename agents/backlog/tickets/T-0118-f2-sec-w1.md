## Context
Audit finding **F2 / SEC-W1** (major / S7 idempotency â€” double financial side-effect on the money path).
Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:3061-3067`,
`agents/backlog/audits/AUDIT-2026-06-01-findings.json:293-...` (id `SEC-W1`),
`agents/backlog/audits/AUDIT-2026-06-01-execution-plan.md:124`.

Every write that triggers an external effect enqueues to an Azure Storage Queue **before** the
`UnitOfWork` commit. `AzureStorageQueueClient.SendAsync` (`AzureStorageQueueClient.cs:14-27`) sends
**immediately** (`:26`) and is not enrolled in any EF transaction; `UnitOfWorkPipelineBehavior`
(`UnitOfWorkPipelineBehavior.cs:19-20`) runs the handler then commits **after**. So every
`queueClient.SendAsync` inside a handler puts a message on the wire before the commit, and that
message is **not rolled back** if the commit fails. Two live failure shapes:
- **Phantom side effect** â€” `CreateOrder.Handler` enqueues `generate-receipt` for cash orders
  (`CreateOrder.cs:376`) inside the handler; if the pipeline commit throws, a receipt message exists
  for an order that was never saved (`GenerateReceiptFunction.cs:48-52` then throws â†’ 5Ã— â†’ poison).
- **Webhook stamp/effect split** â€” `HandlePaymentNotification.HandleCompletedSession` enqueues
  `generate-receipt` and the `OrderConfirmed` push **before** the pipeline commits the
  `ProcessedStripeEvent` stamp + state change (`HandlePaymentNotification.cs:241-257`,
  `:144-159`). On commit failure (incl. the deliberate parallel-retry `DbUpdateException`) the stamp
  rolls back, Stripe retries, the handler re-runs, and a **second** receipt + push fire. The
  `ProcessedStripeEvent` dedup (`ProcessedStripeEventEntityConfiguration.cs:37-38`) is sound for the
  DB state but does **not** cover the queue side effects â€” the whole point of the dedup table is
  defeated for the effects that matter (slice report `:3065`, `:3113`).

Consumers are also not safe to run twice. `GenerateReceiptFunction` guards **receipt creation**
(`:59-70`, eligibility + `order.Receipt is not null`) but the **email re-send is uncovered** â€” the
email fires at `:95` before the `MarkEmailSent`+`CommitAsync` at `:97-99`, so a crash in that window
re-sends on redelivery. `SendPushNotificationFunction` has **no** dedup guard
(`SendPushNotificationFunction.cs:30-122`) and its commit is conditional (`:100-108`).

This ticket is the **Wave-0 tactical fix** under **ADR-0002** (the dispatch contract is frozen):
move the Bucket-A command-handler queue sends behind `IPendingDispatch.Enqueue` realized by a
`PostCommitDispatchBehavior` (D1), wrap every message in `QueueEnvelope<T>` with its deterministic
`MessageKey` (D2.1), and make the receipt consumer idempotent on its terminal effect (D2.2). It does
**not** build the durable outbox table â€” that is Wave-1 (F2-FULL) under the same seam.

## Acceptance criteria
- [ ] **AC1** (D1, post-commit dispatch) â€” Given a Bucket-A command handler that records intent via
  `IPendingDispatch.Enqueue(queue, message, messageKey)`, When the `UnitOfWork` commit **succeeds**,
  Then `PostCommitDispatchBehavior` (registered **outermost**) drains the buffer and calls
  `IQueueClient.SendAsync` **exactly once** per message, strictly **after** the commit. Dispatch
  fires only on `response is BusinessResult { IsSuccess: true }`.
- [ ] **AC2** (D1.2 / D4 ordering, the F2 fix) â€” Given the same handler, When the commit **throws**
  (the parallel-retry `DbUpdateException`, a transient PG error, or cancellation), Then `next()`
  propagates the exception, the dispatch guard is never reached, and **no** `IQueueClient.SendAsync`
  is called â€” the buffer is discarded. When **validation** fails, the handler never runs, no commit
  occurs, and nothing is dispatched. (Depends on T-0117/F11 reorder being merged.)
- [ ] **AC3** (D5 step 3, the call-site migration â€” happens once) â€” Every **Bucket-A** call site
  enumerated in ADR-0002 D5 (`CreateOrder.cs:376`; `CompleteOrder.cs:219,227,266`;
  `ConfirmRecurringOrder.cs:112,118`; `CancelOrder.cs:160`; `TakeOrder.cs:195`; `StartOrder.cs:137`;
  `NotifyOnTheWay.cs:103`; `AddDisputeMessage.cs:67`; `HandlePaymentNotification.cs:241,246,278`) no
  longer calls `IQueueClient.SendAsync` directly â€” each calls `pending.Enqueue(...)` with a
  `QueueEnvelope<T>`. A grep for `queueClient.SendAsync` / `IQueueClient` inside any `Features/**`
  command **Handler** returns **zero** matches outside the Bucket-B/C whitelist (sweeps,
  called-services, the fan-out Function) and the dispatch behavior (verification check #1).
- [ ] **AC4** (D2.1 deterministic keys) â€” Every migrated message is wrapped in `QueueEnvelope<T>`
  whose `MessageKey` follows the frozen formula (`receipt:{OrderId}`, `push:{UserId}:{EventKey}:{OrderId?}`,
  `pay:{OrderId}:{EmployeeId}`). Two invocations with the same domain inputs emit the **same**
  `MessageKey` â€” no `Guid.NewGuid()`/timestamp in a key (proven by TC-KEY-0 in T-0127).
- [ ] **AC5** (D2.1a dual-read) â€” Each touched consumer **dual-reads**: try-parse as
  `QueueEnvelope<T>`; on failure fall back to bare `T` and synthesize the deterministic key from the
  payload fields, so in-flight bare-payload messages at the deploy boundary do not dead-letter.
- [ ] **AC6** (D2.2, idempotent receipt consumer â€” the SEC-W1 terminal-effect close) â€” In
  `GenerateReceiptFunction`, the **email re-send hole (C6)** is closed: the email-sent state is
  marked in a commit that **precedes** the send (claim-first), so a redelivery does **not** re-send
  the order-receipt email. The existing receipt-creation guard (`:59-70`) is retained; the fiscal
  "target not found" path (`:48-52`) **stays transient / bounded-retry** (must NOT be reclassified to
  permanent-ack â€” D3.3 fiscal carve-out).
- [ ] **AC7** (tests prove the hole is closed â€” TC-IDEMP-0, `pairs_with: T-0127`, **same merge**) â€”
  - **TC-DISPATCH-0** (pipeline-integration): wiring `UnitOfWorkPipelineBehavior` +
    `PostCommitDispatchBehavior`, on commit-throw the buffer is **not** drained (no
    `IQueueClient.SendAsync`); on success it drains exactly once after commit; on validation failure
    it drains nothing.
  - **TC-IDEMP-0** (`generate-receipt`): invoke the consumer **twice with the same
    `QueueEnvelope<T>`** and assert `IReceiptService.GenerateReceiptAsync` **and**
    `IEmailService.SendOrderReceiptEmailAsync` are each invoked **exactly once** (the email is the
    terminal effect).
  - **TC-KEY-0**: each Bucket-A producing call site emits the same `MessageKey` for the same inputs.
- [ ] **AC8** (pipeline-order guard) â€” A unit test resolves `IEnumerable<IPipelineBehavior<,>>` and
  asserts the concrete order **PostCommitDispatch â†’ Validation â†’ UnitOfWork**, so a future re-swap
  cannot resurrect F11 or re-introduce before-commit dispatch (verification check #4).
