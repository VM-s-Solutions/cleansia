## Context

Finding **F4** (the ADR-0002 **C6 / D2.2 gap** on `generate-receipt`). The receipt consumer is
idempotent for *receipt creation* but **not for its terminal effect** (the email and the fiscal
sequence consumed during PDF generation). ADR-0002
(`agents/backlog/adr/0002-outbox-dispatch-contract.md`) is explicit: the eligibility +
`order.Receipt is not null` checks (`:269` / verification check #5) cover receipt creation, but
"**the email re-send is NOT covered**" and the contract "requires the *terminal* effect (email) to
be named in TC-IDEMP-0, not just receipt creation" (ADR-0002 D2.2 bullet 1, and the explicit
residual gap in Consequences: "*Receipt email re-send (C6) â€” must be closed in Wave-0*").

Verified against the real code â€” `GenerateReceiptFunction.Run`
(`src/Cleansia.Functions/Functions/GenerateReceiptFunction.cs`):
- The receipt-creation guard exists: eligibility at `:59-64` and `order.Receipt is not null` at
  `:66-70` â€” a redelivery does **not** create a second `OrderReceipt` row.
- **The hole:** on a redelivery where the receipt already exists but the *prior run crashed between
  the send and the commit*, the order is not short-circuited by the creation guard in that window,
  and even where it is, the email/fiscal path is reachable without an email-sent assertion. The email
  fires at `:95` (`SendOrderReceiptEmailAsync`) and `receipt.MarkEmailSent(emailMessageId)` +
  `unitOfWork.CommitAsync` happen at `:97-99`. **A crash between `:95` and `:99` re-sends the email on
  redelivery** (and re-runs `GenerateReceiptAsync` at `:72`, re-consuming a fiscal sequence number for
  BlockingOnline countries). Azure Storage queues are at-least-once (`host.json` `maxDequeueCount: 5`),
  so this redelivery is a live path, not hypothetical.

This ticket sits on top of **T-0118 (F2 â€” post-commit dispatch + idempotent receipt consumer,
ADR-0002 D1/D4)**, which introduces the `QueueEnvelope<T>` wrapper and the deterministic
`receipt:{OrderId}` `MessageKey` (ADR-0002 D2.1 frozen key table). T-0119 closes the **terminal-effect**
half of the contract on that foundation: make the email send and the fiscal-sequence consumption
**happen at most once per OrderId** via a **claim-first (guard-before-send)** ordering, per the
ADR's D2.2 guidance ("mark the email-sent state in a commit that **precedes** the send").

## Acceptance criteria

- [ ] **AC1 â€” Email at-most-once on redelivery.** Given an eligible `generate-receipt` message for an
  order whose receipt + email already completed successfully, When the consumer `Run` is invoked a
  **second time with the same `QueueEnvelope<T>`** (`receipt:{OrderId}`), Then
  `IEmailService.SendOrderReceiptEmailAsync` is **not** called again and the function **acks** (no
  throw). The terminal email effect happens **exactly once** per OrderId. (Verified against
  `GenerateReceiptFunction.cs:95`.)
- [ ] **AC2 â€” Claim-first ordering (the C6 fix).** The email-sent state is committed in a transaction
  that **precedes** the send, OR the send is gated on a re-read of an already-set `EmailSent` marker,
  so the window between `SendOrderReceiptEmailAsync` (`:95`) and `CommitAsync` (`:99`) can no longer
  re-send on a crash-then-redeliver. The accepted residual is the ADR-named one â€” a *rare lost email*
  on a crash after the claim-commit but before the send (ADR-0002 D2.2: "accepting a rare lost email
  on crash") â€” never a double email. The chosen trade-off is documented in code at the call site.
- [ ] **AC3 â€” Fiscal sequence consumed once.** Given a redelivery for an order that already has a
  receipt (fiscal code already issued for BlockingOnline / `BlockingWithOfflineCache` countries â€” see
  the enforcement-mode resolution at `:79-90`), When `Run` is re-invoked, Then
  `IReceiptService.GenerateReceiptAsync` (`:72`) is **not** re-invoked, so no second fiscal sequence
  number is consumed. The existing `order.Receipt is not null` guard (`:66-70`) is the load-bearing
  assertion here; AC verifies it short-circuits **before** the PDF/fiscal call on the redelivery path.
- [ ] **AC4 â€” Blocking-mode hold path stays idempotent.** Given a BlockingOnline order whose initial
  fiscal attempt failed (`receipt.FiscalCode == null`, the hold branch at `:83-90` commits and
  returns to await the retry job), When the message is later redelivered, Then the consumer does **not**
  send a premature email and does **not** double-consume a fiscal sequence; once the signature lands
  the email is still sent **exactly once**.
- [ ] **AC5 â€” Failure classification preserved (D3.3 fiscal carve-out).** A "target not found"
  (`:48-52`) stays **transient / throws** (bounded retry) â€” it must NOT be reclassified to
  permanent-ack (ADR-0002 D3.3 fiscal-queue carve-out + verification check #3). A malformed body
  (invalid/empty `OrderId`, `:36-40`) stays **acked / no throw**. AC verifies the existing classify
  behavior is not regressed by this change.
- [ ] **AC6 â€” Test proves it, test-first.** A test in `Cleansia.Tests` (against the
  `Cleansia.Functions.Core` library that T-0118/FUNC-CORE makes referenceable â€” ADR-0002 D5 step 1)
  invokes `Run` **twice with the same `QueueEnvelope<T>`** and asserts both `GenerateReceiptAsync`
  **and** `SendOrderReceiptEmailAsync` are invoked **once** (ADR-0002 verification check #5 / TC-IDEMP-0
  exact wording for `generate-receipt`). Written **test-first** (red â†’ green; visible in commit order
  per `agents/knowledge/testing.md`). The paired suite is **T-0127 (TC-IDEMP-0)** â€” same merge.

## Architect panel ruling (2026-06-03) — BINDING (governed by ADR-0004; pending owner approval to build)
The fiscal-receipt-idempotency panel (author + compliance challenger + crash-correctness challenger + lead)
**re-scoped F4** and produced **ADR-0004** (`agents/backlog/adr/0004-fiscal-receipt-idempotency-boundary.md`).
Key findings (ground-truth corrections by the panel):
- **The EMAIL half is already done** (T-0118 claim-first). **The FISCAL half is the open, regulatory part.**
- **A unique index on `OrderReceipts.OrderId` ALREADY EXISTS** (EF one-to-one) → receipt is one-per-order; the
  DB backstop ships → **NO migration for T-0119's at-most-once claim**.
- The fiscal window: `GenerateReceiptAsync` consumes the sequence + calls the tax authority BEFORE the row
  commits → a crash-then-redeliver re-burns a sequence + re-registers. `GetNextSequenceForYearAsync` is
  `COUNT(*)+1` (non-atomic, gappy — split to FISCAL-SEQ).

**T-0119 contract (ADR-0004 D-F4.1 + D-F4.3):**
1. `GenerateReceiptHandler.cs` (the real consumer is now `Cleansia.Functions.Core/Handlers/GenerateReceiptHandler.cs`
   post-T-0121 — the original ticket's `GenerateReceiptFunction.cs` line refs are stale; reconcile to the Core
   handler): reorder to **reserve → commit claim → register/PDF → email**; **narrow the `catch(Exception){throw;}`

   the claimed-but-unregistered path is **born retry-eligible at claim time** (C-A).
