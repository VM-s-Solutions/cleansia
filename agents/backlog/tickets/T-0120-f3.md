## Context

Today every queued message that fails 5 times silently disappears. `host.json` sets a single global
`maxDequeueCount: 5` (`src/Cleansia.Functions/host.json` √Ę‚Ä†‚Äô `extensions.queues.maxDequeueCount: 5`)
with **no per-queue handling and no dead-letter monitoring**. After 5 dequeues the Storage-queue
runtime moves the message to `<queue>-poison` and **nothing reads those queues** √Ę‚ā¨‚ÄĚ a receipt or
invoice that fails 5√É‚ÄĒ is silently lost fiscal/financial work. This is **finding F3** ("No
poison/dead-letter consumer on any of the 5 queues").

A second, related defect lives in the consumers: they do not distinguish a **permanent** failure from
a **transient** one. `SendPushNotificationFunction.cs:115-121` `throw`s on **any** exception √Ę‚ā¨‚ÄĚ
including a permanent deserialize failure caught at `:39-43` (and the already-handled missing-field
case at `:45-51` returns, but the catch-all re-throws everything else). Throwing on a permanent error
just burns the 5 retries and then poison-queues a message that can never succeed.

This ticket implements the **D3 poison/dead-letter contract** of **ADR-0002** (the side-effect
dispatch contract / ADR-OUTBOX). Per ADR-0002 the floor frozen for Wave-0 is: *every business queue
has a `-poison` consumer + durable store + alert*, and consumers classify failures permanent (ack) vs
transient (throw). It is the Wave-0 **tactical** poison floor that the Wave-1 durable outbox will sit
on top of unchanged.

The five business queues are enumerated in
`src/Cleansia.Core.Queue.Abstractions/QueueNames.cs:5-9`: `generate-receipt`, `generate-invoice`,
`notifications-dispatch`, `sitewide-promo-fanout`, `calculate-order-pay`.

Source: `agents/backlog/adr/0002-outbox-dispatch-contract.md` (D3, D3.3, verification checks #2/#3/#8/#9);
finding F3.

## Acceptance criteria

- [ ] **AC1 (poison consumer per business queue √Ę‚ā¨‚ÄĚ ADR-0002 D3.1)** √Ę‚ā¨‚ÄĚ Given the five business queues in
  `QueueNames.cs:5-9`, When the project is built, Then a `[QueueTrigger("<queue>-poison", √Ę‚ā¨¬¶)]` Function
  exists for **each** of the five queues (`generate-receipt-poison`, `generate-invoice-poison`,
  `notifications-dispatch-poison`, `sitewide-promo-fanout-poison`, `calculate-order-pay-poison`).
  Verification check #2 (one poison Function per `QueueNames` const) passes.
- [ ] **AC2 (dead-letter is persisted + alerted, never re-processed √Ę‚ā¨‚ÄĚ D3.1)** √Ę‚ā¨‚ÄĚ Given a poisoned message
  body arrives on a `-poison` queue, When the poison consumer runs, Then it writes a durable
  dead-letter record (admin-visible) via the dead-letter store **and** logs at `Error` (so it raises a
  Sentry/AppInsights alert), and **acks without throwing** (no re-poison). The poison consumer never
  attempts to re-process the original effect.
- [ ] **AC3 (fiscal queues get the mandatory durable DB row √Ę‚ā¨‚ÄĚ D3.1)** √Ę‚ā¨‚ÄĚ Given a message poisons on
  `generate-receipt` or `generate-invoice`, When its poison consumer runs, Then a durable `DeadLetter`
  DB row is recorded carrying at least the source queue name and the raw body, so the
  fiscal/financial artifact is recoverable/replayable by an admin. (The other three queues log + alert
  + store at minimum.)
- [ ] **AC4 (failure classification: permanent √Ę‚Ä†‚Äô ack, transient √Ę‚Ä†‚Äô throw √Ę‚ā¨‚ÄĚ D3.3)** √Ę‚ā¨‚ÄĚ Given a
  **malformed or business-rejected** message, When a consumer processes it, Then it logs at `Warning`
  and **acks** (returns, does not throw). Given an **infra/transient** failure, When a consumer
  processes it, Then it **throws** (so the runtime retries up to `maxDequeueCount`, then dead-letters).
  `SendPushNotificationFunction.cs:115-121` is split per this rule: a permanent deserialize/validation
  failure is acked, only transient/infra faults re-throw.
- [ ] **AC5 (fiscal target-not-found stays transient √Ę‚ā¨‚ÄĚ D3.3 carve-out)** √Ę‚ā¨‚ÄĚ Given a "target not found"
  on `generate-receipt`/`generate-invoice` (e.g. `GenerateReceiptFunction.cs:48-52`), When it occurs,
  Then it remains a **transient / bounded-retry throw** and is **not** reclassified to permanent-ack
  (this preserves the silent-loss net the reconciliation sweep depends on; the throw at
  `GenerateReceiptFunction.cs:50-51` is not "simplified" into an ack).
- [ ] **AC6 (host.json note √Ę‚ā¨‚ÄĚ D3.2)** √Ę‚ā¨‚ÄĚ Given `host.json`, When reviewed, Then the global
  `maxDequeueCount: 5` (`extensions.queues.maxDequeueCount`) is left at 5 (ADR-0002 D3.2 requires no
  per-queue override); any change is justified against D3.2. The contract floor is the `-poison`
  consumers + store + alert, not a maxDequeue change.
- [ ] **AC7 (tests prove the holes are closed √Ę‚ā¨‚ÄĚ TEST-FIRST per knowledge/testing.md)** √Ę‚ā¨‚ÄĚ
  - **TC-POISON-0 (D3, check #8):** given a body, a `-poison` consumer writes a durable `DeadLetter`
    row and does **not** throw (does not re-poison). One case per fiscal poison consumer at minimum.
  - **TC-CLASSIFY-0 (D3.3, check #9):** per affected consumer, a malformed body is **acked / no
    throw**; a simulated infra fault **throws**. This is the test that catches
    `SendPushNotificationFunction.cs:115-121`'s throw-on-everything.
  Each test is written **before** its implementation (red √Ę‚Ä†‚Äô green), visible in the diff/commit order
  and noted in the status log.

## MANUAL_STEP (owner-only) ‚ÄĒ EF migration for the new `DeadLetter` table

> **DO NOT let Claude run `dotnet ef`.** This delta is owner-only. It is intended to **fold into the
> owner's Initial regen** (bundled with the other pending deltas ‚ÄĒ e.g. the F2/F4 `ProcessedMessage`
> table if/when it lands). An incremental migration is given below as the alternative.

The durable dead-letter store (ADR-0002 D3 / F3) adds one new table backing the `DeadLetter` domain
entity (`Cleansia.Core.Domain/DeadLettering/DeadLetter.cs`, config
`Cleansia.Infra.Database/EntityConfigurations/DeadLetterEntityConfiguration.cs`, DbSet
`CleansiaDbContext.DeadLetters`). Nullable `TenantId` by design ‚ÄĒ a poison body may be unparseable so
the tenant can be unknown.

**Table `DeadLetters`** (`Auditable` + `ITenantEntity` base columns + the F3-specific columns):

| Column | Type | Null | Notes |
|---|---|---|---|
| `Id` | `varchar(26)` | NO | PK (ULID) |
| `TenantId` | `varchar(26)` | YES | nullable ‚ÄĒ unknown tenant for an unparseable poison body; indexed (base) |
| `SourceQueue` | `varchar(128)` | NO | business queue name w/o `-poison` suffix; indexed |
| `RawBody` | `text` | NO | verbatim poisoned body (unbounded ‚ÄĒ fiscal recovery/replay source) |
| `Error` | `text` | YES | optional error/exception text |
| `DeadLetteredAt` | `timestamptz` | NO | when the row was recorded (UTC); indexed |
| `IsActive` | `boolean` | NO | base |
| `CreatedBy` | `varchar(255)` | NO | base audit |
| `CreatedOn` | `timestamptz` | NO | base audit |
| `UpdatedBy` | `varchar(255)` | YES | base audit |
| `UpdatedOn` | `timestamptz` | YES | base audit |
| `DeactivatedBy` | `varchar(255)` | YES | base audit |
| `DeactivatedOn` | `timestamptz` | YES | base audit |

Indexes: `IX_DeadLetters_TenantId` (base), `IX_DeadLetters_SourceQueue`, `IX_DeadLetters_DeadLetteredAt`.

**Incremental migration (alternative to the Initial regen):**

```bash
dotnet ef migrations add AddDeadLetterTable \
  --project src/Cleansia.Infra.Database \
  --startup-project src/Cleansia.Web.Partner/Cleansia.Web.Partner.csproj \
  --context CleansiaDbContext
# then:
dotnet ef database update --project src/Cleansia.Infra.Database \
  --startup-project src/Cleansia.Web.Partner/Cleansia.Web.Partner.csproj \
  --context CleansiaDbContext
```

No NSwag regen (the queue/dead-letter contract is internal ‚ÄĒ ADR-0002 Consequences).
