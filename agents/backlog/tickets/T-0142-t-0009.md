## Context
Consistency rule **B6** (`agents/knowledge/consistency.md:67-72`) canonicalizes deletion on
**soft-delete via `repo.Deactivate(entity)`** (sets `IsActive=false`, stamps `DeactivatedBy`/
`DeactivatedOn`, preserves history/audit) for any user- or business-facing entity, reserving
`repo.Remove(entity)` (hard delete) for true join/scratch rows. The codebase currently hard-deletes
widely. B6 is an **Architect-owned judgment call** (`consistency.md:158`): adopting it platform-wide
must be ratified as an **ADR**, not done ad-hoc.

This is the Wave-1 ADR ticket for that judgment call (TICKET-MAP Wave 1, `source: T-0009`). It
authors the soft-delete ADR, then sweeps the concrete confirmed instances:

- **DA-10** (major) ‚Äî `DeleteSavedAddress` hard-deletes an audited, user-facing entity:
  `Features/SavedAddresses/DeleteSavedAddress.cs:56` `savedAddressRepository.Remove(saved)`.
  `SavedAddress : Auditable` carries `DeactivatedBy/On` audit fields it never uses.
- **D-09** (latent S10) ‚Äî `SavedAddressRepository.GetByUserAsync` filters only by `UserId`, no
  `IsActive` predicate (`SavedAddressRepository.cs:10-19`); same for `GetDefaultForUserAsync`
  (`:21-27`). The moment delete becomes soft-delete, deactivated addresses resurface in the list.
  S10 (`security-rules.md`) requires "list my saved addresses" to exclude deactivated rows.
- **DA-15** (latent data-loss) ‚Äî `GetSavedAddresses.cs:22` `.Where(s => s.Address != null)` silently
  drops rows with a null `Address` FK instead of surfacing the integrity problem. The ADR must
  classify whether this band-aid stays (and logs) once the soft-delete invariant is in force.
- **IA-13** (minor) ‚Äî `Features/Devices/UnregisterDevice.cs:34` `deviceRepository.Remove(device)`.
  The ADR decides whether `Device` is a true scratch row (keep `Remove`, document why) or a
  business-facing row (move to `Deactivate`). Note `DeviceRepository` already filters `IsActive`
  on its reads (`DeviceRepository.cs:12,18,24`), so the read side is already soft-delete-ready.

The soft-delete primitives **already exist**: `IRepository.Deactivate/DeactivateRange`
(`IRepository.cs:45,47`) and the `Auditable.Deactivated(by, on)` domain method that sets
`IsActive=false` + stamps audit fields (`Auditable.cs:35-42`). This ticket adopts them, it does not
build them.

## Acceptance criteria
- [ ] **AC1 (ADR)** ‚Äî Given B6 is a tracked Architect judgment call, When this ticket completes,
  Then a new accepted ADR under `agents/backlog/adr/` states the soft-delete policy: the
  Deactivate-vs-Remove decision rule (user/business-facing ‚Üí `Deactivate`; true join/scratch carrying
  no history and never referenced ‚Üí `Remove`, documented per-site), the `IsActive`-filter obligation
  on every "list mine" read (S10), the DA-15 null-FK classification, and the per-entity verdict for
  `SavedAddress` and `Device`. The ADR number is added to this ticket's `adrs:` frontmatter and any
  dependent Wave-2 ticket (e.g. CC-02/03/04/06) references it.
- [ ] **AC2 (DA-10)** ‚Äî Given a customer with a saved address, When `DeleteSavedAddress` runs, Then
  the handler calls `repo.Deactivate(saved)` (not `Remove`); the row persists with `IsActive=false`
  and `DeactivatedBy/DeactivatedOn` stamped, and `BusinessResult.Success()` is returned. A handler
  unit test (mocked `ISavedAddressRepository`) asserts `Deactivate` was invoked and `Remove` was not.
- [ ] **AC3 (D-09)** ‚Äî Given a customer has both active and deactivated saved addresses, When
  `GetSavedAddresses` runs, Then only `IsActive` rows are returned (the deactivated address does not
  appear). `GetByUserAsync` and `GetDefaultForUserAsync` carry `.Where(s => s.IsActive)`. A test
  proves a deactivated address is excluded from the list and is never returned as the default.
- [ ] **AC4 (DA-15)** ‚Äî Given a saved address whose `Address` FK is null, When `GetSavedAddresses`
  runs, Then the row is handled per the ADR's decision (surfaced/logged rather than silently dropped,
  or the band-aid is retained with a justifying comment + a non-PII log when one is skipped) ‚Äî the
  chosen behavior has a test pinning it.
- [ ] **AC5 (IA-13)** ‚Äî Given the ADR's verdict for `Device`, When `UnregisterDevice` runs, Then it
  either calls `repo.Deactivate(device)` (if business-facing) or keeps `repo.Remove(device)` with an
  inline comment citing the ADR's scratch-row exception (if scratch). The chosen path has a handler
  test asserting the call.
- [ ] **AC6 (migration)** ‚Äî Given soft-delete now leaves rows in place, When the owner is handed the
  `manual_steps`, Then a `MANUAL_STEP: ef-migration` note flags any schema change needed (e.g. a
  filtered/partial unique index so the single-default-address invariant ‚Äî `ClearDefaultForUserAsync`,
  `SavedAddressRepository.cs:29-39` ‚Äî and any uniqueness still hold across deactivated rows). If no
  schema change is required, the ticket states that explicitly so the owner can skip it.
- [ ] **AC7 (tests precede code)** ‚Äî Each behavioral AC above maps to a test that appears **before/
  with** the implementation in the diff; the status log notes the red‚Üígreen per `agents/knowledge/
  testing.md` (test-first; characterization test pins current behavior of the untested handlers
  first per `testing.md:53-56`).
