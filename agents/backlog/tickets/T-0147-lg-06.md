## Context
Major bug-risk / consistency finding from the 2026-06-01 audit. The four membership commands call
Stripe **bare** ‚Äî no provider `try/catch` (B8) ‚Äî so a transient Stripe failure bubbles as an
unhandled **500** instead of a `PaymentGatewayUnavailable` business failure. The canonical pattern
already exists one feature over in `CreateOrder.cs:357-372` (narrow `catch (StripeException)` ‚Üí
`BusinessResult.Failure(..., BusinessErrorMessage.PaymentGatewayUnavailable)`); the membership
commands never adopted it. The originally-tracked F8 named only `CreateMembershipSubscription`; this
ticket extends it to **all four** call sites the audit enumerated:

- `CreateMembershipSubscription.cs` ‚Äî `CreateCustomerAsync` (`:67`), `CreateSubscriptionAsync`
  (`:85`), `CreateSetupIntentAsync` (`:107`), `CreateEphemeralKeyAsync` (`:108`)
- `CreateMembershipCheckoutSession.cs` ‚Äî `CreateCustomerAsync` (`:64`),
  `CreateMembershipCheckoutSessionAsync` (`:80`)
- `SwapMembershipPlan.cs` ‚Äî `SwapSubscriptionPriceAsync` (`:61`)
- `CancelMembershipSubscription.cs` ‚Äî `CancelSubscriptionAtPeriodEndAsync` (`:38`)

There is also an **S7** dimension: in `CreateMembershipSubscription` phase-2
(`PaymentMethodConfirmed == true`, `:79-94`) `CreateSubscriptionAsync` succeeds at Stripe and the
local `UserMembership` is added (`:94`), but the pipeline commit happens later ‚Äî if that commit
fails, the user is **billed with no local membership row** and there is no guard to reconcile on
retry. This ticket closes the bare-call hole and adds a reconcile-on-retry guard for that phase-2
window so a retried confirm does not double-subscribe.

Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:1486-1491` (LG-06, B8/S7);
finding `agents/backlog/audits/AUDIT-2026-06-01-findings.json` (id LG-06, severity major, type bug);
plan `agents/backlog/audits/AUDIT-2026-06-01-execution-plan.md:156`. Reference pattern:
`src/Cleansia.Core.AppServices/Features/Orders/CreateOrder.cs:357-372` and
`CreatePaymentIntent.cs:147`.

## Acceptance criteria
- [ ] **AC1 (every Stripe call is guarded ‚Äî the hole is closed)** ‚Äî Given any of the four membership
  commands, When a Stripe call it makes throws `StripeException`, Then the handler returns
  `BusinessResult.Failure<T>` with `Error.Message == BusinessErrorMessage.PaymentGatewayUnavailable`
  (an observable 4xx business failure, **not** an unhandled 500), and logs the exception via the
  existing `ILogger<Handler>`. Covers: `CreateCustomerAsync`, `CreateSubscriptionAsync`,
  `CreateSetupIntentAsync`, `CreateEphemeralKeyAsync` (`CreateMembershipSubscription.cs:67,85,107,108`);
  `CreateCustomerAsync`, `CreateMembershipCheckoutSessionAsync` (`CreateMembershipCheckoutSession.cs:64,80`);
  `SwapSubscriptionPriceAsync` (`SwapMembershipPlan.cs:61`); `CancelSubscriptionAtPeriodEndAsync`
  (`CancelMembershipSubscription.cs:38`).
- [ ] **AC2 (narrow catch ‚Äî non-Stripe faults still surface)** ‚Äî Given a non-`StripeException`
  failure (DI misconfig, null-ref, bad state), When a membership handler runs, Then it is **not**
  masked as `PaymentGatewayUnavailable` ‚Äî it bubbles as a 500 so we see it. The `catch` is typed to
  `StripeException` only, matching the rationale comment at `CreateOrder.cs:364-367`.
- [ ] **AC3 (phase-2 reconcile-on-retry ‚Äî S7)** ‚Äî Given a user whose `CreateMembershipSubscription`
  phase-2 (`PaymentMethodConfirmed == true`) previously created a Stripe subscription but failed to
  commit the local `UserMembership`, When the command is retried, Then the handler does **not** create
  a **second** Stripe subscription / second `UserMembership` row ‚Äî it reconciles to the existing Stripe
  subscription (the second attempt does not double the effect). The pre-Stripe
  `GetActiveForUserAsync != null` check (`CreateMembershipSubscription.cs:57`) alone does **not**
  satisfy this AC, because the failed attempt left no local row.
- [ ] **AC4 (a test proves it ‚Äî TDD, red first)** ‚Äî Unit tests in `Cleansia.Tests` drive each of the
  four handlers with a mocked `IStripeClient` configured to throw `StripeException` on the relevant
  call and assert `result.Error!.Message == BusinessErrorMessage.PaymentGatewayUnavailable` (assert the
  constant, never a hardcoded string). A sibling test for AC3 drives phase-2 twice (Stripe sub created,
  first commit simulated-failed) against the mock and asserts the subscription side-effect fires
  **once**. Written **red before** the handler change per `agents/knowledge/testing.md` (must-cover #6
  idempotency/S7 and #8 every `BusinessErrorMessage` path); the status log records the red‚Üígreen
  transition and the test predates the implementation in the diff/commit order.
- [ ] **AC5 (no happy-path regression)** ‚Äî Given Stripe succeeds, When each command runs, Then it
  returns the same `BusinessResult.Success` `Response` as today (subscription/checkout-url/swap/cancel
  unchanged); existing validation-failure branches (UserNotFound, MembershipPlanNotFound,
  MembershipAlreadyActive, MembershipNotFound, MembershipSwapSamePlan) are untouched.
