## Context

Two grounded loyalty correctness gaps from the 2026-06-01 audit, bundled because they live in the
same `LoyaltyService.cs` / loyalty-domain surface and must land after the idempotency rework.

**LG-01q (critical / dead-config)** — `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:1438-1443`,
`AUDIT-2026-06-01-findings.json:1633-1640`. The whole tier-threshold admin tooling is decorative:
`LoyaltyAccount.RecomputeTier()` (`src/Cleansia.Core.Domain/Loyalty/LoyaltyAccount.cs:98-113`)
**hardcodes** `>= 5000 / >= 2000 / >= 500` literals, while `LoyaltyTierConfig.LifetimePointsThreshold`
(`src/Cleansia.Core.Domain/Loyalty/LoyaltyTierConfig.cs:16`) is a per-tenant, admin-editable column
driven end-to-end by `UpdateTierConfig`, `PreviewTierThresholdImpact` and `AdminLoyaltyTierController`.
An admin can edit and preview thresholds, but the domain never reads them — tiers are computed against
the magic numbers, so the edited config has **zero effect** (also a `conventions.md` "no magic numbers"
violation). The correct `ResolveTier(points, thresholds)` switch already exists in
`PreviewTierThresholdImpact.Handler` (`PreviewTierThresholdImpact.cs:102-108`).

**LG-03 (major / gap — no audit trail)** — `AUDIT-2026-06-01-findings.json:411-422`,
`slice-reports.md` Loyalty section. `GrantPointsManually.Command` requires a `Reason` (NotEmpty +
MaxLength 500 — `GrantPointsManually.cs:38-43`) and `RevokePointsManually` mirrors it
(`RevokePointsManually.cs:38-43`), but the handler calls `GrantPointsManuallyAsync` /
`RevokePointsManuallyAsync` **without passing it** (`GrantPointsManually.cs:54-60`,
`RevokePointsManually.cs:54-60`). `LoyaltyService.GrantPointsManuallyAsync` has no description
parameter (`LoyaltyService.cs:181-211`) and the ledger row is created with `description: null` via
`LoyaltyAccount.GrantPoints` → `LoyaltyTransaction.Create` (`LoyaltyAccount.cs:54-72`,
`LoyaltyTransaction.cs:42-60`). Admins are forced to type a justification on every money-adjacent point
adjustment, and it is silently dropped — the compliance question "why did this user get 5,000 points?"
cannot be answered from the ledger. The `LoyaltyTransaction.Description` column **already exists**
(`LoyaltyTransaction.cs:33-34`, MaxLength 500) and `Create` already accepts `description`
(`:42-48`) — nothing is persisted only because the value is never threaded through. The admin ledger
DTO `GetUserLoyaltyActivity.ActivityItem` (`GetUserLoyaltyActivity.cs:19-26`) also does not surface it.

This ticket **rebases on T-0112 (LG-SEC-06)**, which reworks the same `GrantPointsManuallyAsync` /
`RevokePointsManuallyAsync` signatures and the ledger-write path for idempotency; thread `Reason`
through the *post-T-0112* signatures and preserve its idempotency guard (ADR-0002).

## Acceptance criteria

- [ ] **AC1** — Given a tenant whose `LoyaltyTierConfig` rows set thresholds different from the legacy
  defaults (e.g. Silver=300, Gold=1500, Platinum=4000), When a user's `LifetimePoints` crosses one of
  those configured thresholds via a grant/revoke, Then `RecomputeTier()` resolves `CurrentTier` from
  the tenant's **configured** `LifetimePointsThreshold` values — not the `5000/2000/500` literals,
  which are deleted from `LoyaltyAccount.cs:100-106`. A test proves a points value that would be one
  tier under the hardcoded numbers maps to a *different* tier under the configured numbers.
- [ ] **AC2** — Given a tier whose config row is missing for the current tenant, When `RecomputeTier()`
  runs, Then it degrades safely (no exception; falls back to the lowest tier / an unreachable threshold
  for the missing tier, mirroring `PreviewTierThresholdImpact.cs:55-58`), and a test covers the
  missing-config path.
- [ ] **AC3** — Given an admin manual grant with `Reason = "goodwill credit #4821"`, When the command
  is handled, Then the persisted `LoyaltyTransaction.Description` equals that reason (not null). Mirror
  for revoke. A test asserts the persisted `Description` round-trips from `Command.Reason`.
- [ ] **AC4** — `Reason` is threaded `GrantPointsManually/RevokePointsManually.Handler`
  → `ILoyaltyService.GrantPointsManuallyAsync/RevokePointsManuallyAsync` (new `description`/`reason`
  parameter) → `LoyaltyAccount.GrantPoints/RevokePoints` → `LoyaltyTransaction.Create(description:)`.
  The order-driven / referral grant paths (which pass no admin reason) keep passing `null` and are
  unchanged.
- [ ] **AC5** — `GetUserLoyaltyActivity.ActivityItem` (`GetUserLoyaltyActivity.cs:19-26`) surfaces the
  ledger `Description` so the admin ledger view can render the justification; the projection at
  `:66-75` maps `t.Description`. (Customer `GetLoyaltyActivity.ActivityItem` may also surface it if the
  developer judges it safe; default scope is the admin DTO only — see Out of scope.)
- [ ] **AC6** — The T-0112 idempotency guard on the manual grant/revoke path still holds: a
  double-submit with the same `requestId` still lands the effect exactly once *and* records the reason
  on the single surviving row. A test proves the reason-threading did not regress idempotency.
- [ ] **AC7** — All of the above are proven by unit/integration tests in `Cleansia.Tests`, written
  **test-first** (red → green, visible in commit/diff order per `agents/knowledge/testing.md`):
  the threshold-resolution test (AC1/AC2), the reason-persistence test (AC3), and the
  idempotency-non-regression test (AC6).

## Out of scope

- The idempotency `requestId` / unique-index / rate-limit work itself — that is **T-0112 (LG-SEC-06)**,
  this ticket's dependency. T-0148 only consumes its reworked signatures.
- `PreviewTierThresholdImpact` being a read modeled as a `Command` (CQRS violation + needless commit) —
  that is **LG-09**, a separate Wave 3 ticket. Do not convert it here.
- `ParsePerks` de-duplication (**LG-12**) and the stale `OrderFactory` floor comment (**LG-08**) —
  separate tickets.
- Any admin-UI rendering of the new `Reason` column. If a frontend consumer is wired to read the new
  `ActivityItem.Description` field in the same wave, the admin NSwag client must be regenerated
  (`manual_step: nswag-regen`) — but the DTO addition is additive and the consumer wiring is a
  follow-up, so this ticket stays backend-only with `manual_steps: []`. Flag the latent regen to the
  owner if the admin ledger UI is scheduled alongside.
- No new column / migration: `LoyaltyTransaction.Description` and `LoyaltyTierConfig.LifetimePointsThreshold`
  already exist — **no `ef-migration`**.
