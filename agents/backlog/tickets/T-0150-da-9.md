## Context
Audit finding **DA-9** (`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:2112`,
`AUDIT-2026-06-01-findings.json` id `DA-9`): the same magic constants are hand-copied across
backend, frontend, and mobile, so a change to the default country or a length cap has to be hunted
across three stacks. This violates `agents/knowledge/conventions.md` — "No magic numbers/strings —
constants live in a Policy class / enum / theme token."

Confirmed live offenders (real code, 2026-06-01):
- Default-country ISO `"CZE"` literal: `Cleansia.Core.AppServices/Features/SavedAddresses/AddSavedAddress.cs:95`
  and `…/UpdateSavedAddress.cs:127`.
- Lat/long bounds `-90/90` and `-180/180` inline in the `AddSavedAddress` validator
  (`AddSavedAddress.cs:63-71`) and the `UpdateSavedAddress` validator (`UpdateSavedAddress.cs:81-89`).
- The dispute-description `10..2000` cap repeated in the `CreateDispute` validator
  (`Cleansia.Core.AppServices/Features/Disputes/CreateDispute.cs:33-35`), the `Dispute.Description`
  data annotation, the FE component
  (`Cleansia.App/libs/cleansia-customer-features/disputes/src/lib/disputes/disputes.component.ts:78`),
  and the mobile VM
  (`cleansia_android/customer-app/.../features/disputes/CreateDisputeViewModel.kt:59`).

## Acceptance criteria
- [ ] **AC1** — Given the default-country fallback, When `AddSavedAddress.Handler` and
  `UpdateSavedAddress.Handler` resolve a missing country, Then both read the ISO code from a single
  named backend constant (e.g. `AddressDefaults.FallbackCountryIso`), and the literal `"CZE"` no
  longer appears in either handler.
- [ ] **AC2** — Given coordinate validation, When the `AddSavedAddress` and `UpdateSavedAddress`
  validators bound latitude/longitude, Then both reference named `GeoBounds.*` constants
  (`LatMin/LatMax/LonMin/LonMax`), and the inline `-90/90/-180/180` literals are gone from both
  validators.
- [ ] **AC3** — Given the dispute-description length rule, When `CreateDispute`'s validator and the
  `Dispute.Description` annotation apply min/max, Then both reference named constants (e.g.
  `DisputeLimits.DescriptionMin = 10` / `DescriptionMax = 2000`), and the bare `10`/`2000` literals
  are gone from the backend.
- [ ] **AC4** — Given the customer web disputes screen, When `disputes.component.ts:78` enforces the
  description max, Then it reads a single FE constant (one named home, e.g. a shared
  `dispute.constants.ts`) instead of the inline `2000`.
- [ ] **AC5** — Given the customer Android dispute form, When `CreateDisputeViewModel.kt:59` validates
  the description range, Then it reads named Kotlin constants (one object) instead of bare `10..2000`.
- [ ] **AC6** — Given the new constants, When the test suite runs, Then a backend unit test proves the
  validators reject out-of-bounds coords and over-cap descriptions and accept boundary values using
  the named constants (the hole is closed and a test pins the values), per
  `agents/knowledge/testing.md`. `dotnet build Cleansia.Api.sln` succeeds; `npx nx lint` /
  `npx nx test` for the disputes lib pass; the Android module compiles.

## Out of scope
- The `AddSavedAddress` god-method refactor and the triplicated `SavedAddressDto` projection — that
  is DA-8, not this ticket.
- Soft-delete of `SavedAddress`/`DeleteSavedAddress` (DA-10) and the country-default helper extraction
  beyond pointing at the named constant.
- A build-time codegen/shared-source mechanism to sync the constant across stacks — each stack gets
  its own named home that mirrors the same value (per finding fix: "mirror from a single source where
  practical"); no new tooling.
- Adding the missing `errors.*` dispute/address translations (DA-7).

## Implementation notes
- **Serialization cluster:** none. DA-9 does not appear in any cluster in
  `agents/backlog/TICKET-MAP.md`'s shared-file map, so it can run concurrently — but note it edits
  `CreateDispute.cs` (touched by the `AddDisputeMessage`/dispute-backend cluster only for that
  handler, not `CreateDispute`) and `AddSavedAddress.cs`/`UpdateSavedAddress.cs` (DA-8 territory).
  Do **not** run this concurrently with DA-8 (same SavedAddress files) — serialize against it.
- **ADR:** none governs this; it is a `conventions.md` "no magic constants" cleanup. `adrs: []` is
  intentional.
- **Backend home:** put the constants where the convention expects — a small static class
  (`AddressDefaults.FallbackCountryIso`, `GeoBounds.LatMin/LatMax/LonMin/LonMax`,
  `DisputeLimits.DescriptionMin/Max`) in the AppServices layer; reference from the validators,
  handlers, and the `Dispute.Description` annotation (annotation must use a `const int`).
- **FE home:** one named constant module in the disputes lib reused by `disputes.component.ts`.
- **Mobile home:** one Kotlin `object` of constants reused by `CreateDisputeViewModel`.
- **Built TEST-FIRST** per `agents/knowledge/testing.md`: write the validator/boundary tests (AC6)
  before swapping the literals for constants, so the test fails red against the magic numbers first.
- Cited lines verified against live code on 2026-06-01 (see Context).
