## Context

Audit finding **AUD-01** (the single highest-value *feature* fix — `audits/AUDIT-2026-06-01-summary.md`
§1 "Top 3 risks", lines 36-46; ranked-list line 55; confirmed in `…-execution-plan.md` lines 47-49,
164) is: **there is no admin order intervention.** The back-office can *observe* orders but cannot
**cancel, reassign, refund, or override the status** of any order — incidents force manual DB edits.

Proven against the real code:

- **The Admin host has read-only order endpoints.** `Cleansia.Web.Admin/Controllers/AdminOrderController.cs`
  exposes exactly three actions — `GetPagedOrders` (`:16-26`), `GetOrderDetails` (`:28-39`),
  `GetOrderPhotos` (`:41-52`). There is **no** cancel / reassign / refund / status-override action on
  the Admin host at all.
- **The only cancel path is customer-self-service.** `CancelOrder.Handler`
  (`Cleansia.Core.AppServices/Features/Orders/CancelOrder.cs`) gates on `order.UserId != userId`
  (`:75-80`) — an admin (whose `sub` is not the order's `UserId`) gets `OrderNotFound`. It **hardcodes**
  `cancelledBy: "customer"` (`:123`), and its policy `CanCancelOrder` is **CustomerOnly** (`Policy.cs:33`;
  ADR-0001 D2 row, `adr/0001-authorization-model.md:229`). So cancellation is not *generalized* — it
  cannot represent an admin- or system-initiated cancel.
- **The refund money path is the one ADR-REFUND (T-0140) just froze.** `CancelOrder` issues an inline,
  **un-keyed** Stripe refund (`CancelOrder.cs:139-151`, `RefundCheckoutSessionAsync(...)` with no
  idempotency key) and flips `PaymentStatus.Refunded` (`:143`). T-0140 names the single refund seam +
  the deterministic idempotency-key formula; **this ticket is its Wave-2 consumer** (T-0140 AC7 lists
  `AUD-01` as a dependent), so the admin refund/cancel-refund must go through that seam, not re-invent it.
- **`CancelledBy` is a magic string today.** `Order.Cancel(...)` takes `string cancelledBy`
  (`Cleansia.Core.Domain/Orders/Order.cs:441-454`); the column is `CancelledBy` `MaxLength(20)`
  (`Order.cs:136-140`) holding `"customer"`/`"cleaner"`/`"system"` free-text. **AUD-15** (typing the
  order-status email param + introducing a `CancelledBy` enum) **folds into this ticket** per the INDEX
  (AUD-15 `blocks: AUD-01`) and the title.
- **Reassign target exists in the domain.** Assignment is `OrderEmployee`
  (`Cleansia.Core.Domain/Orders/OrderEmployee.cs`); `Order` exposes `_assignedEmployees` /
  `AddAssignedEmployee` / `AvailableSpots` / `IsFullyAssigned` (`Order.cs:94-96,227-228,394-403`) but
  has **no** un-assign/reassign method yet. Status is `OrderStatus` (7 states incl. `OnTheWay`,
  `Cleansia.Core.Domain/Enums/OrderStatus.cs`) tracked via `OrderStatusTrack` / `AddOrderStatus`
  (`Order.cs:320-325`); there is no admin status-override command.

This ticket builds the admin order-operations surface end-to-end (backend commands + AdminOrderController
endpoints + admin-app UI) **and** generalizes cancellation so the actor is typed and admin/system cancels
are first-class. It depends on **T-0100** (ADR-0001 — the fail-closed map + the `AdminOnly` gate the new
permissions must use) and **T-0140** (ADR-REFUND — the frozen refund seam + idempotency key).

> **SIZE = L — must be SPLIT before it goes `ready`** (PM constraint; ADR-0001 rollout note discourages
> running L tickets whole). Proposed split, each child its own ticket + paired test ticket, sequenced so
> the contract locks before consumers (routing.md: architect-confirmed contract → db → backend →
> frontend): **(a)** generalized cancellation + `CancelledBy` enum (folds AUD-15) + admin-cancel command
> through the ADR-REFUND refund seam; **(b)** admin status-override command + endpoint + UI; **(c)** admin
> reassign command + endpoint + UI; **(d)** admin refund-only command (no status change) through the same
> seam. The AC below are the parent contract each child inherits.

## Acceptance criteria

- [ ] **AC1 — `CancelledBy` is a typed enum (folds AUD-15).** Given `Order.Cancel(...)` takes a
  free-text `string cancelledBy` (`Order.cs:443`) and the `CancelledBy` column holds `"customer"`/
  `"cleaner"`/`"system"` (`Order.cs:136-140,123`), When this lands, Then a `CancelledBy` enum
  (`Customer`, `Cleaner`, `Admin`, `System`) exists in `Cleansia.Core.Domain/Enums/`, `Order.Cancel`
  takes it (mapping/persistence preserves existing values for already-cancelled orders), and the
  existing customer path passes `CancelledBy.Customer` (replacing the literal at `CancelOrder.cs:123`).
  Evidence: enum type + updated `Cancel` signature; a domain test asserts each actor round-trips.

- [ ] **AC2 — Admin can cancel any order.** Given an admin authenticated on the Admin host and an order
  the admin does **not** own, When the admin calls the new admin-cancel endpoint, Then the order is
  cancelled with `CancelledBy.Admin` recorded — i.e. **without** the `order.UserId != userId` ownership
  rejection that blocks admins today (`CancelOrder.cs:75-80`). The terminal-state guards (already
  Cancelled / Completed / InProgress, `CancelOrder.cs:86-103`) still apply and return their existing
  `BusinessErrorMessage` codes. Evidence: handler test — admin cancels a non-owned `Confirmed` order →
  `Success`; admin cancels a `Completed` order → `OrderAlreadyCompleted`.

- [ ] **AC3 — Admin cancel/refund goes through the ADR-REFUND seam (no double-refund).** Given ADR-REFUND
  (T-0140) names one refund seam and a deterministic idempotency key, and `CancelOrder.cs:139-151`
  currently issues an **un-keyed** Stripe refund, When an admin cancel-with-refund (or refund-only,
  AC6) executes, Then the Stripe refund is issued **only** via that seam **with the deterministic
  idempotency key**, `RefundAmount` is consumed per ADR-REFUND AC3, and a **retried** admin cancel/refund
  does **not** issue a second Stripe refund. Evidence: a refund-idempotency test (TC-7 / TC-IDEMP
  obligation per T-0140 AC4/AC7) proves the second call is a no-op on Stripe.

- [ ] **AC4 — Admin can override order status.** Given an order in a non-terminal state, When an admin
  calls the status-override endpoint with a valid `OrderStatus` (`OrderStatus.cs`, 7 values incl.
  `OnTheWay`), Then an `OrderStatusTrack` is appended via `AddOrderStatus` (`Order.cs:320-325`), the
  override is restricted to allowed transitions (illegal/ambiguous transitions return a documented
  `BusinessErrorMessage` code rather than corrupting history), and the change is audit-attributed to the
  admin. Evidence: handler test — admin overrides `Confirmed → OnTheWay` → `Success`; an illegal
  override → the documented error.

- [ ] **AC5 — Admin can reassign a cleaner.** Given an order with one or more `OrderEmployee`
  assignments (`OrderEmployee.cs`; `Order._assignedEmployees`, `Order.cs:227-228`) and a target
  employee, When an admin calls the reassign endpoint, Then the order's assignment is updated (an
  un-assign/reassign domain method is added — none exists today) respecting `MaxEmployees` /
  `AvailableSpots` (`Order.cs:94-96,420-429`), and the action is attributed to the admin. Evidence:
  handler test — reassign to an available cleaner → `Success`; reassign that would exceed `MaxEmployees`
  → the existing "no available spots" guard surfaces as a business error (not an unhandled exception).

- [ ] **AC6 — Admin can issue a refund without cancelling.** Given a paid card order, When an admin calls
  the refund-only endpoint, Then a Stripe refund is issued **through the ADR-REFUND seam with the
  idempotency key** (AC3), `PaymentStatus` transitions per ADR-REFUND AC3, and the order's lifecycle
  status is **unchanged**. Evidence: handler test — refund-only on a `Confirmed` paid order leaves
  status `Confirmed` and `PaymentStatus = Refunded`; a retry is idempotent.

- [ ] **AC7 — All four operations are `AdminOnly` and mapped (ADR-0001 D1/D2).** Given T-0100 made the
  authz map fail-closed (`ToPhysicalPolicy` → `Deny` for unmapped, `PolicyBuilder.AssertComplete()`),
  When each new `Policy.*` constant (e.g. `CanAdminCancelOrder`, `CanOverrideOrderStatus`,
  `CanReassignOrder`, `CanRefundOrder`) is added, Then **each** is mapped to **`AdminOnly`** in
  `PolicyBuilder.Map`, the frozen-map snapshot test (T-0100 #4) is updated **in this PR**, and a
  **non-admin** (customer/employee) calling any of the four is **denied (403)**, never fail-open.
  Evidence: map rows + snapshot update; a per-permission test — admin passes, customer/employee fail.
  (Per ADR-0001 D2 header this is additive-rows-with-snapshot-update, no superseding ADR needed.)

- [ ] **AC8 — Admin UI surfaces the four operations.** Given the admin-app order detail/list, When an
  admin opens an order, Then cancel / reassign / status-override / refund actions are available via
  `<cleansia-*>` / PrimeNG controls (no raw form controls), all user-visible strings use `TranslatePipe`
  with keys present in **all 5 locales** (`en, cs, sk, uk, ru`), logic lives in a facade (not the
  component), and every backend `BusinessErrorMessage` code returned by AC2-AC6 has a matching
  `errors.*` translation. Evidence: facade + component + `*.html` using shared components; i18n keys in
  all 5 files; the AUD-14 `OnTheWay` badge/icon gap is **not** reintroduced (status display covers all 7
  `OrderStatus` values).

## Out of scope

- **Customer-web cancellation parity** (AUD-05) and the **partner/mobile** order surfaces — separate
  tickets; this is Admin-host + the generalized cancellation backend only.
- **The `charge.dispute.*` chargeback webhook + `LinkStripeDispute`** (D-06) and **admin dispute
  management / dispute-refund** (`D-01/DA-1/SEC-DSP-06/07`) — separate Wave-2 tickets that also depend
  on ADR-REFUND; AUD-01 is order refunds, not dispute refunds.
- **Re-deciding the refund seam or idempotency-key formula** — frozen by ADR-REFUND (T-0140); this
  ticket consumes it. **Adding the idempotency-key parameter to `IStripeClient.RefundCheckoutSessionAsync`**
  is the T-0140-decided change executed here (it is the `nswag-regen` trigger if any response DTO shifts).
- **Re-deciding the authz model** — ADR-0001 (T-0100) governs; this ticket only adds `AdminOnly` rows.
- **Decomposing `CreateOrder.Handler`** (AUD-06) or the order-wizard facade (AUD-07) — Wave 3.
- **Payroll recalculation on reassign/cancel** (AUD-02/04 territory) — retroactive pay effects are not
  handled here; flag if a reassigned/cancelled order needs pay reversal.

## Implementation notes

- **Serialization clusters (TICKET-MAP):**
  - **`CreateOrder.cs` cluster** is `F2 → AUD-06 → TC-4` — AUD-01 does **not** edit `CreateOrder.cs`, so
    it is not in that cluster. **But** the children that touch `CancelOrder.cs` (the generalized-cancel
    child + the refund children) edit a money-path file ADR-REFUND consumers also touch; **serialize the
    AUD-01 children that edit `CancelOrder.cs` against each other** (do not run the cancel child and a
    refund child that both edit `CancelOrder.cs` concurrently) and against `D-01/DA-1/SEC-DSP-06/07`
    if/when that bundle reaches the same file.
  - **`PolicyBuilder.cs` / `Policy.cs` cluster** (head = T-0100): adding the four new `AdminOnly`
    permissions edits `Policy.cs` + `PolicyBuilder.Map`. This must land **after** T-0100 is `done`
    (fail-closed map + snapshot test in place) and the snapshot is updated **in the same PR** as the new
    rows. Do not run concurrently with another `Policy.cs`/`PolicyBuilder.cs` editor.
- **Governing ADRs:** **ADR-0001** (authorization — the four ops are `AdminOnly`, fail-closed, mapped;
  AC7) and **ADR-REFUND / T-0140** (the refund money path — single seam + deterministic idempotency key;
  AC3/AC6). Cite, do not re-decide either.
- **Routing (routing.md):** lock the contract first — architect confirms the split + the command/DTO
  shapes and the reassign domain method, then **db** if the `CancelledBy` enum needs a column/conversion
  change (the column already exists as `MaxLength(20)`; an enum-conversion may need an EF config change —
  flag `ef-migration` if so at split time), then **backend**, then **frontend**. Spawn a **reviewer per
  developer** instance, concurrent, same child ticket.
- **Manual steps:** `nswag-regen` is required — new Admin endpoints/DTOs change the admin OpenAPI surface;
  the owner regenerates `generate-admin-client` before the admin-app UI child can consume the clients.
  **Hold the frontend (AC8) child until the owner confirms the regen.** Watch for an additional
  `ef-migration` flag if the `CancelledBy` enum-to-column mapping requires a schema/conversion change.
- **TEST-FIRST (per `agents/knowledge/testing.md`):** refund money-math and authorization are on the
  strict red→green list. Write the failing tests **first** for each child — AC3/AC6 refund-idempotency
  (TC-7 / TC-IDEMP per T-0140 AC7), AC7 per-permission `AdminOnly`/deny (T-0100 harness pattern), AC2/AC4/
  AC5 handler tests — confirm they fail for the right reason, then implement to green. The status log of
  each child must note "red → green" and map each AC to a named test; an implementation-first PR fails
  Gate 6.
- **Security gate mandatory** (`security_touching: true`): the four ops are privileged write paths over
  money + assignment + lifecycle. Security reviews the `AdminOnly` gating, the no-double-refund property,
  and that no admin op leaks into a non-admin host/role.
