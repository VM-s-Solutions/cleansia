## Context
The payroll settlement lifecycle is **half-dead**: the domain entities already model the full
state machine, but most transitions have no command/controller/UI behind them, so payroll cannot be
corrected or fully settled in-system (incidents force manual DB edits). Source: `AUD-02` and `AUD-04`
in `agents/backlog/audits/AUDIT-2026-06-01-summary.md` (rows Â§2, lines 56 & 58) and the execution plan
`AUDIT-2026-06-01-execution-plan.md:165`; also the prior-audit confirmation at
`AUDIT-2026-06-01-findings.md:50-53`.

Concretely, the domain layer exposes transitions with **no command wiring**:
- `EmployeeInvoice.Dispute(adminNotes)` â€” `EmployeeInvoice.cs:230`
- `EmployeeInvoice.Reject(adminNotes)` â€” `EmployeeInvoice.cs:243`
- `EmployeeInvoice.UpdateAmounts(bonus, deduction, adminNotes)` (the bonus/deduction adjustment) â€” `EmployeeInvoice.cs:256`
- `PayPeriod.MarkAsPaid()` (period-level settlement) â€” `PayPeriod.cs:124`
- `PayPeriod.Reopen(notes)` â€” `PayPeriod.cs:137` (a `ReopenPayPeriod` command is absent; only `OpenPayPeriod` calls `Reopen` at `OpenPayPeriod.cs:59`).

The wired commands today are only Generate / Approve / MarkInvoicePaid / Cancel / Close
(`Features/EmployeePayroll/*`, `Features/PayPeriods/*`). The `PdfGenerationFailed` /
`PdfGenerationError` recovery state (`EmployeeInvoice.cs:46-51`) has no admin surface to find and
re-run failed PDFs.

**AUD-04 surface reconciliation:** the **Partner host** `EmployeePayrollController.cs` currently
exposes admin-only settlement operations directly to ordinary cleaners â€” `GenerateInvoice`
(`:67`), `ApproveInvoice` (`:79`), `MarkInvoicePaid` (`:91`), `CancelInvoice` (`:103`),
`ClosePayPeriod` (`:115`). Combined with the `BSP-1` fail-open policy fallback, this was the
audited payroll-fraud chain (`findings.md` BSP-1; `slice-reports.md:2769-2771`). The cleaner-facing
surface should be a **read-only "my period pay"** screen; the write operations belong on the Admin host.

This is why this ticket depends on **T-0100 (BSP-1: fail-closed PolicyBuilder + complete permission
map)** â€” the new `Policy.*` consts for these commands must land in the closed map first â€” and on
**T-0143 (F2-FULL outbox)** so any settlement side effects (invoice PDF gen / notification) dispatch
post-commit per ADR-0002 rather than inline on the money path. ADR-0001 (authorization model)
governs the admin-vs-partner host split and policy placement.

## Acceptance criteria
- [ ] **AC1 (adjustment)** â€” Given a `Pending`/`Disputed` `EmployeeInvoice`, When an admin submits a
  bonus/deduction adjustment via the new `UpdateInvoiceAmounts` command on the **Admin host**, Then
  `BonusAmount`/`DeductionAmount`/`TotalAmount` recompute exactly as `EmployeeInvoice.UpdateAmounts`
  (`EmployeeInvoice.cs:256-278`, clamped â‰¥ 0) and the change persists; When the invoice is `Paid`,
  Then the command returns a `BusinessResult` error (no mutation). A handler test proves both paths.
- [ ] **AC2 (invoice dispute/reject)** â€” Given a non-`Paid` invoice, When an admin calls the new
  `DisputeInvoice` / `RejectInvoice` commands, Then status moves to `Disputed` / `Rejected` with
  `AdminNotes` recorded (per `EmployeeInvoice.cs:230-254`); When the invoice is `Paid`, both are
  rejected with a `BusinessResult` error. Tests cover each transition + the guarded `Paid` case.
- [ ] **AC3 (period settlement)** â€” Given a `Closed` `PayPeriod`, When an admin calls the new
  `MarkPayPeriodPaid` command, Then `Status â†’ Paid` and `PaidAt` is set (per `PayPeriod.cs:124-135`);
  When the period is `Open`, the command errors. Given a non-`Paid` period, When an admin calls the
  new `ReopenPayPeriod` command, Then `Status â†’ Open` and `ClosedAt`/`ClosedBy` clear
  (`PayPeriod.cs:137-154`); a `Paid` period cannot be reopened. Tests prove the guards.
- [ ] **AC4 (failed-PDF recovery surface)** â€” Given an `EmployeeInvoice` with `PdfGenerationFailed =
  true`, When an admin opens the invoices list/detail, Then the failed state and
  `PdfGenerationError` are visible and a retry action invokes the existing `RegenerateInvoicePdf`
  command; on success `PdfGenerationFailed` clears (`EmployeeInvoice.cs:163-169`).
- [ ] **AC5 (partner surface reconciliation â€” AUD-04)** â€” Given an authenticated cleaner on the
  **Partner host**, When they hit any settlement write endpoint (`GenerateInvoice`,
  `ApproveInvoice`, `MarkInvoicePaid`, `CancelInvoice`, `ClosePayPeriod` â€” currently
  `EmployeePayrollController.cs:67,79,91,103,115`), Then it is **not reachable** (endpoint removed
  from the partner/mobile-partner hosts; admin-only on Admin host); When they call the read-only
  "my period pay" surface, Then they see only their own period pay (`EmployeeId` resolved from
  session, never the request body â€” same shape as `SEC-EMP-01`). A cross-user/cross-host rejection
  test proves it (harness from TC-AUTHZ-0 / T-0100).
- [ ] **AC6 (authz map closed)** â€” Given the new commands, When the host boots, Then every new
  `Policy.*` const has an explicit Admin-only entry in the closed PolicyBuilder map from T-0100 and
  `AssertComplete` passes (no fail-open fallback). Confirmed by the BSP-1 completeness assertion.
- [ ] **AC7 (admin UI)** â€” Given the admin app payroll screens, When an admin views an invoice /
  pay period, Then adjustment, dispute, reject, mark-period-paid, reopen, and retry-PDF actions are
  available via `<cleansia-*>`/PrimeNG controls (no raw HTML controls), all strings use
  `TranslatePipe` with keys present in all 5 locales (en/cs/sk/uk/ru), and every new
  `BusinessErrorMessage` key has an `errors.*` translation.
- [ ] **AC8 (partner/customer Android + web "my period pay")** â€” Given a cleaner in the partner
  Android app and the partner web app, When they open the payroll screen, Then they see a read-only
  period-pay summary scoped to themselves; no settlement write actions are present.
- [ ] **AC9 (side effects post-commit)** â€” Given any settlement transition that triggers a PDF
  regenerate or notification, When it runs, Then the side effect is dispatched via the T-0143 outbox
  post-commit (ADR-0002), not inline; the handler never calls `CommitAsync` itself.

## Implementation notes
- **Split before `ready` (L â†’ S/M).** This is `L`-sized and must NOT run as one ticket. Suggested
  split, all under the same source: (a) backend invoice-adjustment + dispute/reject commands; (b)
  backend pay-period MarkPaid + Reopen commands; (c) AUD-04 partner-surface reconciliation
  (remove admin endpoints off partner/mobile-partner hosts + read-only "my period pay" query); (d)
  admin UI; (e) partner web + Android read-only surface. The PM splits and re-sizes before any child
  goes `ready`.
- **Serialization clusters:** this ticket is **not** itself a member of any shared-file cluster in
  `TICKET-MAP.md` (those are PolicyBuilder, CleansiaStartupBase, UoW pipeline, host
  ServiceExtensions, dispute backend, CreateOrder, LoyaltyService). **But** it depends transitively
  on the **PolicyBuilder cluster** (`BSP-1 â†’ â€¦ â†’ SEC-EMP-01`) via T-0100 â€” do not start backend work
  until T-0100 is `done`, because the new `Policy.*` consts must be added to the already-closed map.
  Also coordinate the cross-host `EmployeePayrollController.cs` edits with **F1** (Wave 2) â€” both
  touch the payroll surface; serialize so they do not concurrently edit the partner/admin payroll
  controllers.
- **Governing ADRs:** ADR-0001 (authorization model) for the admin-vs-partner host split and Policy
  placement; ADR-0002 (outbox dispatch contract) for post-commit side effects (AC9).
- **TEST-FIRST.** Per `agents/knowledge/testing.md`, write the failing handler/domain-guard tests for
  AC1â€“AC3 and the cross-host authz-rejection test for AC5 **before** the production code, in the same
  merge. For a fix the bar is "the hole is closed + a test proves it."
- Follow CQRS conventions: each new operation is one command file with `Command` + `Handler` +
  `Validator` (FluentValidation, `Cascade.Stop`) + `Response`; handlers hold happy-path only, no
  `CommitAsync`, return `BusinessResult<T>`; DTOs are `record` types; error keys in
  `BusinessErrorMessage` with dot notation.
- Domain methods already exist and already guard `Paid` â€” reuse them, do not duplicate the guards in
  handlers: `EmployeeInvoice.Dispute/Reject/UpdateAmounts/MarkAsPaid/Cancel` (`EmployeeInvoice.cs`),
  `PayPeriod.MarkAsPaid/Reopen/Close` (`PayPeriod.cs`).
- `manual_steps`: **ef-migration** (any new persisted columns/index for adjustment audit fields, if
  the chosen design adds them â€” flag to owner; do not run `dotnet ef`), and **nswag-regen** (new
  admin/partner endpoints + DTOs change the OpenAPI surface â€” owner regenerates the partner/admin
  clients before frontend/Android consumer work begins; hold consumers until confirmed).
- Reconciled endpoints live on the Partner host at `Cleansia.Web.Partner/Controllers/EmployeePayrollController.cs`
  and `Cleansia.Web.Mobile.Partner/Controllers/EmployeePayrollController.cs`; the admin home is
  `Cleansia.Web.Admin/Controllers/AdminPayPeriodController.cs` / `AdminPayConfigController.cs`.
