## Context

Audit finding **DA-2** (`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:2063-2067`,
`AUDIT-2026-06-01-findings.json:2375-2389`, severity **critical**, functional GAP): the dispute
status machine is half-built. The domain models a full lifecycle —
`Pending/UnderReview/WaitingForResponse/Resolved/Closed/Escalated`
(`Cleansia.Core.Domain/Enums/DisputeStatus.cs:6-14`) — and the entity exposes intent-named
transition methods `Close` (`Dispute.cs:92-96`), `Escalate` (`:98-102`) and `LinkStripeDispute`
(`:104-108`), but **grep confirms zero callers** of those three methods in `src`. The only write
paths are:

- `Dispute.UpdateStatus(newStatus, updatedBy)` (`Dispute.cs:64-68`) — **free-sets any value with no
  transition guard**, and
- `Dispute.Resolve(...)` (`:82-90`).

`UpdateDisputeStatus.Handler` (`Cleansia.Core.AppServices/Features/Disputes/UpdateDisputeStatus.cs:45`)
calls `dispute.UpdateStatus(request.NewStatus, actorId)` unconditionally after the not-found guard,
so a dispute can illegally jump `Pending → Closed` skipping resolution, or be **re-opened after
`Resolved`**. The validator only checks `IsInEnum` (`:20-22`) — no legal-transition check exists.

This ticket **closes the correctness hole**: it makes `UpdateStatus` route through the intent-named
methods and enforce a legal-transition table, rejecting illegal transitions as a `BusinessResult`
failure (not a thrown exception, not a silent set). The legal transition graph and the
refund/chargeback money-path are frozen by **T-0140 (ADR-REFUND)** — this is a `depends_on` and must
be `accepted` first (T-0140 AC5 hands the `charge.dispute.*` / `LinkStripeDispute` *webhook wiring*
to the Wave-2 `D-06` ticket; T-0140 AC6 names the admin-only authz gate). This ticket implements the
in-process transition guard only; it does **not** add the Stripe chargeback webhook case.

`security_touching: true` — this is an authorization/lifecycle integrity surface (illegal state
transitions on a money-bearing entity); the Security gate is mandatory per
`agents/process/quality-gates.md`.

## Acceptance criteria

- [ ] **AC1 — Legal transitions succeed.** Given a `Dispute` in state X and a legal target state Y
  per the transition table frozen by T-0140 (ADR-REFUND), When `UpdateDisputeStatus.Handler.Handle`
  is invoked by an actor holding `Policy.CanUpdateDisputeStatus` (Admin), Then the dispute's `Status`
  becomes Y, `Updated(actorId, …)` is stamped, and the handler returns `BusinessResult.Success()`.
  A unit test in `Cleansia.Tests` proves each legal edge.

- [ ] **AC2 — Illegal transitions are rejected (the hole is closed).** Given a `Dispute` in a
  terminal/incompatible state (e.g. `Resolved` → re-open to `Pending`, or `Pending` → `Closed`
  skipping resolution), When `UpdateDisputeStatus.Handler.Handle` is invoked, Then the status is
  **unchanged** and the handler returns `BusinessResult.Failure` with a dedicated
  `BusinessErrorMessage` code (new `dispute.invalid_status_transition`, added in
  `Cleansia.Core.AppServices/Common/BusinessErrorMessage.cs:196-201` next to the other `dispute.*`
  codes). A unit test asserts the failure `Error.Message` equals the new constant for each illegal
  edge — no exception is thrown, no `CommitAsync` side effect occurs.

- [ ] **AC3 — Transition logic lives in the domain entity.** Given the half-built methods
  `Close`/`Escalate` (`Dispute.cs:92-102`), When the change lands, Then `Dispute.UpdateStatus`
  enforces the legal-transition table inside the domain entity (returning a result/failure the
  handler maps, OR exposing a guarded transition the handler consumes — per the shape T-0140 freezes),
  so `Close` and `Escalate` are **reachable** through the existing `UpdateDisputeStatus` write path.
  Grep for callers of `Dispute.Close`/`Dispute.Escalate` now returns the production path (not just
  tests).

- [ ] **AC4 — `Resolve` remains the only writer of resolution fields.** Given `Dispute.Resolve`
  (`:82-90`) sets `RefundAmount`/`ResolvedBy`/`ResolvedOn`/`ResolutionNotes`, When `UpdateStatus` is
  used to reach a non-`Resolved` state, Then it does **not** mutate those resolution fields, and a
  test confirms `UpdateStatus` cannot be used to set `Status = Resolved` (resolution stays owned by
  `ResolveDispute` / `Dispute.Resolve`) so the two money-path writers do not overlap (consistent with
  T-0140 AC3).

- [ ] **AC5 — Authorization unchanged and asserted.** Given `UpdateStatus` is gated by
  `Policy.CanUpdateDisputeStatus` (Admin-only, `Policy.cs:105`,
  `Cleansia.Web.Partner/Controllers/DisputeController.cs:79-84`), When the change lands, Then the
  policy mapping is unchanged and an integration test in `Cleansia.IntegrationTests` confirms a
  non-admin caller is rejected (per `agents/knowledge/testing.md` must-cover #5). This ticket does
  **not** re-home the endpoint to the Admin API (that is the DA-1 / D-01 bundle).

- [ ] **AC6 — Tests predate the implementation (test-first).** Given the must-cover #2 (order/state
  transitions, incl. the illegal ones that must be rejected) in `agents/knowledge/testing.md`, When
  the diff/commits are reviewed, Then the transition unit tests appear **before/with** the
  implementation, the status log records "red → green", and every legal+illegal edge from the T-0140
  table maps to a test case (Reviewer Gate 6).
