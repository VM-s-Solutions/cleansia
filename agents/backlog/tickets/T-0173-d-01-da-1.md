## Context

Disputes can be filed by customers but **cannot be worked by support/admin through the Admin app**.
The only resolve/escalate/close/status path today is mounted on the **Partner API** behind admin-only
policies, where it is unreachable by the wrong actor:

- `src/Cleansia.Web.Partner/Controllers/DisputeController.cs:66-88` exposes `Resolve`
  (`Policy.CanResolveDispute`, AdminOnly) and `UpdateStatus` (`Policy.CanUpdateDisputeStatus`,
  AdminOnly) on the **Partner host** â€” admin-policied endpoints on a partner-authenticated surface
  (**SEC-DSP-07 / DA-1**: admin operations are unreachable; partner duplicates of
  Create/GetById/GetPaged/AddMessage at lines 18-64 are dead/inconsistent).
- There is **no** `src/Cleansia.Web.Admin/Controllers/*Dispute*.cs` and **no**
  `libs/cleansia-admin-features/**/dispute*` (verified 2026-06-01) â€” the resolution workflow has no
  product surface at all (**D-01 / DA-1**, critical gap).
- Dispute resolution **records a `RefundAmount` but issues no refund**: `ResolveDispute.Handler`
  (`src/Cleansia.Core.AppServices/Features/Disputes/ResolveDispute.cs:43-60`) calls
  `dispute.Resolve(...)` which only sets `RefundAmount`/`Status=Resolved`
  (`src/Cleansia.Core.Domain/Disputes/Dispute.cs:82-90`) â€” no Stripe call, no idempotency
  (**SEC-DSP-06**, half-built money path; "the admin UI must not present 'refund' as if it pays out").

This is the **Wave-2 consumer** of `ADR-REFUND` (**T-0140**): the refund money-path contract (single
refund seam, deterministic idempotency key, `RefundAmount` consumption, admin-only authz, dispatch via
`IPendingDispatch`) is frozen there and must be in `accepted` state before this ticket goes `ready`.
The authorization model is `ADR-0001` (**T-0100 / BSP-1**) â€” the `Policy.*` gates that may resolve /
refund a dispute live in the completed `PolicyBuilder` map.

Sources: audit findings `D-01`, `DA-1`, `SEC-DSP-06`, `SEC-DSP-07`
(`agents/backlog/audits/AUDIT-2026-06-01-findings.json`); user stories `US-admin-0042` (chargeback
reconciliation â€” context only) and the customer dispute story `US-customer-0007`.

## Acceptance criteria

- [ ] **AC1 â€” Admin host owns dispute management.** Given an authenticated admin, When they call the
  new `src/Cleansia.Web.Admin/Controllers/DisputeController` (GetPaged / GetById / Resolve /
  UpdateStatus / AddMessage-as-staff, following the order-management / invoice-management archetype),
  Then each action is gated by its existing `Policy.*` (`CanViewDisputeList`, `CanViewDispute`,
  `CanResolveDispute`, `CanUpdateDisputeStatus`, staff `AddMessage` per SEC-DSP-01's split) resolved
  from the ADR-0001 `PolicyBuilder` map, and an authz integration test (TC-AUTHZ family) proves a
  non-admin caller is rejected (403/NotFound, never 200).

- [ ] **AC2 â€” Dead Partner endpoints removed.** Given the Partner host, When the diff is reviewed,
  Then `Resolve` and `UpdateStatus` are **removed** from
  `src/Cleansia.Web.Partner/Controllers/DisputeController.cs:66-88`, the duplicated customer-policied
  Create/GetById/GetPaged endpoints (lines 18-52) are removed (no deliberate partner-comment scope is
  defined here â€” see Out of scope), and a test asserts those routes no longer exist on the Partner API
  (SEC-DSP-07 closed: no admin-policied action mounted on a partner-authenticated host).

- [ ] **AC3 â€” Resolve issues a real, idempotent refund.** Given an admin resolves a dispute with a
  non-null `RefundAmount`, When `ResolveDispute.Handler` runs, Then it issues the Stripe refund through
  the single refund seam frozen in ADR-REFUND (T-0140) using the **deterministic idempotency key**
  (keyed on dispute/order identity), `Dispute.RefundAmount` and the dispute/payment status transitions
  are written exactly as the ADR specifies, and a TC-7 (refund money-math) + idempotency test proves a
  retried `Resolve` does **not** double-refund (SEC-DSP-06 closed; today the handler at
  `ResolveDispute.cs:43-60` makes no Stripe call).

- [ ] **AC4 â€” Status transitions are guarded.** Given a dispute in a given `DisputeStatus`, When admin
  calls `UpdateStatus`/Resolve/Escalate/Close, Then illegal jumps (e.g. Resolved â†’ Pending,
  re-resolving an already-Resolved dispute and overwriting its refund) are rejected with a
  `BusinessResult` failure, and legal transitions route through the intent-named domain methods
  (`Dispute.Close`/`Escalate`/`Resolve`, `Dispute.cs:82-102`) rather than the free `UpdateStatus` set
  at `Dispute.cs:64-68` / `UpdateDisputeStatus.cs:45`. (Transition-table semantics are owned by DA-2 /
  T-mapped sibling â€” see Out of scope; this ticket consumes the guard, it does not define the legal
  graph if DA-2 lands first.)

- [ ] **AC5 â€” Admin dispute UI exists.** Given the admin app, When an admin opens the new
  `libs/cleansia-admin-features` disputes-management feature, Then a list (C-section list archetype,
  facade + signals, `<cleansia-*>`/PrimeNG only, OnPush) and a detail/resolution panel render via the
  generated admin client; the resolve panel only presents "issue refund" once AC3 is wired (it must
  not imply a payout that does not happen); and every user-visible string has a `TranslatePipe` key
  present in all 5 locales (`en, cs, sk, uk, ru`).

- [ ] **AC6 â€” No PII / contract regression on the new admin surface.** Given the shared dispute DTOs,
  When the admin list/detail render, Then the admin surface does not introduce a PII leak beyond what
  the (separately-tracked) SEC-DSP-03/08 DTO decisions allow, and the refund-success notification (if
  any) is dispatched via `IPendingDispatch` per ADR-0002 D1 â€” not a direct `IQueueClient.SendAsync`
  from the resolve handler (cf. the existing direct send in `AddDisputeMessage.cs:67-77`).
