## Context

Finding **D-06** (Wave 2, source story `US-admin-0042` —
`agents/backlog/stories/AUDIT-2026-06-01-user-stories.md:91-124`): chargeback linkage is **half-built
dead code**. The `Dispute` aggregate exposes `LinkStripeDispute(stripeDisputeId, updatedBy)`
(`Cleansia.Core.Domain/Disputes/Dispute.cs:104-108`) and persists a nullable `StripeDisputeId`
column (`Dispute.cs:38`; `DisputeEntityConfiguration.cs:42`, `varchar(100)`), but **nothing calls
it**. `Constants.StripeEventType` (`Cleansia.Core.AppServices/Common/Constants.cs:21-54`) has no
`charge.dispute.*` member, so a real bank chargeback (`charge.dispute.created`) hits the webhook,
matches the `_ => BusinessResult.Success(orderId)` fall-through in the event switch
(`HandlePaymentNotification.cs:201-213`), and is silently ignored. Finance ends up with a real money
movement that links to no internal `Dispute`/`Order`.

The money-path contract that governs this wiring is frozen by **ADR-REFUND** (ticket `T-0140`, AC5),
which hands the chargeback-handler *implementation* to this ticket. `T-0140` must be `done`
(ADR `accepted`) before this goes `ready`. This ticket implements the producer side only — it does
**not** issue refunds or respond to Stripe.

A correlation gap noted in the story is real and must be closed here: `charge.dispute.*` events carry
`charge` + `payment_intent` but **no order metadata**, so the existing `ExtractOrderId` metadata path
cannot resolve them. `IOrderRepository` has no lookup-by-payment-intent method
(confirmed: no `PaymentIntent`/`GetByStripe*` member in
`Cleansia.Core.Domain/Repositories/IOrderRepository.cs`), so a new repository query is required.

## Acceptance criteria

- [ ] **AC1 — New chargeback created when order has no open dispute.** Given Stripe sends
  `charge.dispute.created` whose charge resolves (via `payment_intent` → `Order.StripePaymentIntentId`)
  to an `Order` with **no** open dispute, When the webhook is processed, Then a new `Dispute` is
  created for that order and `LinkStripeDispute(dp_…, "stripe-webhook")` records the Stripe dispute id,
  with `Status` set to reflect Stripe's dispute status — proven by a handler test asserting the
  persisted `StripeDisputeId` and `Status`.

- [ ] **AC2 — Existing open dispute is linked, never stacked.** Given a `charge.dispute.created` whose
  order **already** has an open platform dispute (`GetOpenDisputeForOrderAsync` returns non-null,
  `IDisputeRepository.cs:19`), When processed, Then the existing dispute is linked via
  `LinkStripeDispute` and **no** second dispute is created — mirroring the `DisputeAlreadyExists` guard
  (`CreateDispute.cs:54-60`). A test asserts the dispute count stays 1 and the existing row carries the
  Stripe id.

- [ ] **AC3 — Idempotent on redelivery (S7).** Given the same Stripe event id is delivered more than
  once, When the webhook is processed again, Then the existing `ProcessedStripeEvents` gate
  (`HandlePaymentNotification.cs:144-159`) short-circuits — no duplicate dispute is created and no field
  is re-mutated. A test replays the event and asserts no second side effect (S7,
  `agents/knowledge/security-rules.md:83-91`).

- [ ] **AC4 — Status reflected on update/closed events.** Given a subsequent `charge.dispute.updated`
  or `charge.dispute.closed` event for an already-linked Stripe dispute id, When processed, Then the
  matching `Dispute` is found by its `StripeDisputeId` and its `Status` is updated to reflect Stripe's
  current status (e.g. won → Resolved-class, lost → Closed/Escalated-class) per the mapping frozen in
  ADR-REFUND — without creating a new dispute. A test drives created→updated→closed and asserts the
  final status.

- [ ] **AC5 — Uncorrelated charge is a no-op success (S6).** Given a `charge.dispute.*` event whose
  charge resolves to no local `Order` (no matching `StripePaymentIntentId`), When processed, Then the
  handler returns success and logs at most a warning — with no PII / no raw Stripe payload above Debug
  level (S6, `security-rules.md:78-81`) — so Stripe does not retry indefinitely. A test asserts success
  with no dispute written.

- [ ] **AC6 — Invalid signature still rejected.** Given an invalid webhook signature, When the event
  is received, Then the request is rejected exactly as the existing path does
  (`HandlePaymentNotification.cs:128-134`) and no dispute is created. Covered by the existing
  signature-rejection test path (regression assertion).

- [ ] **AC7 — Tests prove the hole is closed (test-first).** Given `agents/knowledge/testing.md` TDD
  red-green discipline, When the change is reviewed, Then handler tests for AC1–AC6 exist, were written
  before the production wiring, and pass — and the `charge.dispute.*` branch did not exist before this
  ticket.

## Implementation notes

- **Governing ADRs:** ADR-REFUND (`T-0140`, AC5) freezes *which* `charge.dispute.*` events we act on,
  how an event resolves to a `Dispute`/`Order`, when `LinkStripeDispute` is called, and the Stripe-
  status → `DisputeStatus` mapping — follow it, do not re-decide. ADR-0001 (authorization: this is a
  signed-webhook ingress, not a user-authz path) and ADR-0002 (post-commit dispatch: any chargeback
  notification side effect goes through `IPendingDispatch`, never a direct `IQueueClient.SendAsync`
  from this handler) are also in force.
- **Build TEST-FIRST** per `agents/knowledge/testing.md` ("write the failing test first"; webhook /
  money-path logic is on the red-green list). Author the AC1–AC6 handler tests before the wiring.
- **Wiring anchors (verified 2026-06-01):**
  - Add `charge.dispute.created` / `.updated` / `.closed` consts + an `IsChargebackEvent(...)` predicate
    to `StripeEventType` (`Cleansia.Core.AppServices/Common/Constants.cs:21-54`).
  - Branch `charge.dispute.*` *before* the `ExtractOrderId` metadata lookup in
    `HandlePaymentNotification.cs` (these events have no order metadata; the `_ =>` fall-through at
    `:212` currently swallows them). Keep it after the idempotency gate (`:144-159`) so S7 holds.
  - **New repository query required:** add a lookup-by-payment-intent to `IOrderRepository`
    (`Cleansia.Core.Domain/Repositories/IOrderRepository.cs` — confirmed no such method today) +
    `OrderRepository`, to correlate `charge.payment_intent` → `Order.StripePaymentIntentId`
    (`Order.cs:103`). Use the tenant-ignoring read pattern already used at
    `HandlePaymentNotification.cs:184` (`GetByIdIgnoringTenantAsync`), then set the tenant override
    (`:193-196`) before writing.
  - Find-by-Stripe-id for AC4: add a `GetByStripeDisputeIdAsync(...)` to `IDisputeRepository`
    (`IDisputeRepository.cs`) / `DisputeRepository`; reuse `GetOpenDisputeForOrderAsync` (`:19`) for
    AC2 and mirror the `DisputeAlreadyExists` guard shape from `CreateDispute.cs:54-60`.
  - Domain methods already exist — call `Dispute.LinkStripeDispute` (`Dispute.cs:104-108`) and
    `UpdateStatus`/`Resolve`/`Close`/`Escalate` (`Dispute.cs:64-102`); do **not** re-implement persistence
    (no `CommitAsync` in the handler — UnitOfWork pipeline owns commit).
- **Security gate (security_touching=true):** signed Stripe ingress + new write path. Enforce S6
  (no PII / raw payload above Debug, `security-rules.md:78-81`, AC5) and S7 (idempotent side effects via
  the existing `ProcessedStripeEvents` gate, `:83-91`, AC3). Security review required before `done`.
- **Serialization cluster:** This ticket edits `HandlePaymentNotification.cs` + `Constants.cs` +
  `IOrderRepository`/`OrderRepository` + `IDisputeRepository`/`DisputeRepository` — **none** of these
  are in a TICKET-MAP shared-file serialization cluster, so it is **not** cluster-constrained. Note: the
  dispute-backend cluster (`SEC-DSP-01 → DA-2 → D-01 bundle`, per TICKET-MAP) touches
  `AddDisputeMessage.Handler` + dispute controllers, **not** these files — no overlap — but coordinate
  with `DA-2` (dispute transition guards) so the AC4 status transitions stay reachable.
- **No manual step:** the `StripeDisputeId` column already exists in the model snapshot/migrations — no
  EF migration needed; this is backend-only with no DTO surface exposed to clients, so no NSwag regen.
