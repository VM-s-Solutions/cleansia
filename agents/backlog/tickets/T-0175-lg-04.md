## Context

`MembershipPlan` is a complete domain entity â€” `Create` / `UpdatePricing` / `UpdateBenefits` /
`Deactivate` (`src/Cleansia.Core.Domain/Memberships/MembershipPlan.cs:122-167`), `ITenantEntity`,
soft-deleted via `IsActive` â€” seeded with two plans carrying **placeholder Stripe price ids**
(`sql-scripts/insert_seed_data.sql:2436-2470`, comment: *"replace with the actual Price ids â€¦ before
deploying"*). Yet there is **no admin surface to manage it**: no `Features/Memberships/Admin/*`, no
`AdminMembershipController` (confirmed absent from `src/Cleansia.Web.Admin/Controllers/` â€” only
`AdminLoyaltyController`, `AdminLoyaltyTierController`, `AdminPromoCodeController` exist), and no admin
sidebar entry. `IMembershipPlanRepository` itself documents the gap: *"admins manage future plans
through a back-office surface that doesn't exist yet"* (`IMembershipPlanRepository.cs:5-10`); today it
exposes only `GetByCodeAsync` + `GetActivePlansAsync`.

Consequence: plans, prices, discount %, trial length, free-cancellation window, express-upgrade flag,
and the critical `StripePriceId` can only be changed by hand-editing the production database. An admin
cannot launch a price change, run a promo trial, deactivate a retired plan, or fix the placeholder
Stripe ids without engineering + raw SQL.

Source: audit finding **LG-04** (major GAP) â€” `audits/AUDIT-2026-06-01-findings.json` and
`audits/AUDIT-2026-06-01-slice-reports.md:1879-1883`; story **US-admin-0014** "Manage Membership Plans
from the Admin back office" (`stories/AUDIT-2026-06-01-user-stories.md:4002-4069`). Governed by
**ADR-0001** (ADR-AUTHZ) â€” new admin permissions must be mapped fail-closed, which is why this ticket
depends on **T-0100** (the PolicyBuilder fail-closed default + `AssertComplete` + shared host
registration must land first, or the new `Policy.*` constants brick the boot via AssertComplete or
fall through to a non-deny default).

## Acceptance criteria

- [ ] **AC1 (plan list, permission-gated)** â€” Given an authenticated admin holding the membership-view

  (`*cleansiaPermission`) nor can reach the route, and the backend returns 403. Evidence: new
  `GetPagedMembershipPlans` query returning `PagedData<MembershipPlanListItem>`; `[Permission(...)]`-
  guarded `get-paged` endpoint; per-permission gating test.
- [ ] **AC2 (create with explicit Stripe Price id)** â€” Given the admin has registered the Stripe
  Product/Price out of band and copied the Price id, When they submit a unique Code, Name, price,
  billing interval, discount %, free-cancellation window, trial days, express-upgrade flag, and

  customer plan switcher (`GetMembershipPlans` â†’ `GetActivePlansAsync`) immediately offers it with no
