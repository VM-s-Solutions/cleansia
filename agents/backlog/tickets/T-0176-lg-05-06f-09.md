## Context

Three grounded loyalty-referral gaps from the 2026-06-01 audit, bundled because they live on the same
admin-referral surface (`AdminReferralController` + the `loyalty-referrals` admin lib + the admin
sidebar/routes) and ship as one operator-facing capability.

**LG-06 (major / gap â€” no operational controls).** `findings.json:454-468`,
`slice-reports.md:1891-1895`. The `Referral` aggregate exposes `MarkQualified`/`MarkExpired` domain
methods (`src/Cleansia.Core.Domain/Loyalty/Referral.cs:96-119`), but the **only** callers are
`ReferralService` acting as the system actor (order-complete / sweep). `AdminReferralController`
(`src/Cleansia.Web.Admin/Controllers/AdminReferralController.cs`) is **read-only** â€” `get-paged` +
`by-user` â€” and the admin referrals UI (`referrals-list.component` + `referrals-list.facade.ts`) has
filters and a table but **no row actions**. There is no command to reverse a fraudulent `Qualified`
referral (and claw back the symmetric point grants recorded on the row,
`Referral.PointsAwardedToReferrer`/`PointsAwardedToReferred`, `Referral.cs:41-43`) nor to manually
qualify a legitimate one stuck in `Accepted`. Self-referral rings cannot be remediated in-product;
refunded/disputed qualifying orders have no admin remedy.

**LG-05 (major / gap â€” orphaned endpoint).** `findings.json:439-453`, `slice-reports.md:1885-1889`.
`AdminReferralController.GetReferralsByUser` (`by-user/{userId}` â†’ `GetReferralsByUser.Query`,
`AdminReferralController.cs:37-50`; handler returns `AsReferrer`/`AsReferred` lists,
`GetReferralsByUser.cs:15-67`) is live and the generated admin client exposes `.byUser(userId)`
(`admin-client.ts:9445`), but grep across `libs/cleansia-admin-features` returns **no consumer** â€”
only the generated client references it. The referrals facade calls only `getPaged`
(`referrals-list.facade.ts:36`). Admins investigating a specific user (fraud, dispute, "did my
friend's referral count?") cannot see that user's referral relationships in-app despite the backend
being ready.

**LG-09 (minor / discoverability).** `findings.json:498-506`, `slice-reports.md:1913-1917`. The
`loyalty/users` admin route â†’ `loyalty-user-detail` lib (`app.routes.ts:162-169`) is reachable only
by deep-link from the user-management screen; the Loyalty sidebar group (`app.component.ts:100-120`)
lists Promo Codes, Tiers, Referrals â€” no user-loyalty entry. Defensible as a drill-in, but there is
no standalone "look up a user's loyalty/referrals" entry. Bundled here as the natural home for the
LG-05 by-user panel.

Audit ranking placed LG-06 > LG-05 > LG-09 (`slice-reports.md:1936/1935/1939`).

## Acceptance criteria

- [ ] **AC1 (LG-06 reverse)** â€” Given a referral in `Qualified` status that granted points to both
  parties, When an admin invokes the new `ReverseReferral` command, Then both sides' granted points
  are revoked via the loyalty manual-revoke path (`ILoyaltyService.RevokePointsManuallyAsync`,
  `LoyaltyService.cs:213`) using the points recorded on the row
  (`Referral.PointsAwardedToReferrer`/`PointsAwardedToReferred`), the referral moves to a terminal
  status, and the admin's supplied reason is persisted on the resulting ledger rows. A test proves
  both ledger revocations land and the row is terminal.
- [ ] **AC2 (LG-06 force-qualify)** â€” Given a referral stuck in `Accepted` that the admin determines
  is legitimate, When the admin invokes `ForceQualifyReferral`, Then `Referral.MarkQualified` runs
  with the admin as actor and the symmetric grants are applied through the loyalty grant path
  (`GrantPointsManuallyAsync`, `LoyaltyService.cs:181`). A test proves the grants land and the status
  is `Qualified`. (If the audit's "force-qualify" is judged out of an M-sized scope by the architect,
  it may be deferred â€” see Out of scope â€” but the reverse/clawback path AC1 is mandatory.)
- [ ] **AC3 (LG-06 idempotency)** â€” Given the same `ReverseReferral`/`ForceQualifyReferral` request is
  submitted twice (retry), When both are handled, Then the points effect lands **exactly once** and a
  second invocation on an already-reversed/already-qualified row is a no-op or a guarded business
  error â€” never a double clawback / double grant. This preserves the ADR-0002 idempotency contract on
  the manual grant/revoke path reworked by T-0112 and consumed via T-0148. A test proves run-twice
  safety.
- [ ] **AC4 (LG-06 authz)** â€” Given the new referral-intervention endpoints, When a caller without the
  new admin permission hits them, Then they receive 403; the new `Policy.*` constant (e.g.
  `CanInterveneReferral`) is mapped **AdminOnly** in `PolicyBuilder.Map` and added to the checked-in
  permission-map snapshot in the **same PR** (per ADR-0001 D2 + the additive-row rule, ADR-0001
  Consequences/C4). `PolicyBuilder.AssertComplete()` still passes at boot (no unmapped permission). A
  test asserts the mapping and a non-admin gets 403.
- [ ] **AC5 (LG-05 wire by-user)** â€” Given an admin on the user-loyalty / referrals surface, When they
  open a specific user, Then the UI consumes `adminReferralClient.byUser(userId)` and renders the
  user's referral relationships (the `AsReferrer` and `AsReferred` lists from
  `GetReferralsByUser.Response`) in two stacked tables. The `.byUser(` call now has at least one
  consumer in `libs/cleansia-admin-features` (it currently has none outside the generated client).
- [ ] **AC6 (LG-06 UI actions)** â€” Given the referrals admin table, When an admin views a row, Then
  reverse (and, if in scope, force-qualify) row actions are available, gated by the new permission,
  with a reason input, and on success the table refreshes to show the new status. The reason field is
  required (mirrors the manual grant/revoke `Reason` contract surfaced by T-0148). All new
  user-visible strings use `TranslatePipe` with keys present in all 5 locales
  (en/cs/sk/uk/ru); any new backend error key has a matching `errors.*` entry.
- [ ] **AC7 (LG-09 sidebar)** â€” Given the admin sidebar Loyalty group (`app.component.ts:103-119`),
  When the admin opens it, Then a discoverable entry leads to the user-loyalty / referral lookup
  surface (the natural home for the LG-05 by-user panel) â€” closing the discoverability gap. The label
  uses a `sidebar.*` translation key present in all 5 locales.
- [ ] **AC8 (test-first)** â€” All backend behavior (AC1-AC4) is proven by unit/integration tests in
  `Cleansia.Tests`, written **test-first** (red â†’ green, visible in commit/diff order per
  `agents/knowledge/testing.md`): the clawback test (AC1), the authz/map test (AC4), and the
  run-twice idempotency test (AC3) exist and pass.
