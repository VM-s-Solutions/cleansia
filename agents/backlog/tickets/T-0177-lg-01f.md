## Context

Audit finding **LG-01 / LG-01f** (CRITICAL gap): the referral-expiry sweep is implemented but **never
invoked**, so referrals never expire automatically.

`ReferralService.ExpireStaleReferralsAsync` (`src/Cleansia.Core.AppServices/Services/ReferralService.cs:236`)
and its interface declaration (`src/Cleansia.Core.AppServices/Services/Interfaces/IReferralService.cs:74`)
have **zero callers** anywhere in the solution — grep returns only the definition and the interface
line. The 90-day qualifying window is therefore only enforced *lazily* inside `ProcessOrderCompletedAsync`
(`ReferralService.cs:180-185`, `referral.MarkExpired(SystemActor)`), which fires **only** when the
invitee completes an order. A referred user who signs up and never books leaves the `Referral` row stuck
in `Accepted` forever; the admin "Expired" filter never receives rows through the intended automatic
path, and referral conversion/expiry reporting is permanently wrong.

The fix mirrors the existing daily-sweep pattern `SendMembershipLifecycleNotificationsFunction`
(`src/Cleansia.Functions/Functions/SendMembershipLifecycleNotificationsFunction.cs:23-43`), which is a
`[TimerTrigger]` Function that dispatches a MediatR `Command` so the work commits through the UoW
pipeline (the established pattern — the Function does **not** resolve the service directly). The repo
side already exists and is correct: `ReferralRepository.GetExpirableAsync`
(`src/Cleansia.Infra.Database/Repositories/ReferralRepository.cs:50-56`) filters
`Status == ReferralStatus.Accepted && AcceptedOn < cutoff`.

Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:1857-1861` (LG-01),
`agents/backlog/audits/AUDIT-2026-06-01-findings.json` (id `LG-01`).

## Acceptance criteria

- [ ] **AC1** — Given a `Referral` in `Accepted` whose `AcceptedOn` is older than
  `ReferralPolicy.QualifyingWindowDays` (90), When the referral-expiry sweep runs, Then the referral is
  transitioned to `Expired` (via `Referral.MarkExpired(SystemActor)`) and the change is **committed**.
- [ ] **AC2** — Given a `Referral` in `Accepted` whose `AcceptedOn` is **within** the qualifying window,
  When the sweep runs, Then the referral is left untouched (still `Accepted`) — i.e. the sweep honors the
  exact `GetExpirableAsync` cutoff (`ReferralRepository.cs:54`).
- [ ] **AC3** — Given a referral already in a terminal state (`Qualified` / `Expired`), When the sweep
  runs, Then no transition is attempted (no-op), proving the sweep is **safe to run twice** (idempotent
  — S7 / `testing.md` must-cover #6).
- [ ] **AC4** — A new `[TimerTrigger]` Azure Function exists in `src/Cleansia.Functions/Functions/`
  (mirroring `SendMembershipLifecycleNotificationsFunction`) that invokes the sweep on a recurring
  cadence via `IMediator.Send(...)`, so the work commits through the UoW pipeline (ADR-0002 D1.3: the
  Functions host runs the full MediatR pipeline). On dispatch the Function logs trigger time and result;
  failures are logged at error, not swallowed silently.
- [ ] **AC5** — `ExpireStaleReferralsAsync` (or its command wrapper) now has at least one production
  caller (the new Function) — `agents/tools` consistency grep for the method shows a non-test, non-
  definition reference.
- [ ] **AC6 (tests, TEST-FIRST)** — Unit tests written **before** the implementation cover AC1–AC3
  against mocked `IReferralRepository` (`GetExpirableAsync`): expirable referrals get `MarkExpired` +
  commit asserted; in-window referrals untouched; a second run over already-terminal referrals produces
  no further effect. Each test maps to an AC and predates the code in the diff/commit order (per
  `agents/knowledge/testing.md` TDD loop).

## Implementation notes

- **Depends on T-0143 (F2-FULL).** Per ADR-0002 D1.3, F2-FULL must decide whether the Functions host
  gets the post-commit dispatch behavior / drainer. The sweep itself only writes a domain transition
  (`MarkExpired`) with **no queue side-effect**, so the contract concern is purely that the commit runs
  through the pipeline — invoke via `IMediator.Send(Command)` (the `SendMembershipLifecycleNotifications`
  shape), not a raw service call, so the UoW pipeline commits it. Do **not** call `CommitAsync()` in the
  handler (patterns-backend rule).
- **Governing ADR: ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`) — the Functions
  host runs the full MediatR pipeline (D1.3); a sweep is a Bucket-B-adjacent producer but here has no
  enqueue, so no `IPendingDispatch` / `QueueEnvelope` is needed (it only does a domain write).
- **Serialization cluster:** not listed in the `TICKET-MAP.md` shared-file map. No file overlap with the
  `LoyaltyService.cs` cluster (this touches `ReferralService` + a new Function file). It may add a thin
  `ExpireStaleReferrals.Command`/handler under `Features/Referrals` and a new Function file — both new
  files, no concurrent-edit collision. Confirm at intake that no sibling LG ticket is concurrently
  editing `ReferralService.cs`; if so, serialize behind it.
- **Built TEST-FIRST** per `agents/knowledge/testing.md` (idempotency is must-cover #6; the test
  predates the implementation in commit order and the status log notes red→green).
- Key files: sweep `ReferralService.cs:236-251`; cutoff `ReferralRepository.cs:50-56`; mirror Function
  `SendMembershipLifecycleNotificationsFunction.cs:23-43`; lazy-expiry branch `ReferralService.cs:180-185`.
- No `manual_steps`: no EF migration (no schema change), no NSwag regen (queue/Function contract is
  internal — no API surface change).
