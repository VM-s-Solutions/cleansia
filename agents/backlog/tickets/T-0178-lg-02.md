## Context
Web referral share links dead-end in a 404, breaking the customer-acquisition viral loop on web.

The rewards page builds and shares a public link of the form `/r/{code}`
(`src/Cleansia.App/libs/cleansia-customer-features/rewards/src/lib/rewards/rewards.component.ts:207-209`
— `referralUrl()` returns `${window.location.origin}/r/${code}`, consumed by `shareReferralCode()` at
:219 and the share text at :222). But the customer app has **no route that matches `r/...`**: the
trailing wildcard `apps/cleansia.app/src/app/app.routes.ts:141-144` (`path: '**' → redirectTo:
NOT_FOUND`) swallows it, so every shared link 404s. The route enum the share URL should track
(`libs/core/services/src/lib/enums/routes.enum.ts:36-54`) has no referral-landing member.

The downstream signup machinery already exists and only needs to be *fed*: `RegisterFacade`
(`libs/cleansia-customer-features/register/src/lib/register/register.facade.ts`) already owns a
`referralCode` signal (:47), a `referralCode` form control (:167), `validateReferralCodeNow()` (:55),
the `setReferralCode()` mirror (:93-96), and passes the code into `authService.register(...)` (:110) —
but **nothing reads a URL param to pre-fill it**. The only current entry point is the manual "Add a
referral code" dialog. The backend already accepts the code fail-soft (a bad code never blocks signup),
so **no backend change is required**.

Source: finding LG-02 (`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:1863-1867`); story
`US-customer-0042` (`agents/backlog/stories/AUDIT-2026-06-01-user-stories.md:5544-5625`).

## Acceptance criteria
- [ ] **AC1 — Route resolves instead of 404** — Given a referral URL `/r/{code}` produced by
  `referralUrl()` (`rewards.component.ts:207`), When an unauthenticated visitor opens it, Then the app
  routes to the registration experience (NOT the `not-found` page reached via the `**` wildcard at
  `app.routes.ts:141-144`), and the share URL and the new route are defined from a single source — a new
  `CleansiaCustomerRoute` member in `routes.enum.ts:36-54` — so they cannot drift.
- [ ] **AC2 — Code captured and pre-applied** — Given the visitor landed via `/r/{code}`, When the
  registration view initializes, Then the captured `code` is placed into `RegisterFacade.referralCode`
  and the `referralCode` form control (mirroring `setReferralCode()` at `register.facade.ts:93-96`)
  without the visitor opening the manual "Add a referral code" dialog.
- [ ] **AC3 — Code validated, state shown** — Given a pre-applied code, When the form initializes, Then
  `validateReferralCodeNow()` (`register.facade.ts:55`) runs **once** (no auto-validation debounce loop)
  and drives the same `ReferralUiState` machine the dialog uses (`valid` shows the referrer first name;
  `invalid`/network failure shows the localized message).
- [ ] **AC4 — Bad/unknown codes never block signup** — Given the URL code is invalid, expired,
  self-referral, or unverifiable (network failure), When the page loads and the visitor proceeds, Then
  registration is still fully usable and submittable, consistent with the fail-soft contract in
  `register.facade.ts:106-110` (bad code silently skipped, never a hard error).
- [ ] **AC5 — Code survives to account creation** — Given a valid pre-applied code, When the visitor
  completes signup, Then the code is sent to `authService.register(...)` (`register.facade.ts:110`) and
  accepted by the backend `Register` command, so the referral is recorded against the new account.
- [ ] **AC6 — Already-logged-in invitee handled** — Given an already-authenticated customer opens
  `/r/{code}` (the `customerGuestGuard` at `guest.guard.ts:13` would otherwise bounce `register` →
  `orders`), When the route activates, Then the user is redirected to a sensible signed-in destination
  (e.g. rewards/orders), not the 404 page, and is not shown a broken/blank registration form.
- [ ] **AC7 — All new strings localized in 5 locales** — Given any new copy (e.g. an "applying your
  referral…" state or redirect notice), When it renders, Then it uses `TranslatePipe` keys present in
  **all five** locale files `apps/cleansia.app/src/assets/i18n/{en,cs,sk,uk,ru}.json` — no hardcoded
  strings.
- [ ] **AC8 — Test proves capture/pre-apply/validate/fail-soft** — A test (written test-first) covers
  the route-param → facade capture, the single-shot validate, and the fail-soft path; each AC maps to a
  case and the test appears before/with the implementation in the diff.

## Out of scope
- Any backend change. `Register.cs` and the `referralClient.validate` endpoint already accept and
  validate codes → **no new command/DTO/endpoint and no NSwag regeneration**. If one turns out to be
  needed, flag `manual_step: nswag-regen` rather than editing generated clients.
- Mobile / Android — deep-link + bottom-sheet handling already exist there (`ReferralCodeBottomSheet.kt`,
  `MainActivity.kt`). This is web-only parity.
- Partner and Admin apps — no `/r/` route added there.
- A dedicated marketing landing page (hero, referrer avatar, value-prop copy) — separate ticket.
- Changing referral reward/credit logic, points, or `ReferralService.AcceptAsync` rules.
- Persisting the code across a later return visit (cookies/localStorage, attribution windows, OG/social
  preview meta) — capture is in-session through signup only.
- EF migrations / DB seed / generated-client edits.

  facade, not the component) and C3 (canonical client-call pipe for the `validate` call).

  note "red: <test> failing → green".

  references.
- Feed the captured param into the existing `RegisterFacade` plumbing — reuse `referralCode` signal

  authenticated invitee is redirected to a signed-in destination instead of bounced to 404.
- New copy keys must land in all 5 locale files (en, cs, sk, uk, ru).
