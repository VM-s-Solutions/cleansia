## Context

**Finding F1** (`agents/backlog/audits/AUDIT-2026-06-01-execution-plan.md:51-53`): the
`generate-invoice` queue is **fully dead**. `GenerateInvoiceFunction` is a no-op stub â€” it

(`Cleansia.Functions/Functions/GenerateInvoiceFunction.cs:20-26`). Today invoice PDFs are only ever
produced **inline** inside the monthly close timer (`PayPeriodBackgroundService.cs:295-381`,

(`GenerateInvoiceMessage(EmployeeId, PayPeriodId, LanguageCode)` â€”

ties this to AUD-02/04: the payroll path is *"both unreachable and unverifiable"* until F1 is wired.

This ticket makes the function **do the work**: consume a `generate-invoice` message and run the
already-built `GenerateInvoice.Command` (`Cleansia.Core.AppServices/Features/EmployeePayroll/GenerateInvoice.cs:14`)
through MediatR â€” the same shape `CalculateOrderPayFunction` already uses

the durable outbox)** so the enqueueâ†’consume path is at-least-once and idempotent, and on **T-0171

## Acceptance criteria

> The hole is "the consumer is a no-op stub" â€” closing it is the function actually generating the

- [ ] **AC1 â€” The function runs the real invoice command instead of logging a stub.**
  Given a well-formed `generate-invoice` message with a valid `EmployeeId`/`PayPeriodId` that has
  unpaid `OrderEmployeePay` rows, When `GenerateInvoiceFunction.Run` consumes it, Then it sends
  `GenerateInvoice.Command(EmployeeId, PayPeriodId)` via `IMediator` and an `EmployeeInvoice` is
  persisted (the `Response(InvoiceId)` is logged) â€” the `// TODO: not yet implemented` /
  `Task.CompletedTask` stub at `GenerateInvoiceFunction.cs:20-26` is gone. Diff + test prove it.

- [ ] **AC2 â€” Queue-trigger (no JWT) context handled like the sibling consumers.**
  Given the function runs with no JWT/tenant context (same as `CalculateOrderPayFunction` and
