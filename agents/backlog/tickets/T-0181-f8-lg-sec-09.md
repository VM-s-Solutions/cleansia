## Context

Two findings name the same hole in the sitewide-promo fan-out, from opposite ends:

- **F8** (major / bug — `findings F8/LG-SEC-09`, `audits/AUDIT-2026-06-01-findings.json` id `F8`):
  `SendSitewidePromoFanoutFunction` re-throws on any failure **outside** the per-recipient inner
  try/catch (e.g. a `ToListAsync` page read throws on page 40 of 50,
  `SendSitewidePromoFanoutFunction.cs:108`), so the queue retries the campaign from `offset = 0`
  (`:105`, `:149`) and **re-enqueues to every user already processed on the prior attempt**. Up to
  5× duplicate promo pushes to early-paged users on a flaky run (`host.json` `maxDequeueCount: 5`).
  No persisted cursor, no per-campaign identity. The outer catch re-throws at
  `SendSitewidePromoFanoutFunction.cs:162`.
- **LG-SEC-09** (minor / security — same source id): the **producer** side. `SendSitewidePromo.Handler`
  (`SendSitewidePromo.cs:67-91`) enqueues exactly one fan-out message and returns success with **no
  campaign id and no dedupe**. A retried/double-submitted admin action enqueues a *second full fan-out*,
  double-pushing a marketing notification to the entire opted-in user base. Spam, not money — but a
  doublable mass side-effect.

**What the governing ADR already decided, and why this ticket exists now.** ADR-0002
(`agents/backlog/adr/0002-outbox-dispatch-contract.md`) **D2.3** classifies
`SendSitewidePromoFanoutFunction.cs:123` as **Bucket C — a fan-out producer, not an effect-realizer**:
it has no commit to gate and no MediatR pipeline, so it **stays on direct `IQueueClient.SendAsync`**,
is **excluded from TC-IDEMP-0 and from verification check #3**, and its at-least-once safety comes from
the **downstream** push consumer's `push:{UserId}:{EventKey}` dedup (D2.1 key table). D2.3 is explicit
that "a mid-stream crash re-enqueues recipients from offset 0; this **doubles cost, not effect**" and
that "a campaign-level resume marker is a **nice-to-have optimization, not a contract requirement**."

So the *effect* (a user receiving the same push twice) is already absorbed by the D2.2 downstream
guard once that lands. **This ticket delivers the named nice-to-have**: a per-campaign identity +
resume cursor so a flaky run does not re-enqueue (re-cost) the whole base, and a producer-side idempotent
enqueue so a double-submitted admin action does not start a second campaign. It does **not** weaken or
replace the downstream dedup — it is the cost/spam-reduction layer D2.3 deferred. Per dep, the
campaign-progress state rides on **T-0143**'s durable outbox/store substrate rather than introducing a
fresh ad-hoc table.

## Acceptance criteria

> The hole is closed and a test proves it. Tests are written **TEST-FIRST** per `knowledge/testing.md`
> (the cursor/skip logic and the producer dedup are pure logic → strict red→green; the consumer
> page-resume is contract test-first). The downstream `push:{UserId}:{EventKey}` dedup (D2.2) remains
> the load-bearing effect guard and MUST NOT be removed or relaxed by this ticket.

- [ ] **AC1 — Every campaign carries a stable `CampaignId` (producer side, LG-SEC-09).**
  Given `SendSitewidePromo.Handler` (`SendSitewidePromo.cs:69-89`) builds one
  `SendSitewidePromoMessage`, When the admin action runs, Then the message carries a deterministic
  per-campaign `CampaignId` (added to `SendSitewidePromoMessage`,
  `Cleansia.Core.Queue.Abstractions/Messages/SendSitewidePromoMessage.cs`). A unit test (TC-KEY-0 shape,
  ADR-0002 verification #6) proves two invocations with the **same domain inputs** that the producer
  intends as one campaign emit the **same** `CampaignId` — not a fresh `Guid` per send that would defeat
  the dedup.

- [ ] **AC2 — A double-submitted admin action does not start a second full fan-out (LG-SEC-09).**
  Given a `CampaignId` already enqueued/dispatched, When `SendSitewidePromo.Handler` runs again for the
  same campaign identity, Then the second invocation does **not** enqueue a second fan-out message (it
  short-circuits idempotently, returning `BusinessResult.Success`), so the entire opted-in base is not
  double-pushed. A handler unit test (mock the dispatch/store) proves the second call enqueues nothing.

- [ ] **AC3 — Per-campaign resume cursor: a retry resumes, it does not restart (F8).**
  Given a campaign that processed N pages and then failed on a later page read
  (`SendSitewidePromoFanoutFunction.cs:108`) so the message is redelivered, When the Function re-runs
  for the same `CampaignId`, Then it resumes **from the last persisted cursor** (e.g. last processed
  `UserId`, consistent with the existing `.OrderBy(x => x.UserId)` stable order at `:102`) instead of
  restarting at `offset = 0` (`:105`). A test drives a simulated mid-campaign failure + redelivery and
  asserts the already-processed recipients are **not** re-enqueued on the retry.

- [ ] **AC4 — The cursor advances durably and survives the crash window.**
  Given the consumer enqueues a page of per-user `notifications-dispatch` messages, When the page
  completes, Then the campaign progress is persisted (riding on T-0143's outbox/store substrate) such
  that a crash/redelivery after that point resumes past it. A test proves a row left at cursor-position
  P is picked up at P (not 0) on the next run.

- [ ] **AC5 — Terminal completion is idempotent and re-runs are no-ops.**
  Given a campaign whose cursor has reached the end of the opted-in set, When the same message is
  redelivered, Then the Function recognizes the campaign as **already complete** for that `CampaignId`
  and returns without re-paging/re-enqueuing. A test asserts a completed-campaign redelivery enqueues
  zero downstream messages.

- [ ] **AC6 — Downstream dedup is preserved (no regression of the D2.2 effect guard).**
  Given ADR-0002 D2.3 makes the downstream `push:{UserId}:{EventKey}` dedup the load-bearing *effect*
  guard, When this ticket lands, Then that guard is unchanged and still present — this ticket adds the
  cost/spam-reduction cursor layer **on top of** it, not in place of it. Verified by diff (the
  `notifications-dispatch` consumer guard is untouched) and by the existing TC-IDEMP-0 staying green.

- [ ] **AC7 — Per-recipient continue and tenant scoping behavior preserved.**
  Given the existing per-recipient `try/catch`-and-continue (`SendSitewidePromoFanoutFunction.cs:137-146`)
  and the campaign tenant override (`:71-74`), When the cursor logic is added, Then a single bad
  per-user enqueue still does not poison the campaign, and the tenant override behavior is not changed by
  this ticket (any `ClearTenantOverride` hardening is F9, out of scope here). Test or diff confirms the
  continue-on-single-failure path still holds.

## Out of scope

- **Replacing or relaxing the downstream `push:{UserId}:{EventKey}` dedup (ADR-0002 D2.2/D2.3)** — that
  is the effect guard and stays as-is; this ticket is the cost/spam layer.
- **The durable outbox / store substrate itself** — delivered by **T-0143 (F2-FULL)**, this ticket's
  dependency. T-0181 *uses* that substrate for campaign progress; it does not build the outbox.
- **F9 (tenant-override leak: `SetTenantOverride` without `ClearTenantOverride`,
  `SendSitewidePromoFanoutFunction.cs:71-74`)** and **F10 (raw `messageText` logged at Error,
  `:159-161`)** — separate findings, separate tickets. Do not fold them in.
- **LG-SEC-07 (per-campaign rate limit)** — the findings cross-reference it ("combine with the
  LG-SEC-07 rate limit"); it is a distinct concern and not this ticket's edit.
- **NSwag regen** — `SendSitewidePromoMessage` and the queue contract are **internal** (ADR-0002
  Consequences: "No NSwag change"). Adding `CampaignId` to the queue message does not change the admin
  HTTP contract; if the admin `Command`/controller surface gains a client-supplied id, re-flag
  `manual_step: nswag-regen` at design time. As scoped (server-derived `CampaignId`), `manual_steps: []`.

## Implementation notes

- **Governing ADR:** **ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`). The
  load-bearing clauses: **D2.3** (fan-out producer stays direct; resume marker is the named nice-to-have
  this ticket delivers; effect safety is the downstream dedup — do not regress it), **D2.1** (the
  `push:{UserId}:{EventKey}` key table the downstream guard uses), and **D5 Bucket B/C** (the producer
  `SendSitewidePromo.cs:88` is Bucket B and migrates to the outbox in T-0143; the Function
  `SendSitewidePromoFanoutFunction.cs:123` is Bucket C and stays direct). ADR-0001/0003 do **not** govern
  this work.
- **Serialization clusters (IMPORTANT — read TICKET-MAP §Shared-file map).** T-0181 is **not** itself a
  listed cluster member, but it shares two surfaces with cluster work and must serialize against them:
  - `SendSitewidePromo.cs` (the producer handler) is migrated to the outbox by **T-0143** under its
    Bucket-B migration (T-0143 AC7 lists `SendSitewidePromo.cs:88`). T-0181 edits the **same handler** to
    add the `CampaignId`/idempotent-enqueue — so T-0181 must run **after T-0143 is `done`** (already the
    `depends_on`) and must not edit `SendSitewidePromo.cs` concurrently with any in-flight T-0143 slice.
  - `SendSitewidePromoFanoutFunction.cs` is the cursor surface; serialize any concurrent ticket that
    touches it (e.g. F9/F10 if they run) — never two instances editing this Function at once.
  T-0143 sits in the `UnitOfWorkPipelineBehavior.cs` cluster (TICKET-MAP row 3); T-0181 does **not**
  touch that pipeline file — it only consumes the substrate that ticket produced.
- **Why the dependency on T-0143.** The campaign-progress state (cursor + campaign identity dedup) should
  ride on the **durable outbox/store substrate** T-0143 lands, not a fresh ad-hoc table (which would add
  an unflagged `ef-migration`). Confirm the final substrate shape from T-0143's merged diff before
  designing the progress row; if a dedicated campaign-progress table is unavoidable, escalate to add
  `manual_step: ef-migration` and an architect note **before** going `ready`.
- **Real code anchors:** producer — `SendSitewidePromo.cs:67-91` (handler), `SendSitewidePromoMessage.cs`
  (add `CampaignId`); consumer — `SendSitewidePromoFanoutFunction.cs:104-151` (the page loop + `offset`
  restart at `:105/:149`), `:108` (the page-read throw F8 names), `:115-146` (per-recipient enqueue +
  continue), `:157-163` (outer catch + re-throw), `:71-74` (tenant override), `:102` (`OrderBy(UserId)`
  stable order the cursor relies on); `QueueNames.SitewidePromoFanout` (`QueueNames.cs:8`); downstream
  key `push:{UserId}:{EventKey}` (ADR-0002 D2.1).
- **TEST-FIRST** per `agents/knowledge/testing.md`: the `CampaignId` determinism (AC1, TC-KEY-0 shape),
  the producer double-submit short-circuit (AC2), and the resume-from-cursor skip logic (AC3/AC4/AC5)
  are **logic** → strict red→green→refactor, tests visible **before** implementation in commit order /
  status log, each AC mapped to a case. The Function consumer body must be reachable from the test
  project via `Cleansia.Functions.Core` (ADR-0002 D5 step 1 / FUNC-CORE) — confirm it is extracted before
  writing the consumer-resume test. This is `security_touching` (mass side-effect / S7 idempotency) so the
  Security gate is mandatory; route per `routing.md` (architect confirm substrate fit → backend producer
  → functions consumer), spawn a **reviewer** in parallel with every developer instance, then
  security → qa.
