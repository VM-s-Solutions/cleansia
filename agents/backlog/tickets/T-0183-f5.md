## Context
Finding **F5** (`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:920-929`, High):
four `Cleansia.Functions` timer triggers ship with the dev placeholder cron `every 2 minutes`
(720 runs/day each) instead of the cadence each function's own XML doc-comment documents. The
mismatch (1) breaks the documented **Materialize → Reminder ordering guarantee** — both fire on the
same 2-min tick, so `SendRecurringOrderReminders` can sweep before that tick's materialization
completes and miss same-day orders until the next tick; (2) causes constant cross-tenant table scans
(membership / orders / prefs) — wasted DB load and Function execution cost; and (3) makes the
idempotency stamps load-bearing for correctness rather than a safety net (any stamp gap → push spam
every 2 min). Stamps currently hold, so this is a cost + ordering-correctness fix, not user-visible
spam today — but it is fragile.

The four offending triggers, with the doc cadence each disagrees with:
- `MaterializeRecurringBookingsFunction.cs:19` — `"0 */2 * * * *"`; doc (`:9`) says "Daily at 02:00 UTC".
- `SendRecurringOrderRemindersFunction.cs:23` — `"0 */2 * * * *"`; doc (`:9-13`) says "Daily at 02:30 UTC … runs 30min after Materialize".
- `SendMembershipLifecycleNotificationsFunction.cs:24` — `"0 */2 * * * *"`; doc (`:9-17`) says daily, tighten to "03:00 UTC".
- `SendNewJobsDigestTimerFunction.cs:24` — `"0 0/2 * * * *"`; doc (`:10`) says "every 30 minutes (`0 0,30 * * * *`)".

## Acceptance criteria
- [ ] **AC1** — Given `MaterializeRecurringBookingsFunction`, When the `TimerTrigger` cron is read, Then it resolves to the documented **02:00 UTC daily** schedule (`0 0 2 * * *`) and the XML doc-comment matches.
- [ ] **AC2** — Given `SendRecurringOrderRemindersFunction`, When the cron is read, Then it resolves to **02:30 UTC daily** (`0 30 2 * * *`), preserving the ">= 30min after Materialize" ordering the doc-comment promises (`:11-13`).
- [ ] **AC3** — Given `SendMembershipLifecycleNotificationsFunction`, When the cron is read, Then it resolves to the documented **daily 03:00 UTC** slot (`0 0 3 * * *`).
- [ ] **AC4** — Given `SendNewJobsDigestTimerFunction`, When the cron is read, Then it resolves to the documented **every 30 minutes** schedule (`0 0,30 * * * *`).
- [ ] **AC5** — Given dev vs prod must differ without a code change, When the schedule is configured, Then each trigger reads its cron from an app-setting via `%AppSetting%` cron syntax (e.g. `[TimerTrigger("%RecurringMaterializeCron%")]`) with the production default supplied in config (and a dev override path documented), so promotion is config-only.
- [ ] **AC6** — Given the changes, When the test suite runs, Then a test asserts each of the 4 functions' effective schedule equals its documented production cadence and that Materialize fires strictly before Reminder, proving the hole is closed (the test exists and was written first — see Implementation notes).
- [ ] **AC7** — Given the schedules now run on documented intervals, When idempotency is re-checked, Then the per-entity stamps (`RecurringReminderSentAt`, `RenewalReminderSentAt`/`CancellationReminderSentAt`, digest watermark) remain the duplicate-suppression mechanism — no idempotency behavior is removed or weakened by this ticket.
