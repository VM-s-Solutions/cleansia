## Context

Finding **F6** (`agents/backlog/audits/AUDIT-2026-06-01-findings.json`, `where:
FiscalRetryService.cs:38-93`, severity major): `FiscalRetryService.ProcessDueRetriesAsync` processes a
batch of up to 50 receipts due for fiscal retry, then commits **the whole batch in one
`CommitAsync`** at `FiscalRetryService.cs:91`. Per-receipt exceptions are already swallowed correctly
(`:79-86`) so one bad row does not abort the loop — but **every** mutation accumulates in a single
`DbContext` and persists only at that one final commit.

Two live failure shapes (per F6 `impact`):
1. **All-or-nothing loss.** If `CommitAsync` throws (a DB blip), the retry-tracking state for the
   entire batch is lost — `FiscalNextRetryAt` / `FiscalRetryCount` / fiscal-code (`SetFiscalData`)
   and `MarkEmailSent` set via `MarkFiscalRetryAttempted` (`ReceiptService.cs:243,266,276`) and
   `MarkEmailSent` (`FiscalRetryService.cs:72`). Because this is a **timer**, not a queue, there is
   no automatic retry of the lost work.
2. **Held-email double-send (partial S7 exposure).** For `BlockingOnline` /
   `BlockingWithOfflineCache` countries the confirmation email is released at
   `FiscalRetryService.cs:70-72` after a successful fiscal retry, but `receipt.MarkEmailSent(...)` is
   **in-memory only** until the batch commit. If that commit fails, the email already went out but
   `EmailSent` is never persisted → the next 5-minute tick (`RetryFailedFiscalRegistrationsFunction.cs:14`)
   re-reads `GetDueForRetryAsync` and **re-sends the held email** (S7 — idempotency on side effects).

This is a **Wave-2** durability fix governed by **ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`):
D2.2 names the receipt **email** as a terminal effect that must be idempotent (claim-first / persist
the sent-marker in a commit that precedes the send), and D3.4 establishes the fiscal-queue
reconciliation backstop. It depends on **T-0143 (F2-FULL)** so the durable-dispatch contract is in
force before we change the fiscal retry commit cadence.

## Acceptance criteria
- [ ] **AC1 — per-receipt durability.** Given a batch of N receipts due for retry, When one receipt's
      persistence fails (simulated `CommitAsync` fault on that receipt), Then the retry-tracking state
      of the **already-processed** receipts in the same batch is durably persisted (not rolled back),
      and the loop continues / the unprocessed remainder stays due — i.e. the single batch-wide
      `CommitAsync` at `FiscalRetryService.cs:91` is replaced by per-receipt (or bounded-chunk)
      commits inside the loop. Proven by a test.
- [ ] **AC2 — held-email is not re-sent after a commit failure (S7).** Given a `BlockingOnline`
      receipt whose fiscal retry succeeds and whose held email is released, When the persistence of
      that receipt's `EmailSent` marker is interrupted, Then on the **next** timer tick the email is
      **not** sent again — the email-sent state is committed before (or atomically with) the marker so
      a failed commit does not leave `EmailSent` false while the email already went out (claim-first
      per ADR-0002 D2.2). Proven by a test asserting `IEmailService.SendOrderReceiptEmailAsync` fires
      **at most once** per receipt across two consecutive ticks.
- [ ] **AC3 — bounded run preserved.** Given more than `BatchSize` (50) receipts are due, When the
      timer runs, Then at most `BatchSize` receipts are processed per tick (existing
      `FiscalRetryService.cs:22,29` cap unchanged) and the multi-tenant override is still set/cleared
      per receipt (`:44-48`).
- [ ] **AC4 — test proves the hole is closed.** A new xUnit test in `Cleansia.Tests` (or the
      established fiscal test target) exercises AC1–AC2 against `FiscalRetryService` with a faulting
      `IUnitOfWork` and asserts no batch-wide loss and no double-email. Test written **first** (red →
      green) per `agents/knowledge/testing.md`.

## Implementation notes
- Touch point: `src/Cleansia.Core.AppServices/Services/FiscalRetryService.cs` — replace the single
  trailing `CommitAsync` (`:91`) with a per-receipt (or small-chunk) commit inside the `foreach`
  (`:38-89`), keeping the existing per-receipt `try/catch` so one failed commit re-queues only the
  unprocessed remainder (F6 `fix`). Ensure the held-email release (`:63-77`) persists `EmailSent`
  **before** (or atomically with) the email send so a later commit failure cannot cause a re-send
  (ADR-0002 D2.2 claim-first).
- Per-receipt mutations to make durable: `MarkFiscalRetryAttempted` / `SetFiscalData`
  (`ReceiptService.cs:243,266,276`) and `MarkEmailSent` (`FiscalRetryService.cs:72`).
- Keep the tenant-override discipline (`ClearTenantOverride` / `SetTenantOverride`,
  `FiscalRetryService.cs:44-48`) inside the per-receipt boundary so each commit writes under the
  correct tenant.
- **Governing ADR: ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`) — D2.2 (terminal
  email effect idempotency, claim-first) and D3.4 (fiscal-queue durability framing). Depends on
  **T-0143 (F2-FULL)**.
- **Serialization cluster: none.** `FiscalRetryService.cs` does not appear in any shared-file cluster
  in `agents/backlog/TICKET-MAP.md` — no concurrent-edit collision to serialize against.
- **Build TEST-FIRST** per `agents/knowledge/testing.md`: write the failing batch-loss / double-email
  tests before the fix; spawn the reviewer instance concurrently with the backend developer.
