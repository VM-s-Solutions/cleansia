## Context

Two GDPR compliance gaps block an EU PROD launch where the data, entities, and APIs already exist
but no UI consumes them. From the audit (`audits/AUDIT-2026-06-01-findings.md:73-74`):

- **IA-01** â€” *"Admin GDPR back-office has zero UI; the request/consent audit surface is unreachable."*
  The admin API and audit data already exist: `src/Cleansia.Web.Admin/Controllers/AdminGdprController.cs:15-49`
  exposes `GET export/{userId}`, `POST delete-account/{userId}`, `GET consents/{userId}`, `GET requests`
  (paged), each gated by `Policy.CanAdminExportUserData` / `CanAdminDeleteUserAccount` /
  `CanAdminViewUserConsents` / `CanViewGdprRequests` (all â†’ `PhysicalPolicy.AdminOnly`, mirrored at
  `libs/core/services/src/lib/auth/policy.ts:299-302`). Handlers/DTOs exist (`Features/Gdpr/GetAllGdprRequests.cs`,
  `AdminGetUserConsents.cs`, `AdminExportUserData.cs`, `AdminDeleteUserAccount.cs`; `GdprRequestDto`,
  `UserConsentDto`, `GdprExportDto`). `AdminExportUserData.Handler` already writes the Article-30 audit
  row (`AdminExportUserData.cs:39-52`). The client is already generated: `AdminGdprClient` at
  `libs/core/admin-services/src/lib/client/admin-client.ts:4580` (methods :4594, :4648, :4698, :4761).
  Nothing in `libs/cleansia-admin-features/` consumes any of it. (Story **US-admin-0042**.)

- **IA-03** â€” the partner GDPR page is inert: `libs/cleansia-partner-features/gdpr/src/lib/gdpr/gdpr.component.ts:22`
  is `export class PartnerGdprComponent {}` with a static legal-copy template, while the backend is fully
  reachable for partners: `src/Cleansia.Web.Partner/Controllers/GdprController.cs:16-61` exposes
  `GET export`, `POST delete-account`, `GET consents`, `POST consents`, `POST consents/withdraw`, gated by
  `CanExportOwnData`/`CanDeleteOwnAccount`/`CanViewOwnConsents`/`CanGrantConsent`/`CanWithdrawConsent`

Governing decision: **ADR-0001 (ADR-AUTHZ)** â€” the frozen policy map. The admin surface must respect

## Acceptance criteria

### Admin "Data Protection" back-office (IA-01 / US-admin-0042)
- [ ] **AC1 â€” GDPR request audit list (paged).** Given an admin with `CanViewGdprRequests` opens the
  Data Protection page, When the list loads, Then a `cleansia-table` shows rows from `GET /AdminGdpr/requests`
  with columns User Id, Request Type, Status (translated `GdprRequestStatus`), Processed By, Completed At,
  Notes, Created On â€” server-side paged (offset/limit), newest first â€” rendering distinct empty / loading /
  loaded states.
- [ ] **AC2 â€” Per-user consent viewer.** Given an admin with `CanAdminViewUserConsents` selects a user,
  When they open that user's consents, Then `GET /AdminGdpr/consents/{userId}` results render each
  `ConsentType` (translated) with Granted/Withdrawn state, Granted At, Withdrawn At, Created On.
- [ ] **AC3 â€” Admin export (DSAR).** Given an admin with `CanAdminExportUserData`, When they trigger
  "Export data" and `GET /AdminGdpr/export/{userId}` succeeds, Then the `GdprExportDto` downloads as a
  file with a success confirmation, And a new `GdprRequest` (type Export) subsequently appears in the
  requests list (proving the existing audit-trail write).
- [ ] **AC4 â€” Admin erasure (gated + confirmed).** Given an admin with `CanAdminDeleteUserAccount`, When
  they choose "Erase account", Then a `ConfirmationService` dialog requires explicit confirmation before
  `POST /AdminGdpr/delete-account/{userId}` is called, success shows a confirmation, and the control is
  hidden when the policy is absent.
- [ ] **AC5 â€” Per-control permission gating.** Given an admin missing one of the four GDPR policies, When
  the page renders, Then each control (list / consents / export / erase) is independently hidden via
  `*cleansiaPermission="Policy.CanXxx"`, and any API 403 surfaces through `SnackbarService.showApiError`
  without crashing the view.

### Partner GDPR self-service (IA-03 / US-partner-0007)
- [ ] **AC6 â€” Export.** Given an authenticated partner on the GDPR page, When they activate "Export my
  data", Then the partner GDPR `export` endpoint is called and a `my-data-export.json` of their
  `GdprExportDto` downloads with a success snackbar; on failure an error snackbar shows and no file
  downloads.
- [ ] **AC7 â€” View consents.** Given an authenticated partner, When the page loads, Then consents are
  fetched from the partner `consents` endpoint and each `ConsentType` renders as a toggle reflecting
  granted/withdrawn state, with a loading state while fetching.
- [ ] **AC8 â€” Change a consent.** Given consents are displayed, When the partner toggles one, Then the
  app calls grant (`consents`) or withdraw (`consents/withdraw`) accordingly, shows a success snackbar,
  and the toggle reflects persisted server state after re-fetch (IP/user-agent captured server-side, not
  sent from the client).
- [ ] **AC9 â€” Delete account (confirmed).** Given an authenticated partner, When they activate "Delete my
  account", Then a confirm dialog is required first; on confirm `delete-account` is called, on success a
  success snackbar shows and the partner is logged out/redirected; on a blocked deletion (active order /
  pending invoice) the partner-localized error shows and they remain logged in.
- [ ] **AC10 â€” Unauthenticated gating.** Given the partner GDPR page is opened while not authenticated,
  When it renders, Then no consents call is made and no data-subject actions are offered (static legal
  sections may still render), matching the customer page's authenticated-gating behavior.

### Cross-cutting
- [ ] **AC11 â€” i18n in all 5 locales.** Every new user-visible string (admin feature labels +
  `GdprRequestStatus`/`ConsentType` translations in `apps/cleansia-admin.app/src/assets/i18n/*`; partner
  action/consent/outcome keys under `pages.gdpr.*` in `apps/cleansia-partner.app/src/assets/i18n/*`)

- [ ] **AC12 â€” Frontend conventions (C1â€“C8) + tests prove behavior.** Each facade extends

  this single-feature state. Specs (written test-first) cover the success/failure/gating paths above.
