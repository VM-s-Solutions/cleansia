## Context
Audit finding **IA-02** (major, gap): the customer-web profile "Email Notifications" toggle is a
dead-end. `profile.component.html:349-356` renders `<p-toggleSwitch>` bound to a single
local-only signal `emailNotifications = signal(true);` (`profile.component.ts:115`) that is **never
loaded from or saved to** the backend — it always reads "on" and changes nothing server-side. The
real 11-category preferences API already exists and is honored everywhere else: handlers
`Features/Notifications/GetMyNotificationPreferences.cs` / `UpdateNotificationPreferences.cs`,
exposed by `Cleansia.Web.Customer/Controllers/NotificationPreferencesController.cs` at
`GET /api/NotificationPreferences/GetMine` and `PUT /api/NotificationPreferences/Update`, but it is
consumed **only by the Android customer app** (`customer-app/.../core/notifications/NotificationPreferencesRepository.kt`).
This violates `knowledge/conventions.md` ("empty/loading/error states are part of the work") and the
three-data-states bar — the control has no real state at all.

Sources: `agents/backlog/audits/AUDIT-2026-06-01-findings.json` (id IA-02) and
`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md:606-612`.

## Acceptance criteria
- [ ] **AC1** — Given an authenticated customer opens the profile preferences section, When the
  section loads, Then it calls `GET /api/NotificationPreferences/GetMine` via the generated
  `NotificationPreferencesClient.getMine()` and renders one control per category for all 11 booleans
  in `NotificationPreferencesDto` (`OrderUpdates`, `CleanerOnTheWay`, `OrderCompleted`,
  `OrderCancelled`, `RefundIssued`, `MembershipExpiring`, `MembershipCancelled`, `TierUpgrade`,
  `Promo`, `DisputeReply`, `RecurringScheduled`) reflecting the server values.
- [ ] **AC2** — Given the preferences are loaded, When the customer flips any category and saves,
  Then `PUT /api/NotificationPreferences/Update` is called with an `UpdateNotificationPreferencesCommand`
  carrying all 11 fields, and on success the UI reflects the persisted state returned by the endpoint
  (re-reading via `getMine()` returns the changed values).
- [ ] **AC3** — Given the preferences API is in-flight or fails, When the section is shown, Then all
  three data states are observable: a loading state (skeleton/spinner) while fetching, the loaded
  control state, and an error state with a snackbar (mapped via `TranslateService`) on failure — no
  silent dead-end. Save shows a `saving` state and success/error snackbar.
- [ ] **AC4** — The fake local-only `emailNotifications = signal(true);` (`profile.component.ts:115`)
  and its dead-end `<p-toggleSwitch>` block (`profile.component.html:349-358`) are removed/replaced;
  no UI control remains that mutates only client state.
- [ ] **AC5** — Every new user-visible string has a `TranslatePipe` key present in all 5 customer
  locale files (`apps/cleansia.app/src/assets/i18n/{en,cs,sk,uk,ru}.json`) — including a label per
  category. No hardcoded strings; no `any`.
- [ ] **AC6** — A Jest spec for the new facade exists and passes, covering: load populates the 11
  signals from `getMine()`, save sends all 11 fields via `update()`, and the loading→loaded and
  loading→error state transitions with error→snackbar mapping (per `knowledge/testing.md` frontend
  rule: test the facade over the component).

## Out of scope
- Backend changes — `NotificationPreferencesController`, the handlers, and `NotificationPreferencesDto`
  already exist and are correct; this ticket only builds the web consumer.
- NSwag regeneration — the generated `customer-services` client already exposes
  `NotificationPreferencesClient.getMine()/update()`, `NotificationPreferencesDto`, and
  `UpdateNotificationPreferencesCommand` (`libs/core/customer-services/src/lib/client/customer-client.ts:2611-2740, 9824`).
  No `manual_step: nswag-regen`.
- Android/iOS/admin/partner notification UI (Android already consumes this API).
- Adding new notification categories or changing defaults (the lazy-create defaults live server-side).

## Implementation notes
- **Built TEST-FIRST** per `agents/knowledge/testing.md`: write the facade Jest spec encoding the
  load/save/three-state behavior before/with the implementation (Gate 6 rejects after-the-fact tests
  for pure logic). Logic lives in the **facade**, not the component (`patterns-frontend.md`).
- **Reuse, don't reinvent**: mirror the existing `GdprFacade`
  (`libs/cleansia-customer-features/gdpr/src/lib/gdpr/gdpr.facade.ts`) — `@Injectable`, extends
  `UnsubscribeControlDirective`, injects `CustomerClient` + `SnackbarService` + `TranslateService`,
  `takeUntil(this.destroyed$)`, signal-based loading/saving/data state, snackbar on success/error.
- Use `<cleansia-*>`/PrimeNG controls (no raw form controls); `ChangeDetectionStrategy.OnPush`.
  Surface lives in the profile preferences section (`profile.component.html` near line 349) or a new
  child section/component delegating to the facade.
- **Contract**: `getMine()` lazy-creates defaults on first call, so a save always has a row to patch.
  `update()` takes the full `UpdateNotificationPreferencesCommand` (all 11 booleans) and returns the
  persisted `NotificationPreferencesDto`.
- **Serialization cluster**: none. The TICKET-MAP shared-file clusters are all backend/config
  surfaces (`PolicyBuilder.cs`, `CleansiaStartupBase.cs`, `UnitOfWorkPipelineBehavior.cs`,
  `ServiceExtensions.cs`, dispute/`CreateOrder`/`LoyaltyService`). This ticket touches only customer
  frontend files. It does edit `profile.component.ts`/`.html`, so it must **not** run concurrently
  with another customer-profile ticket if one is added later — serialize on that file pair if so.
- **ADR**: none in force — the 3 accepted ADRs (0001 authorization, 0002 outbox dispatch, 0003
  partitioned rate-limiting) govern backend/infra, not a pure frontend consumer.
