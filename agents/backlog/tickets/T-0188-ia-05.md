## Context

The device read side is half-built and orphaned. `DeviceDto`
(`src/Cleansia.Core.AppServices/Features/Devices/DTOs/DeviceDto.cs:3-9`) is shaped exactly like a
"list my devices" row yet is referenced **only by its own declaration** ŌĆö dead code, which
`agents/knowledge/conventions.md` ("No dead code") treats as a hard fail. The data access already
exists and is implemented (`IDeviceRepository.GetByUserIdAsync`,
`src/Cleansia.Core.Domain/Repositories/IDeviceRepository.cs:9`, used by `GdprDeletionService` and
`SendPushNotificationFunction`) but is never surfaced to the user who owns the devices: there is **no
`GetMyDevices` query**, no list endpoint on any `DeviceController`, and no client UI.

`Features/Devices/` contains only `RegisterDevice.cs` and `UnregisterDevice.cs`; `UnregisterDevice`
requires the caller to already know the handset's `DeviceId` string (fine for the app self-unregistering
on sign-out, useless from a *different* phone for a device you no longer hold). On the session side
`Logout.cs` revokes one refresh token and `RefreshToken.cs` rotates the current one ŌĆö neither lets a
user enumerate or revoke *other* sessions. The only device lifecycle a user benefits from today is the
invisible `DataRetentionBackgroundService.CleanStaleDevicesAsync` bulk sweep. A user who loses a phone
cannot see or cut off that device's push + session.

This ticket closes that gap: a `GetMyDevices` query returning `DeviceDto`, an ownership-checked
revoke-by-id command that also kills the device's refresh tokens, the mobile self-service surface, and
the optional admin read panel.

Sources: audit finding **IA-05**
(`agents/backlog/audits/AUDIT-2026-06-01-findings.json:1053-1068`,

## Acceptance criteria

- [ ] **AC1 ŌĆö List my devices (query + endpoint).** Given an authenticated user with one or more
  registered devices, When the app requests the device list via the new `GetMyDevices` query exposed on
  the `DeviceController` (S2 `[Permission]` policy), Then it receives a list of **`DeviceDto`** rows (the
  existing record, now wired up ŌĆö not a new parallel type) showing platform, device label/id and
  `LastActiveAt`, scoped to the caller's own `UserId` **derived from the JWT via `IUserSessionProvider`,
  never from request input** (S1), the current device is identifiable in the response, and a test proves
  user A's call never returns user B's devices.

- [ ] **AC2 ŌĆö Revoke a device I no longer hold (ownership-checked, NotFound on miss).** Given a device in
  my list, When I revoke it by its server-side id through the new revoke-by-id command, Then that
  `Device` row is removed (receives no further push) **only if it belongs to me**; a device id I do not
  own returns **NotFound** (not Forbidden) and does not reveal the row exists (S3), proven by a test
  asserting cross-user revoke yields NotFound and leaves the row intact.

- [ ] **AC3 ŌĆö Revoke also ends that device's session, not just its push token.** Given I revoke a device
  with an active sign-in, When the revoke completes, Then the refresh token(s) associated with that
  device are revoked through the existing `IRefreshTokenService.RevokeAsync` path that `Logout` already
  uses, so the lost handset can no longer mint new access tokens on next refresh ŌĆö covered by a test.

- [ ] **AC4 ŌĆö Mobile self-service surface.** Given the partner or customer mobile app profile/settings
  area, When I open "Your devices", Then I see my devices with platform and last-active, can revoke any
  device **other than the one I am on**, and the list reflects the change after revoke; the three explicit
  data states (loading, empty, error) are all handled per `conventions.md` production-ready bar. Strings
  live in Android string resources (no hardcoded text).

- [ ] **AC5 ŌĆö No leaked identifiers; dead code resolved.** Given the new query returns `DeviceDto`, When
  the response is serialized, Then it carries **no `TenantId`** and no foreign user identifiers (S4);
  `DeviceDto.UserId` is removed from the returned shape or justified as the caller's own id; **and
  `DeviceDto` is no longer dead code** (it is now produced by `GetMyDevices`), satisfying the audit's
  "wire it up or delete it" disposition.

- [ ] **AC6 (optional / admin) ŌĆö Admin can view a user's registered devices.** Given an admin on a user's
  detail page, When they open that user's devices panel, Then they see the user's registered devices
  (read-only acceptable for v1) sourced from the same `GetByUserIdAsync` data via the generated admin
  client, the admin endpoint is gated by an admin `[Permission]` policy (S2), and the DTO carries no
  cross-user PII beyond what admins are already authorized to see (S4).

## Out of scope

- Renaming/relabelling devices or setting a friendly device name.
- Geolocation, IP, browser/OS fingerprint, or a per-device "login activity / audit history" timeline
  (the `Device` entity stores no IP/location; `RefreshToken` `DeviceLabel`/`IpAddress` metadata is a
  separate, larger story).
- A "revoke ALL other sessions / sign out everywhere" bulk action (distinct story; v1 is per-device).
- Changing `DataRetentionBackgroundService.CleanStaleDevicesAsync` cutoff/notify behaviour.
- Web (Angular customer/partner SPA) self-service device management beyond the optional admin read panel
  in AC6 ŌĆö web push session management is not in this ticket.
- Any change to `RegisterDevice` / `UnregisterDevice` semantics or the FCM dispatch path
  (`SendPushNotificationFunction`); 2FA / trusted-device / "remember this device" flows.
