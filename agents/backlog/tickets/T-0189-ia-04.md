## Context
`LastLoginAt` is a phantom DTO field surfaced to admins but **structurally always null** — no
login tracking exists anywhere. Per finding **IA-04** (`agents/backlog/audits/AUDIT-2026-06-01-findings.json`):

- The admin contract promises a "last login" timestamp:
  - `Features/AdminUsers/DTOs/AdminUserDetailDto.cs:18` — `DateTimeOffset? LastLoginAt`
  - `Features/AdminUsers/DTOs/AdminUserListItem.cs:16` — `DateTimeOffset? LastLoginAt`
- But it is hardcoded to null in the mappers:
  - `Mappers/AdminUserMappers.cs:22` — `LastLoginAt: null`
  - `Mappers/AdminUserMappers.cs:41` — `LastLoginAt: null`
- `Core.Domain/Users/User.cs` has **no** last-login field, so there is nothing to populate from.

This is a half-built feature: someone modeled the read side and never populated it. It violates the
"no dead code" convention and the production-ready bar. Closing IA-04 means: add the field to the
domain entity, write it on every successful token issuance, and populate it through the mappers so
the existing admin surface stops lying. (If we instead chose to defer, the finding's alternative is
to delete the DTO fields — this ticket takes the *implement* path because the field is already in the
admin contract and useful for spotting dormant/compromised admin accounts.)

## Acceptance criteria
- [ ] **AC1 — field exists** — Given the `User` aggregate (`Core.Domain/Users/User.cs`), When the
  domain is built, Then `User` exposes a `DateTimeOffset? LastLoginAt` with a private setter and a
  behavior method (e.g. `RecordLogin(DateTimeOffset)`) that sets it; a unit test asserts the method
  sets the value.
- [ ] **AC2 — persisted** — Given `UserEntityConfiguration` (`Infra.Database/EntityConfigurations/UserEntityConfiguration.cs`),
  When the model is configured, Then `LastLoginAt` is mapped as a nullable column. **manual_step:
  ef-migration** — the migration is owner-run; this ticket only adds the entity/config change and
  flags it.
- [ ] **AC3 — written on every login path** — Given a successful authentication, When a token is
  issued, Then `LastLoginAt` is updated to "now" for that user. This must hold for **all** four
  entry paths: `Login.Handler`, `AdminLogin.Handler`, `PartnerLogin.Handler`, `GoogleAuth.Handler`
  (`Features/Auth/*.cs`). Implement once at the shared choke point — `TokenService.GenerateTokenAsync`
  (`Services/TokenService.cs:22`) issues the access token for all four — rather than four times.
  A handler/service unit test (mocked `IUserRepository`/clock) proves `RecordLogin` is invoked on
  success and **not** invoked when `IsEmailConfirmed` is false (the early-return at
  `TokenService.cs:24-29`) or when the login otherwise fails.
- [ ] **AC4 — not bumped on refresh** — Given a token refresh, When `RefreshToken.Handler`
  (`Features/Auth/RefreshToken.cs`) rotates the token, Then `LastLoginAt` is **not** changed
  (refresh is not a fresh interactive login). A test asserts the value is untouched on refresh.
- [ ] **AC5 — surfaced (backend mappers)** — Given a user who has logged in, When admin user
  detail/list is mapped, Then `MapToAdminDetailDto` and `MapToAdminListItem`
  (`Mappers/AdminUserMappers.cs:41` and `:22`) return the persisted `user.LastLoginAt` instead of
  the hardcoded `null`. A mapper unit test asserts the value flows through.
- [ ] **AC6 — surfaced (admin web)** — Given the admin user-management table
  (`libs/cleansia-admin-features/admin-user-management/.../admin-user-management.models.ts`), When an
  admin views the list, Then a "Last login" column renders the regenerated `lastLoginAt`
  (formatted date, empty when null), with translation keys added under
  `pages.admin_user_management.columns.*` in all 5 locales (en/cs/sk/uk/ru). **manual_step:
  nswag-regen is NOT required** — the field already exists in the generated client
  (`libs/core/admin-services/.../admin-client.ts`); only its value changes server-side. The
  frontend work is gated on AC5 landing (the value becoming non-null at runtime), not on a contract
  change.
- [ ] **AC7 — test proves the hole is closed** — A test demonstrates the end-to-end: a user with a
  recorded login surfaces a non-null `LastLoginAt` in the admin detail DTO (previously guaranteed
  null). Written TEST-FIRST.

## Out of scope
- Device/active-session enumeration & revoke UI — that is **IA-05** (separate ticket); do not build
  a session list here.
- Showing last-login on partner/customer self-profile surfaces — admin surface only for this ticket.
- Login-attempt / failed-login auditing, lockout, or per-device last-seen — only successful
  interactive login timestamp on the `User`.
- Any change to the `JwtTokenResponse` contract or token claims.

## Implementation notes
- **No DTO/contract field is added or removed** — `LastLoginAt` already exists on both admin DTOs
  and in the generated admin client, so **no nswag-regen is needed**. The only manual step is the
  **ef-migration** for the new `User.LastLoginAt` column (owner-run — flag it and HOLD the persisted
  read until the owner confirms the migration is applied).
- **Single write point.** All four login handlers call `TokenService.GenerateTokenAsync`
  (`Services/TokenService.cs:22`); record the login there on the success branch (after the
  `IsEmailConfirmed` early-return at lines 24-29, before/at access-token issuance) so the four paths
  stay consistent and the no-double-implementation rule holds. `RefreshToken` issues its access token
  via its own private `GenerateAccessToken` (`RefreshToken.cs:101`), so it will **not** be affected —
  that satisfies AC4 by construction; still add the regression test.
- **Persistence / commit.** Recording the login mutates a tracked `User`; do **not** call
  `CommitAsync()` in the handler/service — the `UnitOfWorkPipelineBehavior` commits (per
  `knowledge/patterns-backend.md`). Use the injected clock/`DateTimeOffset` source the codebase
  already uses for auditing, not `DateTime.UtcNow` inline, so the test can assert deterministically.
- **Domain method**, not a public setter — keep the `User` setter private (matches every other
  mutator on `User.cs`, e.g. `ConfirmEmail()`, `UpdateLanguagePreference()`).
- **Serialization cluster:** this ticket is **NOT** listed in any shared-file serialization cluster
  in `agents/backlog/TICKET-MAP.md`. It does touch `TokenService.cs` and the four `Features/Auth/*`
  login handlers — no other Wave-2 ticket in the map edits those, so it can run concurrently, but
  the PM must still avoid co-scheduling it with any unlisted auth-handler work. The four login
  handlers are edited by a single instance (do not fan out per-handler against the same files).
- **Governing ADR:** none of the three accepted ADRs (0001 authorization, 0002 outbox-dispatch,
  0003 rate-limiting) governs this login-tracking field, hence `adrs: []`. Follow the backend/db/
  frontend pattern catalogs in `agents/knowledge/`.
- **Build TEST-FIRST** per `agents/knowledge/testing.md` (red → green → refactor): the domain method
  test, the `TokenService` success/early-return tests, the refresh-untouched regression, and the
  mapper test predate the implementation (visible in the status log / commit order; Reviewer enforces
  Gate 6).
