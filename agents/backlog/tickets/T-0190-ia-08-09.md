## Context
Two related gaps in admin self-service, from the audit:

- **IA-08** â€” There is no admin "change my password" path. `AdminAuthController`
  (`src/Cleansia.Web.Admin/Controllers/AdminAuthController.cs`) exposes only `Login`/`RefreshToken`/
  `Logout`. `AdminUserController` lets an admin manage *other* admins but not rotate their own
  credentials. The customer/partner reset flow (`ChangePassword`,
  `src/Cleansia.Core.AppServices/Features/Users/ChangePassword.cs`) is `[AllowAnonymous]` and keyed off
  an emailed `ResetPasswordCode` â€” there is no *authenticated* change-my-password command. An admin who
  needs to rotate a password today depends on another admin or direct DB access.
- **IA-09** â€” `AdminUserDetailDto` (`src/Cleansia.Core.AppServices/Features/AdminUsers/DTOs/AdminUserDetailDto.cs:15-16`)
  exposes `BirthDate` and `PreferredLanguageCode`, but neither `CreateAdminUser.Command`
  (`CreateAdminUser.cs:15-20`) nor `UpdateAdminUser.Command` (`UpdateAdminUser.cs:13-17`) accepts them.
  Worse, `UpdateAdminUser.Handler` calls `user!.Update(firstName, lastName, phoneNumber)`
  (`UpdateAdminUser.cs:66-69`) and `User.Update` defaults `birthDate = null`
  (`src/Cleansia.Core.Domain/Users/User.cs:132-140`), so **every** admin edit silently wipes the stored
  `BirthDate`. The detail screen can display a birth date / language the edit form can never set, and a
  routine edit is a data-loss footgun.

Source: `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md` (IA-08 Â§660, IA-09 Â§669);
`agents/backlog/audits/AUDIT-2026-06-01-findings.json` (IA-08 Â§1100, IA-09 Â§1116).

Owner decision (per IA-08 fix note): admin password rotation is an **authenticated** "change my
password" endpoint on the admin host (current-password + new-password), **not** the anonymous
email-link reset reused. This avoids exposing the `[AllowAnonymous]` reset surface on the admin API.

## Acceptance criteria
- [ ] **AC1 (IA-09 data-loss closed)** â€” Given an admin row with a non-null `BirthDate`, When an admin
  edits only the name via `UpdateAdminUser`, Then `BirthDate` is unchanged in the DB. A unit/handler
  test asserting `BirthDate` is preserved across a name-only update is added and passes (currently it
  would be nulled by `User.Update(..., birthDate: null)`).
- [ ] **AC2 (IA-09 write contract)** â€” Given the create/update admin forms, When an admin submits
  `BirthDate` and `PreferredLanguageCode`, Then `CreateAdminUser.Command` and `UpdateAdminUser.Command`
  accept both fields, the handlers persist them via `User.Update(..., birthDate)` and
  `User.UpdateLanguagePreference(languageCode)` (`User.cs:181-185`), and a subsequent `GetAdminUserById`
  returns the values that were written. Validators reject an unknown `PreferredLanguageCode` and a
  future-dated `BirthDate`. Tests cover persist-and-read-back for both fields.
- [ ] **AC3 (IA-08 authenticated change)** â€” Given a logged-in admin, When they POST current + new
  password to the new authenticated admin endpoint, Then with a correct current password and a valid new
  password the password is changed (`User.UpdatePassword`) and the result is success; with a wrong
  current password the request is rejected with a business error and the password is unchanged. A handler
  test covers both branches.
- [ ] **AC4 (authz)** â€” Given the new authenticated change-password endpoint, When called, Then it is
  guarded so it can only change the caller's own credentials (admin id from the session/JWT, never from a
  client-supplied id) and is reachable by any authenticated admin (per ADR-0001). A test proves a caller
  cannot target another admin's id.
- [ ] **AC5 (admin-web surface)** â€” Given the admin app, When an admin opens their profile, Then there
  is a "change password" form and the admin create/edit form has `BirthDate` and `PreferredLanguageCode`
  inputs wired to the regenerated client, using `<cleansia-*>`/PrimeNG controls and `TranslatePipe` with
  keys present in all 5 locales (`en/cs/sk/uk/ru`). The edit form pre-fills existing `BirthDate`/language
  and a name-only save does not clear them.
- [ ] **AC6 (error parity)** â€” Every new `BusinessErrorMessage` key (e.g. wrong-current-password,
  invalid language code) has a matching `errors.*` translation in all 5 admin i18n files.

## Out of scope
- Admin email-link "forgot password" (anonymous) flow â€” explicitly not reused/added here; only the
  authenticated change is in scope.
- Partner/customer profile or password flows (already wired end-to-end per audit non-findings).
- IA-04 `LastLoginAt` tracking (separate ticket) â€” `AdminUserDetailDto.LastLoginAt` display is untouched.
- Reworking `User.Update` callers outside `UpdateAdminUser` (other features keep current behavior).

## Implementation notes
- **Serialization caveat:** IA-08/09 has no entry in the TICKET-MAP serialization clusters. **However,
  if AC4 requires a new logical permission constant**, that touches `Policy.cs` / `PolicyBuilder.cs`,
  which is the **BSP-1 â†’ IDA-SEC-04 â†’ SEC-DSP-01/02 â†’ SEC-EMP-01** cluster â€” do **not** run concurrently
  with any of those, and add the map row + `AssertComplete` per ADR-0001 (fail-closed). Prefer reusing an
  existing authenticated/`AdminOnly` policy to avoid the cluster entirely; confirm with architect during
  contract lock. **ADR-0001 (ADR-AUTHZ) governs** the endpoint's policy.
- **Backend (contract first, then handlers):**
  - Add `BirthDate` (`DateOnly?`) and `PreferredLanguageCode` (`string?`) to `CreateAdminUser.Command`
    (`CreateAdminUser.cs:15-20`) and `UpdateAdminUser.Command` (`UpdateAdminUser.cs:13-17`); validate
    language code against the supported set and `BirthDate` not in the future.
  - `UpdateAdminUser.Handler` (`UpdateAdminUser.cs:66-69`): pass `birthDate: command.BirthDate` and call
    `user.UpdateLanguagePreference(command.PreferredLanguageCode)`. `CreateAdminUser.Handler` likewise.
  - New authenticated change-password command/handler on the admin host using `User.UpdatePassword` +
    a current-password check (mirror the hash-compare used in `ChangePassword.cs` / `CheckIfPasswordSame`),
    exposed via `AdminAuthController` (`[Authorize]`, id from session) â€” not `AdminUserController`.
- **Manual steps:** none (no schema change â€” `BirthDate`/`PreferredLanguageCode` already exist on
  `User` and on `AdminUserDetailDto`). The admin DTOs/commands change shape, so the client must be
  regenerated before AC5 â€” flag **`manual_step: nswag-regen`** to the owner at `in_review` and **hold**
  the frontend slice until confirmed. (Not pre-listed in frontmatter `manual_steps` because the trigger
  is the backend contract change landing first; PM adds it when backend merges.)
- **Test-first (per `agents/knowledge/testing.md`):** write the failing tests for AC1 (BirthDate
  preserved on name-only update), AC2 (persist/read-back), AC3 (wrong vs. right current password), and
  AC4 (cannot target another id) **before** the fixes. Spawn a reviewer instance in parallel with the
  backend dev and again with the frontend dev.
