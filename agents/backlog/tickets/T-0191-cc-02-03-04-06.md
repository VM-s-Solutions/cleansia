## Context
Four related catalog-config gaps from the 2026-06-01 audit (slice report
`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md`, CC slice §2545-2649 and the catalog-security
slice §1962-2034). All four are confirmed in real code:

- **CC-02 / CAT-01 / CAT-SEC-01 (critical, data-integrity)** — `DeleteService` and `DeletePackage`
  hard-delete with an **existence-only** validator and no in-use guard.
  `DeleteService.Handler` calls `serviceRepository.Remove(service!)`
  (`src/Cleansia.Core.AppServices/Features/Services/DeleteService.cs:35`); `DeletePackage` is identical
  (`Features/Packages/DeletePackage.cs:30-37`). `OrderServices.ServiceId` is `ON DELETE CASCADE`
  (`src/Cleansia.Infra.Database/Migrations/20260519203658_Initial.cs:1734-1738`; `PackageServices`
  cascade `:1166-1170`), so a delete silently strips the line item from every historical order
  (including Completed/invoiced/receipted) — corrupting receipts and pay calc
  (`basePay = services × serviceRate`). `EmployeePayConfig → Service` is `Restrict`
  (`EmployeePayConfigEntityConfiguration.cs:61`), so a referenced row instead throws a raw 500.
  The sibling deletes already guard: `DeleteCurrency.cs:32-34,55-59`, `DeleteCountry.cs`, `DeleteLanguage.cs`
  all call `IsInUseAsync`; `IServiceRepository`/`IPackageRepository` have **no `IsInUseAsync` at all**.
- **CC-03 (critical, unreachable lifecycle)** — the `IsActive` "deactivated" state is **filtered on but
  never set**. Customer overview hides inactive rows on purpose
  (`Features/Services/GetServiceOverview.cs:17-23`, comment: *"Deactivated services are admin-only
  state and must not appear in the booking wizard catalog."*; `GetPackageOverview.cs:24`), but
  `UpdateService.Handler` never touches `IsActive`, there is **no** Activate/Deactivate command in
  `Features/Services` or `Features/Packages`, `ServiceFilter.cs` exposes only `SearchTerm`, and
  `service-management.facade.ts:103-120` only deletes. So the only way to retire a service is the
  dangerous hard-delete of CC-02 — no safe "retire but keep history" path.
- **CC-04 (major, write-locked field)** — `Currency.SetAsDefault(bool)` exists
  (`src/Cleansia.Core.Domain/Internationalization/Currency.cs:39-42`) and `IsDefault` is read
  (delete-protection `DeleteCurrency.cs:29,50`; sort `GetCurrencyOverview.cs:18`) but is **never set**:
  `CreateCurrency`/`UpdateCurrency` don't touch it and no `SetDefaultCurrency` command exists (the
  only `SetDefault` in the codebase is `SetDefaultSavedAddress.cs`). The platform default currency is
  frozen at its seed value forever.
- **CC-06 (major, missing model)** — `Language` has only `Code`/`Name`
  (`src/Cleansia.Core.Domain/Internationalization/Language.cs`), no `IsDefault` (unlike Currency/Country).
  Service/Package validators require a translation for **all 5** languages (`CreateService.cs:67-74`)
  with no designated fallback for partial data or a newly added 6th language. This is the only piece of
  the bundle that needs an **owner decision** before code (the audit flags it for `questions/open.md`).

`security_touching: true` because every new admin write command (Deactivate/Activate/SetDefault)
must be gated with a granular `[Permission(Policy.Can…)]` per **ADR-0001 (ADR-AUTHZ)**, not bare
`[Authorize]` — the exact deviation CAT-SEC-05 calls out elsewhere in this domain.

Depends on **T-0142** (the B6 soft-delete ADR): CC-02's preferred fix is *soft-deactivate referenced
catalog rows* rather than hard-delete, so the Deactivate-vs-Remove decision rule must already be
ratified. Do not start this ticket until T-0142 is `done`.

## Acceptance criteria
- [ ] **AC1 (CC-02 in-use guard, Service)** — Given a Service referenced by any `OrderService`,
  `PackageService`, or `EmployeePayConfig`, When `DeleteService` runs, Then it does **not** call
  `repo.Remove`; instead it returns
  `BusinessResult.Failure<Response>(new Error(nameof(command.ServiceId), BusinessErrorMessage.ServiceInUse))`,
  and per the T-0142 ADR a referenced row is soft-deactivated (`IsActive=false` via the rich
  domain/repo method) rather than hard-deleted. A new `IServiceRepository.IsInUseAsync` (checks
  `OrderServices` + `PackageServices` + `EmployeePayConfig`) backs the guard, mirroring
  `DeleteCurrency.cs:32-34,55-59`. A handler/validator test (mocked repo) proves an in-use service is
  not removed and the `ServiceInUse` error is returned; a not-in-use service still hard-deletes.
- [ ] **AC2 (CC-02 in-use guard, Package)** — Same as AC1 for `DeletePackage` /
  `IPackageRepository.IsInUseAsync` / `BusinessErrorMessage.PackageInUse`, with its own test.
- [ ] **AC3 (CC-02 error contract + i18n)** — Given the new `ServiceInUse` / `PackageInUse`
  `BusinessErrorMessage` constants (dot-notation, e.g. `service.in_use` / `package.in_use`), When the
  admin app surfaces them, Then a matching `errors.*` key exists in **all 5** locale files
  (`en/cs/sk/uk/ru`) and the admin facade renders it via `SnackbarService.showApiError` (note:
  the broader CAT-02 swallow-all-errors fix is a separate ticket; this AC only requires the two new
  keys are wired and translated).
- [ ] **AC4 (CC-03 deactivate/activate command)** — Given an active Service, When an authorized admin
  invokes `DeactivateService`, Then `IsActive=false` is set via the domain method, the row persists,
  and `GetServiceOverview`'s existing `Where(s => s.IsActive)` (`GetServiceOverview.cs:21`) hides it
  from the customer booking wizard while history survives; `ActivateService` reverses it. Package
  equivalents exist. Each command carries `[Permission(Policy.Can…)]` (ADR-0001). Handler tests assert
  the `IsActive` transition and the permission attribute is present.
- [ ] **AC5 (CC-03 admin filter)** — Given a mix of active and inactive services, When the admin list
  loads, Then `ServiceFilter` (and `PackageFilter`) expose an optional `IsActive?` passed through
  `GetPagedServices`/`GetPagedPackages`, and the admin facade renders an activate/deactivate status
  toggle + an active/inactive filter (replacing delete as the default retire path). An admin can list
  only retired services.
- [ ] **AC6 (CC-04 set default currency)** — Given two currencies where currency A is default, When an
  authorized admin invokes `SetDefaultCurrency(B)`, Then in **one transaction** A's `IsDefault` is
  cleared and B's is set via `Currency.SetAsDefault(...)` (`Currency.cs:39-42`); exactly one currency
  is default afterward. The command carries `[Permission(Policy.Can…)]` (ADR-0001) and is exposed on
  the admin currency endpoint + a "Set as default" action in `currency-management.facade.ts`, mirroring
  `SetDefaultSavedAddress.cs`. A handler test proves the prior default is cleared and exactly one
  default remains.
- [ ] **AC7 (CC-06 owner decision recorded first)** — Given CC-06 requires an owner choice, When this
  ticket reaches the CC-06 work, Then a `blocking: yes` entry exists in
  `agents/backlog/questions/open.md` asking the owner to choose **(a)** introduce `Language.IsDefault`
  + a `SetDefaultLanguage` flow + relax the "all-languages-required" validator to a fallback, **or**
  **(b)** formally document translations as mandatory-for-all and define add-a-language behavior.
  No CC-06 schema/code lands before the owner answers (the analyst/architect frame the question; PM
  holds the CC-06 sub-work).
- [ ] **AC8 (CC-06 implements the chosen path)** — Given the owner picked (a), When CC-06 ships, Then
  `Language.IsDefault` is added, `SetDefaultLanguage` (single-default, one transaction, `[Permission]`-
  gated per ADR-0001) exists, the translation-completeness validator
  (`CreateService.cs:67-74`/`UpdateService.cs:71-78`) is aligned to the fallback rule, and tests pin
  the single-default invariant + fallback resolution. If the owner picked (b), Then the validator/docs
  are updated to the documented rule and no `Language.IsDefault` column is added (and the
  `ef-migration` manual step is marked not-needed for CC-06). The status log records which path shipped.
- [ ] **AC9 (manual steps flagged)** — Given new commands/DTOs change the OpenAPI surface and CC-06(a)
  adds a `Language.IsDefault` column, When implementation completes, Then the ticket flags
  `MANUAL_STEP: nswag-regen` (admin client) and `MANUAL_STEP: ef-migration` to the owner and dependent
  frontend work is **held** until the owner confirms. Claude never runs these.
- [ ] **AC10 (tests precede code)** — Each behavioral AC maps to a test that appears **before/with** the
  implementation in the diff per `agents/knowledge/testing.md` (test-first; characterization test pins
  current `DeleteService`/`DeletePackage` behavior before the guard is added); the status log notes the
  red→green for each.

## Implementation notes
- **L size — MUST be split before `ready`** (PM constraint / `agents/process/ticket-lifecycle.md`).
  Suggested split along the natural seams (none share a file, so they can run as parallel sub-tickets
  once T-0142 is done, each with its own reviewer):
  (a) **CC-02** backend+db — `IsInUseAsync` on both repos + guard + soft-deactivate referenced rows +
      `ServiceInUse`/`PackageInUse` errors + 5 locales;
  (b) **CC-03** backend+frontend — Deactivate/Activate commands (Service+Package) + `IsActive?` on the
      filters/paged queries + admin toggle/filter UI;

  (d) **CC-06** analyst/architect → backend+db+frontend — owner question first (AC7), then the chosen
      path (AC8). (d) is gated on the owner answer and naturally lands last.
- **Built TEST-FIRST** per `agents/knowledge/testing.md`: write the characterization test pinning current
  `DeleteService`/`DeletePackage` hard-delete behavior first, confirm green, then red→green the guard.
  Handler/validator unit tests use mocked repos + `Cleansia.TestUtilities` builders; assert
  `Remove`/`Deactivate`/`SetAsDefault` invocation and the returned `Error`, not log strings.
- **ADRs that govern:** **ADR-0001 (ADR-AUTHZ)** — every new admin write command is
  `[Permission(Policy.Can…)]`-gated, reusing real `Policy.*` constants, never bare `[Authorize]`
  (cf. CAT-SEC-05). **T-0142's soft-delete ADR (B6)** — Deactivate-vs-Remove rule for the CC-02
  referenced-row path; this ticket *consumes* that ADR (T-0142 must be `done`).
- **Backend rules** (`agents/knowledge/patterns-backend.md`): handlers happy-path only; all checks in
  FluentValidation validators with `Cascade.Stop`; never call `CommitAsync()` (UnitOfWork pipeline
  commits); commands return `BusinessResult<T>`; DTOs are positional `record`s; error keys are
  `category.specific` constants in `BusinessErrorMessage` with a matching `errors.*` key in all 5
  locales. Mirror the proven sibling: `DeleteCurrency.cs` (in-use guard) and `SetDefaultSavedAddress.cs`
  (single-default-in-one-transaction).
- **Frontend rules** (`agents/knowledge/patterns-frontend.md`): logic in facades not components;
  `<cleansia-*>`/PrimeNG only; `TranslatePipe` on every string; no `any`; errors via
  `SnackbarService.showApiError`. Never hand-edit the NSwag client; flag `manual_step: nswag-regen`.
- **Serialization cluster:** **none.** None of the touched files
  (`DeleteService.cs`, `DeletePackage.cs`, `Currency.cs`, `Language.cs`, the catalog repos/filters,
  `service-management.facade.ts`, `currency-management.facade.ts`) appear in the TICKET-MAP shared-file
  map (TICKET-MAP §15-27), so this ticket is **collision-free** and its sub-tickets may run concurrently
  with other Wave-2 work — subject only to the `depends_on: T-0142` gate.
