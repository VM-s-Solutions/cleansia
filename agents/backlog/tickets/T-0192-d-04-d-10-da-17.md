## Context
Three customer-web (`cleansia.app`) feature gaps surfaced by the 2026-06-01 audit, all behind
backend that already exists and is already exposed on the customer host:

- **D-04 (major, webâ†”mobile parity)** â€” `UploadDisputeEvidence` is built and exposed on the customer
  host (`src/Cleansia.Web.Customer/Controllers/DisputeController.cs:64-93`, `CanUploadDisputeEvidence`

- **D-10 (minor)** â€” backend `DisputeFilter` supports a status filter (`GetPagedDisputes`), but the
  customer store passes only `offset/limit` (`dispute.actions.ts:8-11`, `dispute.effects.ts:17-22`

## Acceptance criteria
- [ ] **AC1 (D-04 render)** â€” Given a dispute whose `DisputeDetails.evidence` is non-empty, When the
      customer opens the detail dialog, Then each evidence item renders as a link/thumbnail using its
      SAS `blobUrl`, in the detail body of
      `disputes.component.html` (currently ends at the message thread, line 188-264).
- [ ] **AC2 (D-04 refund)** â€” Given a dispute with a non-null `DisputeDetails.refundAmount`, When the
      detail dialog is open, Then the agreed refund amount is shown (currency-formatted); when null it
      is hidden (the three-state render is respected, no empty label).
- [ ] **AC3 (D-04 upload)** â€” Given an eligible open dispute, When the customer picks a file via a
      `<cleansia-*>`/PrimeNG file control and confirms, Then the facade calls
      `customerClient.disputeClient.uploadEvidence(...)` through the C3 client-call pipe and on success
      re-loads the detail; the picker enforces the **same MIME/size whitelist the backend enforces**
      (mirror Android's client-side validation), and rejects out-of-whitelist files with a translated
      error before upload.
- [ ] **AC4 (D-10 status filter)** â€” Given the disputes list, When the customer selects a status in a
      new filter control, Then `loadCustomerDisputes` carries the status through
      `dispute.actions.ts` â†’ `dispute.effects.ts` into the corresponding `getPaged` filter argument
      (the slot currently passed `undefined`) and the list re-queries; clearing the filter restores
      the unfiltered list.
- [ ] **AC5 (D-10 unread)** â€” Given a dispute that has a staff message newer than the customer's last
      view of that dispute, When the list renders, Then that row shows an "unread"/"new reply" badge;
      opening the dispute clears the badge. (Implemented client-side via a per-dispute last-viewed
      timestamp â€” see Out of scope / open question; no backend change.)
- [ ] **AC6 (DA-17 manage)** â€” Given a logged-in customer, When they open a new saved-address
      management screen, Then it lists their saved addresses from the existing customer saved-address
      store and offers **Set default** and **Delete** actions wired to the existing
      `SavedAddressController` routes (`setDefault`, `delete`); set-default reflects the single-default
      invariant and delete removes the row from the list. The screen is reachable from customer
      navigation.
- [ ] **AC7 (i18n)** â€” Every new user-visible string has a `TranslatePipe` key present in **all 5**
      customer locales (`apps/cleansia.app/src/assets/i18n/{en,cs,sk,uk,ru}.json`); no hardcoded
      strings, no `any`.
- [ ] **AC8 (tests, TEST-FIRST)** â€” Jest facade specs (written before the implementation per
      `agents/knowledge/testing.md`) prove: evidence/refund render mapping and the empty/loading/error
      states; the upload success path (re-loads detail) and the client-side MIME/size rejection path
      (errorâ†’snackbar, no client call); the status-filter value reaches the load action; the
      unread-badge computation (new staff message after last-viewed â†’ badge; after view â†’ cleared);
      and the saved-address set-default/delete success + errorâ†’snackbar mappings.
