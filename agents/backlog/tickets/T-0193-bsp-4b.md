## Context

ADR-0003 (`agents/backlog/adr/0003-partitioned-rate-limiting.md`) ships **per-IP-only** partitioned
rate limiting in Wave 0 (T-0115). The ADR's "Residual risk" section names two classes it explicitly
does **not** close, both of which per-IP partitioning *structurally cannot* touch:

- **Distributed credential-stuffing** against `/Login` â€” a botnet of N IPs Ã— the per-IP window =
  N Ã— guesses/min against one account; each bot stays under its own IP window. (ADR-0003 Residual
  risk, bullet 1.)
- **Distributed confirmation-code guessing** â€” the worst surface, because the code space is small and
  the lookup is account-unbound. Verified in real code: `ConfirmUserEmail` looks up by **code alone**
  with no account binding (`Cleansia.Core.AppServices/Features/Auth/ConfirmUserEmail.cs:37,65` â†’
  `IUserRepository.GetByConfirmationCodeAsync`, implemented in
  `Cleansia.Infra.Database/Repositories/UserRepository.cs:45-48` as a global
  `FirstOrDefaultAsync(u => u.ConfirmationCode == token)`), the endpoint is `[AllowAnonymous]`
  (`Cleansia.Web.Customer/Controllers/AuthController.cs:58-67` + the Partner/Mobile equivalents), and
  the code is a **6-digit numeric** (`User.cs:95` `CreateWithPassword`, `User.cs:175`
  `UpdateConfirmationCode` â€” `Random.Shared.Next(100000, 999999)`, 15-min expiry). A 10^6 space with
  no per-code attempt cap is brute-forceable across a botnet in hours. Per-`sub` cannot apply
  (anonymous); per-IP cannot stop distribution.

The owner answered **Q-RATELIMIT-03 â†’ "SHIP IT"** (`agents/backlog/questions/answered.md`): Wave 0
ships per-IP-only and **this** (BSP-4b) is the **fast-follow**, not an in-wave blocker. This ticket is
that fast-follow: it adds the per-identity controls the partitioned limiter cannot â€” an
**account-lockout-after-N-failed-logins** counter and a **per-confirmation-code attempt counter** â€”
so the distributed residual is bounded by attempts-per-identity, not by source IP. ADR-0003 governs;
this ticket does **not** re-litigate the partition key or the limiter shape.

## Acceptance criteria

- [ ] **AC1 â€” Per-code attempt cap closes the confirmation-code hole.** Given a user with an active
  6-digit `ConfirmationCode`, When the `ConfirmUserEmail` path receives N wrong codes that resolve to
  that user's account (or, where no user resolves, N consecutive wrong-code attempts against the
  confirm surface within the window), Then the (N+1)-th attempt is rejected by an attempt-counter
  guard **before** a successful confirmation can be brute-forced, and a test proves a wrong-code spray
  that would have eventually hit a valid code is stopped at the cap. (ADR-0003 Residual risk bullet 2;
  hole at `ConfirmUserEmail.cs:37,65`.)
- [ ] **AC2 â€” Account-lockout-after-N-failed-logins.** Given an internal-auth user, When N consecutive
  failed `Login` attempts occur for that account (failure path today is
  `Login.cs:31-34` / `Handler` `Login.cs:81-85`), Then further login attempts for that account are
  refused for a cooldown window even from a **new IP** (proving distributed stuffing is bounded
  per-account, not per-IP), and a successful login or cooldown expiry resets the counter. A test proves
  attempt N+1 from a different IP is refused. (ADR-0003 Residual risk bullet 1.)
- [ ] **AC3 â€” Honest users are not locked out.** Given a legitimate user who fails login once or twice
  then succeeds (or confirms with the correct code on first try), Then no lockout/throttle fires and
  the success resets any partial counter â€” a test asserts a 1-2-failure-then-success sequence yields a
  normal token, and the counter is cleared.
- [ ] **AC4 â€” Lockout/throttle state is server-side and survives process restart.** Given the lockout
  and per-code attempt counters live on persisted state (new `User`/identity columns, e.g. a failed-
  attempt count + lockout-until timestamp, alongside the existing `ConfirmationCode` /
  `ConfirmationCodeExpiresAt` fields on `User.cs:50-52`), Then the guard is enforced by the
  application layer (validator/handler/repository), not by the in-process rate limiter, and a test
  confirms the counter is read from the store (not memory). (Distinct from T-0115's in-memory limiter.)
- [ ] **AC5 â€” Observable + S6-safe rejection.** Given a lockout or per-code-cap rejection, Then it
  returns a deterministic business error (a new `BusinessErrorMessage` constant in the existing
  `auth.*` namespace â€” e.g. `auth.account_locked` / `auth.too_many_attempts`, following the
  `category.specific_error` pattern in
  `Cleansia.Core.AppServices/Common/BusinessErrorMessage.cs`) with the corresponding `errors.auth.*`
  keys in all 5 locale files, and the log line for the rejection never logs the attempted code or the
  password (S6).
- [ ] **AC6 â€” No weakening of T-0115's limiter.** Given this ticket, Then it makes **zero** edits to
  `CleansiaStartupBase.cs`, the limiter policies, or the partition-key functions â€” the only
  rate-limiter interaction is that these per-identity controls sit *behind* the per-IP windows
  T-0115 already establishes. A grep proves no `AddPolicy`/`AddRateLimiter`/`AddFixedWindowLimiter`
  edits are in this diff. (Serialization safety â€” see Implementation notes.)
- [ ] **AC7 â€” Test-first evidence.** Given the tests for AC1-AC4, Then the diff shows the failing tests
  land before/with the implementation and the status log records redâ†’green with each AC mapped to a
  case, per `agents/knowledge/testing.md` (this is security + auth logic â†’ strict).
