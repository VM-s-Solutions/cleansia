## Context

ADR-0003 (T-0115) fixed *how* the rate limiter partitions callers but deliberately left *which*
endpoints carry a window out of scope, naming this follow-up explicitly: **"Partitioning is not
coverage. A correctly partitioned policy applied to *some* endpoints does not satisfy S5 for the
money/side-effect endpoints that carry no `[EnableRateLimiting]` at all â€” those remain S5 gaps
(tracked as `BSP-4d`)"** (`agents/knowledge/security-rules.md:89-92`; ADR-0003 Consequences
`agents/backlog/adr/0003-partitioned-rate-limiting.md:524-529`, Alternatives `:481-483`).

T-0115 preserved the policy names `"auth"`/`"interactive"` precisely so existing
`[EnableRateLimiting]` sites were untouched and so this coverage sweep can land as a pure
attribute-addition with no startup or limiter change. This ticket closes the S5 *coverage* gap by
adding `[EnableRateLimiting]` to the verified-uncovered money/side-effect endpoints that today have
no window â€” money (Stripe checkout/subscribe/swap), payroll mutations, dispute create/message/
evidence, recurring-booking writes, and device registration.

**Verified uncovered today** (zero `EnableRateLimiting` matches):
- `src/Cleansia.Web.Customer/Controllers/MembershipController.cs` â€” `Subscribe:15`, `Cancel:27`,
  `CreateCheckoutSession:46` (Stripe checkout â€” money), `SwapPlan:67`.
- `src/Cleansia.Web.Customer/Controllers/DisputeController.cs` â€” `CreateDispute:16`, `AddMessage:52`,
  `UploadEvidence:64`.
- `src/Cleansia.Web.Customer/Controllers/RecurringBookingController.cs` â€” Create/Update/SetActive/Delete.
- `src/Cleansia.Web.Customer/Controllers/DeviceController.cs` â€” `Register:19`.
- Partner host (`src/Cleansia.Web.Partner/Controllers/`): `PaymentController.cs`, `PayPeriodController.cs`,
  `PayConfigController.cs`, `EmployeePayrollController.cs`, `DisputeController.cs` â€” all 0 coverage.
- Mobile-customer equivalents (`src/Cleansia.Web.Mobile.Customer/Controllers/`): `MembershipController.cs`,
  `RecurringBookingController.cs`, `DeviceController.cs` â€” all 0 coverage (same audience surface).

Depends on T-0115 (the partitioned policies must exist before attributes can reference them) and is
**security_touching** â€” closes a named S5 gap, so the Security gate is mandatory.

## Acceptance criteria

- [ ] **AC1 â€” Money endpoints carry a window.** Given the codebase, Then `MembershipController`
  `Subscribe`, `Cancel`, `CreateCheckoutSession`, `SwapPlan` (Customer **and** Mobile.Customer hosts)
  each carry `[EnableRateLimiting("auth")]` (authenticated, money/Stripe side-effect â†’ the per-`sub`
  `auth` partition). `GetMine`/`GetPlans` (reads) are **not** changed.
- [ ] **AC2 â€” Payroll mutations carry a window.** Given the Partner host, Then the side-effecting
  (write) actions on `PaymentController`, `PayPeriodController`, `PayConfigController`,
  `EmployeePayrollController` carry `[EnableRateLimiting("auth")]`; pure read/list actions are not
  rate-limited (consistent with how `interactive` vs `auth` is split elsewhere â€” see Implementation notes).
- [ ] **AC3 â€” Dispute side-effects carry a window.** Given Customer, Mobile.Customer and Partner
  `DisputeController`, Then `Create`/`AddMessage`/`UploadEvidence` (and partner-side dispute mutations)
  carry `[EnableRateLimiting("auth")]`.
- [ ] **AC4 â€” Recurring-booking + device writes carry a window.** Then `RecurringBookingController`
  Create/Update/SetActive/Delete (Customer + Mobile.Customer) and `DeviceController` `Register`
  (Customer + Mobile.Customer) carry `[EnableRateLimiting("auth")]`.
- [ ] **AC5 â€” Webhook stays excluded.** Then no Stripe `webhook` action gains `"auth"`/`"interactive"`
  (a 429 to Stripe reads as a retry-trigger; webhooks are SEC-W3's separate `"webhook"` policy â€”
  ADR-0003 D5.2 / `:339`). Webhook actions remain unchanged.
- [ ] **AC6 â€” Hole closed, proven by test.** Given the new host harness from T-0115, When a single
  authenticated caller exceeds the per-`sub` `auth` window against a now-covered money endpoint
  (e.g. `Membership/Subscribe`), Then the request over the limit returns `429` with a `Retry-After`
  header; before this ticket the same flood returned `200`/`400` indefinitely (no window). At least
  one money endpoint and one payroll endpoint have such a coverage test (red against pre-ticket code,
  green after).
- [ ] **AC7 â€” No partition/limiter regression.** Then `CleansiaStartupBase.cs` is **not** modified by
  this ticket (no startup/limiter/pipeline change); only controller attribute additions + tests. The
  T-0115 verification grep (`AddFixedWindowLimiter` absent; policies via `AddPolicy`) still passes.

cluster (BSP-4 â†’ SEC-W3 â†’ PROD-CONFIG). T-0115 preserved the policy names specifically so coverage

**Governing ADR: ADR-0003** (`agents/backlog/adr/0003-partitioned-rate-limiting.md`). The exact
uncovered surface is enumerated in its Consequences `:524-529` and in `security-rules.md Â§S5
:89-92`. D1 (`:124-125`) guarantees attribute-only addition: names are stable, so each endpoint just
gains an attribute.

**Attribute form (match the existing canonical site).** Follow
`src/Cleansia.Web.Customer/Controllers/PaymentController.cs:16` exactly:
`using Microsoft.AspNetCore.RateLimiting;` + `[EnableRateLimiting("auth")]` on the action method
(authenticated money/side-effect mutations â†’ `"auth"`, which routes authenticated callers to the
per-`sub` 30/min partition; anonymous to the per-IP 10/min â€” both from T-0115/D6). Reserve
`"interactive"` for read-mostly/idempotent UI chatter if a borderline action warrants the looser
60/min window â€” justify any such choice in the diff. Reads/lists get no window.

**TEST-FIRST (`agents/knowledge/testing.md`).** New endpoint behavior (a window where there was
none) is integration-level: write the coverage test first against the **new host harness built in
T-0115** (the MediatR/DbContext-level `BaseIntegrationTest`/`PostgresContainerFixture` cannot exercise
limiter middleware). Red (pre-attribute: flood returns success forever) â†’ green (429 + `Retry-After`
past the window) per AC6. The status log must show redâ†’green and map AC6 to its cases; the Reviewer
enforces Gate 6 against `testing.md` and ADR-0003's reviewer checklist item on coverage.

**Security gate mandatory** (`security_touching: true`): closes a named S5 coverage gap. Reviewer +
security verdicts required before `done`. Residual classes (distributed stuffing, confirmation-code
guessing) remain BSP-4b's â€” do not re-litigate here.
