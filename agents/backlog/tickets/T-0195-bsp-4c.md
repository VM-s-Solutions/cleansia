## Context

ADR-0003 (`agents/backlog/adr/0003-partitioned-rate-limiting.md`) gives the *server* a partitioned
rate limiter that rejects with `429` and sets a `Retry-After` header (D6): the limiter prefers the
lease's actual `RetryAfter` metadata, falling back to `window + 0–15s jitter`, precisely so rejected
clients **desync** instead of all re-spiking at the fixed-window rollover. ADR-0003 D6 explicitly
files the **client-side** counterpart as the non-blocking follow-up **BSP-4c**: *"A client-side
back-off-jitter follow-up for the SPA/mobile clients is filed as BSP-4c"* (ADR-0003 §D6, last line;
also Consequences → "Tracked Wave-0 follow-ups" → BSP-4c). This ticket is that follow-up.

Today the clients do not honor `Retry-After` at all and do not back off:

- **SPA (all three apps).** The shared `HttpErrorInterceptorFn`
  (`src/Cleansia.App/libs/core/services/src/lib/interceptors/http-error.interceptor.ts:12-48`)
  catches every non-OK, non-`404` response — including `429` — and immediately shows an error
  snackbar (`:38-41`) then re-throws. It never reads `Retry-After`, never waits, never retries. The
  per-app error interceptors (e.g.
  `libs/core/customer-services/src/lib/interceptors/error.interceptor.ts:22-42`) only handle `401`
  refresh and pass `429` straight through. So a `429` surfaces as an instant generic error and any
  client-side caller that retries does so with no delay — the exact thundering-herd that the
  server's jitter (D6) is trying to prevent. The shared interceptor is wired into all three SPAs via
  `COMMON_INTERCEPTORS_FN`
  (`libs/core/services/src/lib/interceptors/index.ts:7-10`, consumed in each app's `app.config.ts`,
  e.g. `apps/cleansia.app/src/app/app.config.ts:84-87`), so one shared change covers customer +
  partner + admin.

- **Android (both apps, shared `:core`).** The shared `NetworkErrorInterceptor`
  (`src/cleansia_android/core/.../auth/NetworkErrorInterceptor.kt:29-49`) only surfaces a snackbar
  for `IOException` and `5xx`; a `429` falls straight through to the call site with no back-off and
  no `Retry-After` handling. OkHttp clients are assembled per app in `NetworkModule.kt`
  (e.g. `partner-app/.../core/network/NetworkModule.kt:199-212` `provideAuthOkHttp`); there is no
  429-aware interceptor in the chain. The existing single-flight retry precedent to mirror is
  `AuthAuthenticator` (401-refresh-retry) and the SPA `CustomerRefreshCoordinator`.

This is a resilience/UX hardening only — it changes how clients *react* to a `429`, not the server
contract. No backend, DTO, or NSwag change.

## Acceptance criteria

- [ ] **AC1 — SPA honors `Retry-After` with a single bounded retry.** Given a request that returns
  `429` carrying a `Retry-After` header, When it flows through the shared SPA interceptor, Then the
  client waits the header's delay **plus a small random jitter** before **one** automatic retry; if
  the retry also `429`s (or no attempts remain), the error surfaces as today. (Closes the hole;
  grounded in `http-error.interceptor.ts` which currently does none of this.)
- [ ] **AC2 — SPA jitter desyncs the herd.** Given two SPA requests rejected in the same window with
  the same `Retry-After`, Then their computed wait times differ by the jitter (they do not retry at
  the same instant). (Mirrors server D6 intent.)
- [ ] **AC3 — SPA fallback when header absent.** Given a `429` with **no** `Retry-After`, Then the
  client backs off a default window (1 min) + jitter rather than retrying immediately, and still
  caps at one retry.
- [ ] **AC4 — SPA does not double-snackbar or retry forever.** Given a `429`, Then at most one
  retry occurs and **at most one** error snackbar is shown (only after the retry is exhausted) — the
  current immediate snackbar at `http-error.interceptor.ts:38-41` must not fire on the first `429`.
- [ ] **AC5 — Android honors `Retry-After` with one bounded retry.** Given an OkHttp response with
  code `429`, When it passes through the new shared `:core` back-off interceptor, Then the
  interceptor sleeps for the parsed `Retry-After` delay (seconds, or absent → default window) **plus
  jitter** and retries the request **once**; a second `429` is returned to the caller unchanged.
- [ ] **AC6 — Android retry is cancellation-safe.** Given the call's coroutine/connection is
  cancelled during the back-off wait, Then the wait aborts and no retry is issued (consistent with
  `networkCall` / `NetworkErrorInterceptor` cancellation handling at
  `NetworkCall.kt:52-56`, `NetworkErrorInterceptor.kt:71-85`).
- [ ] **AC7 — Android back-off interceptor wired in both apps.** Then the new interceptor is added to
  the auth OkHttp chain in **both** `NetworkModule.kt`s (partner `:199-212`, customer equivalent),
  ordered so it wraps retries around the auth/timezone/logging interceptors.
- [ ] **AC8 — Bounded, no silent infinite loop.** On both platforms the maximum total attempts is a
  single retry (one back-off), the jitter range is 0–15s (matching server D6), and the back-off
  honors `Retry-After` values that exceed it; refresh/login/auth paths are not affected by this
  change.
- [ ] **AC9 — Tests prove it (TEST-FIRST).** A Jest spec for the SPA interceptor (fake-time the
  `Retry-After` wait, assert single retry, jitter desync, header-absent fallback, no first-attempt
  snackbar) and a JVM/OkHttp `MockWebServer` test for the `:core` interceptor (429-then-200 →
  retried after delay; 429-then-429 → surfaced; cancellation aborts) exist and pass; each AC maps to
  a case.

## Out of scope

- Any server/limiter change — that is T-0115 (ADR-0003 deliverable 1). This ticket only consumes the
  `Retry-After` the server already emits.
- Changing which endpoints carry a limiter (BSP-4d) or account-lockout throttle (BSP-4b).
- Multi-step exponential back-off or a retry queue — a **single** bounded retry with jittered delay
  is the agreed scope (ADR-0003 D6 is about desyncing the herd, not exhaustive retry).
- Retrying non-`429` statuses, idempotency tagging of non-idempotent POSTs beyond what is already
  safe, or touching the `401` refresh path (`error.interceptor.ts`, `AuthAuthenticator`).
- iOS — no iOS client exists in `src/` yet; when it lands it inherits this behavior as a sibling
  ticket.
- No NSwag regen, no EF migration, no backend.

## Implementation notes

**Serialization cluster (TICKET-MAP):** none. This ticket is **not** in any shared-file
serialization cluster — the server-side `CleansiaStartupBase.cs` cluster
(**BSP-4 → SEC-W3 → PROD-CONFIG**) is backend-only; this is purely client code in different repos.
It edits the shared SPA interceptor + the two Android `NetworkModule.kt`s, none of which collide
with another open ticket. (DA-9, the only other frontend+mobile shared-constants ticket, touches
CZE/Mapbox/2000-char constants, not the network layer — no overlap.)

**Governing decision: ADR-0003 D6** — read it for the exact jitter range (0–15s) and the
"prefer lease metadata, else window+jitter" shape the *server* uses; the client mirrors it so the
two stay coherent. This ticket has **no** ADR authority of its own — it is a faithful client
consumer of the D6 contract.

**Depends on T-0115** (ADR-0003 deliverable 1): the server must actually emit `429 + Retry-After`
before there is anything for the client to honor. Hold this ticket until T-0115 is `done`.

Edit sites:
- **SPA:** extend the shared interceptor — `http-error.interceptor.ts:12-48`. Add a `429` branch
  that reads `error.headers.get('Retry-After')`, computes `delay = (parsed ?? 60s) + random(0,15s)`,
  and performs a single `retryWhen`/`delay`-style re-dispatch via `next(req)` before falling through
  to the existing snackbar. Suppress the snackbar (`:38-41`) on the first `429`. Keep it in
  `COMMON_INTERCEPTORS_FN` (`index.ts:7-10`) so all three apps inherit it; no per-app
  (`app.config.ts`) wiring change needed.
- **Android:** add a new shared interceptor in `:core`
  (`src/cleansia_android/core/.../network/` next to `NetworkCall.kt`), parse `Retry-After`
  (delta-seconds), back off with jitter, retry once via `chain.proceed`. Register it in the auth
  chain of **both** apps' `NetworkModule.kt` (partner `provideAuthOkHttp:199-212`, customer
  equivalent). Respect cancellation like `NetworkErrorInterceptor`/`networkCall` already do.

**TEST-FIRST (`agents/knowledge/testing.md`).** This is client resilience **logic** (delay math,
jitter, single-retry bound, header parsing) → test-first is mandatory for the logic, test-alongside
for any view. SPA: Jest spec on the interceptor with fake timers. Android: JVM unit test with OkHttp
`MockWebServer` (429→200 retried; 429→429 surfaced; cancel aborts). The status log must show
red→green and each AC mapped to a case (testing.md "How the ticket shows TDD happened").

**Gates:** reviewer per developer (frontend dev + Android dev run concurrently, each with a
reviewer). `security_touching: false` → no security gate. QA verifies the observable retry/back-off
on a live `429` from a T-0115 host. Not a hot path → optimizer not required.
