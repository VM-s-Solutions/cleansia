## Context
Wave 3 is consistency/quality cleanup: the same operation written several ways. The variance audit
`audits/consistency-violations.md` recorded the deviations against the canonical forms in
`knowledge/consistency.md`, and `tools/check-consistency.mjs` machine-detects a subset (187-violation
baseline). This ticket consolidates the **mechanical, behavior-preserving** canonicalizations from
the Wave-3 epic — the original per-finding tickets `T-0001..T-0008, T-0010..T-0015` (source field) —
into one sweep, scoped to the clusters that are pure refactors with no contract or behavior change:

- **Paged-query A\*** (T-0001/A1+A3+A5, T-0002/A7, T-0003/A6):
  - `Features/PromoCodes/Admin/GetPagedPromoCodes.cs` and `Features/Referrals/Admin/GetPagedReferrals.cs`
    use `record Query` + inline `Offset`/`Limit` + bespoke `repo.GetPagedAdminAsync(...)` + hand-built
    `new PagedData<T>` (A1/A3/A5).
  - `Features/PayConfig/GetPagedPayConfigs.cs` has `Filter { get; set; }` → must be `{ get; init; }` (A7).
  - `Features/Services/GetPagedServices.cs` has `AsNoTracking` before `Include` and projects after
    materialization → restore canonical read-path order (A6).
- **Command Response B1** (T-0004): `Features/Disputes/CreateDispute.cs` (`ICommand<string>`),
  `Features/Disputes/UpdateDisputeStatus.cs` and `Features/SavedAddresses/DeleteSavedAddress.cs`
  (bare `ICommand`) → wrap the returned id/flag in a `record Response(...)`.
- **Validator base B3** (T-0006): `Features/PayConfig/{Create,Update}PayConfig.cs`,
  `Features/PayPeriods/{Create,Update}PayPeriod.cs` (`UserEmailValidator<Command>`),
  `Features/Employees/UpdateEmployee.cs` + `Features/Users/UpdateCurrentUser.cs`
  (`BaseUserValidator<Command>`) → inherit `AbstractValidator<Command>` and compose the shared rule
  via `.SetValidator(...)`. (Move the *ownership* check out of the validator only in the
  B4/security ticket, not here — see Out of scope.)
- **List facades C\*** (T-0010/C1, T-0011/C2+C3+C8):
  - `cleansia-customer-features/order-wizard/.../order-wizard.facade.ts`,
    `.../recurring-bookings.facade.ts`, `.../rewards/rewards.facade.ts` use
    `DestroyRef`/`takeUntilDestroyed`/bare `firstValueFrom` → extend `UnsubscribeControlDirective`
    + `takeUntil(this.destroyed$)` (C1).
  - `cleansia-admin-features/fiscal-failures/.../fiscal-failures-list.facade.ts` omits a
    `totalRecords` signal (C2); `cleansia-partner-features/invoices/.../invoices.facade.ts` resets
    `loading` inline in `catchError` instead of `finalize` (C3); `cleansia-partner-features/orders/.../orders.facade.ts`
    mixes NgRx into single-feature state (C8).
    - **NOTE (collision-reconciled 2026-06-01):** `cleansia-customer-features/disputes/.../disputes.facade.ts`
      is **explicitly OUT of this ticket's C8 scope** — it is fully rewritten by **T-0202** (DA-5/DA-6/PERF-F1)
      onto the cleansia-table archetype, and T-0202 owns the NgRx-vs-signals decision for that facade. Do
      not touch `disputes.facade.ts` here (it would collide with + contradict T-0202). T-0202 depends_on
      T-0196 so it rebases on the canonical base.
- **Sealed Android states E1/E2** (T-0013/E1, T-0014/E2):
  - partner `features/auth/viewmodels/LoginViewModel.kt`, `features/orders/viewmodels/OrderDetailsViewModel.kt`,
    `features/earnings/viewmodels/EarningsSummaryViewModel.kt`, `features/dashboard/viewmodels/DashboardViewModel.kt`
    use flag-bag `data class …UiState` → sealed interface `Loading`/`Error`/`Loaded` (E1).
  - customer `features/disputes/CreateDisputeViewModel.kt`, `features/membership/MembershipViewModel.kt`,
    `features/profile/ProfileViewModel.kt` use loose `_submitting`/`_error`; partner uses an
    `enum OrderAction inFlight` → standardize on the shared `ActionState`
    (`customer-app/.../ui/state/ActionState.kt`, already exists) + a `SharedFlow(replay=0)` success
    effect (E2).

These are refactors, **not** behavior changes. AC = behavior unchanged (characterization test green)
+ the smell removed + `check-consistency.mjs` clean for the touched area.

## Acceptance criteria
- [ ] **AC1 (TEST-FIRST, A\*)** — Given the existing paged-query behavior of
  `GetPagedPromoCodes`, `GetPagedReferrals`, `GetPagedPayConfigs`, `GetPagedServices`, When a
  characterization test pins the current paging/filter/sort/total output for each (written and green
  *before* the refactor per `testing.md` "changing existing untested code"), Then after converting
  each to the canonical paged-query form (A1 `class Request : DataRangeRequest`, A3 Specification,
  A4 `GetPagedSort<XxxSort>` + `GetCountAsync`, A5 `items.MapToDto(total, request)`, A6
  `Include → AsNoTracking → Select(MapToDto) → ToListAsync`, A7 `Filter { get; init; }`) the same
  characterization tests stay green — identical items, total, and page metadata for the same inputs.
- [ ] **AC2 (TEST-FIRST, B1)** — Given `CreateDispute`/`UpdateDisputeStatus`/`DeleteSavedAddress`,
  When a handler test pins the current success/failure outcome, Then each command returns
  `ICommand<Response>` with a `record Response(...)` wrapping the prior scalar/flag, the returned
  value is equivalent to before, and the controller still returns the same payload (no wire change
  beyond the Response wrapper — see Implementation notes for the nswag note).
- [ ] **AC3 (TEST-FIRST, B3)** — Given the validators currently inheriting `UserEmailValidator<Command>`
  / `BaseUserValidator<Command>`, When a validator test pins every current `BusinessErrorMessage`
  code each can emit (valid input passes; each rule's code fires), Then after switching to
  `AbstractValidator<Command>` + `.SetValidator(...)` composition the identical set of codes fires
  for the identical inputs — no rule added, removed, or reordered.
- [ ] **AC4 (TEST-FIRST, C\*)** — Given the customer facades using `DestroyRef`/`takeUntilDestroyed`/
  `firstValueFrom` and the list facades missing `totalRecords` / resetting `loading` in `catchError` /
  mixing NgRx, When a Jest facade spec pins the current state transitions (empty/loading/error,
  totals, and load-then-error resets `loading`), Then after migrating to `UnsubscribeControlDirective`
  + `takeUntil(this.destroyed$)` (C1), adding the `totalRecords` signal (C2), the canonical
  `takeUntil → catchError(() => of(null)) → finalize` pipe (C3), and moving single-feature state off
  NgRx into signals (C8), the specs stay green — same observable state for the same calls.
- [ ] **AC5 (TEST-FIRST, E1/E2)** — Given the partner flag-bag `…UiState` data classes and the loose
  one-shot action booleans, When a ViewModel test (or, if no Android harness exists yet, a recorded
  state-sequence reviewed against the screen) pins the current Loading→Loaded→Error sequence and the
  one-shot submit success/error path, Then after converting to a sealed `…UiState`
  (`Loading`/`Error`/`Loaded`, E1) and the shared `ActionState` + `SharedFlow(replay=0)` (E2) the
  rendered sequence is unchanged and impossible flag combinations are no longer representable.
- [ ] **AC6 (consistency gate)** — `node agents/tools/check-consistency.mjs --paths=<each touched dir>`
  reports zero A1/A5/B1/B3/C1/C2/C3/E1/E2 violations for every file this ticket touches, and the
  global baseline drops by the count this ticket clears (documented in the status log). No new
  violations introduced elsewhere.
- [ ] **AC7 (behavior unchanged, overall)** — `dotnet test src/Cleansia.Tests`, the touched Jest
  specs, and the Android module build/tests are green; no DTO field, error code, route, or rendered
  output changed beyond the mechanical Response/UiState wrappers; the Reviewer confirms the diff is
  refactor-only.

## Implementation notes
- **Canonical forms** are in `knowledge/consistency.md`: paged queries §A (A1–A8), commands §B
  (B1, B3), frontend list features §C (C1, C2, C3, C8), mobile §E (E1, E2). Full code samples in
  `knowledge/patterns-backend.md` / `patterns-frontend.md` / `patterns-mobile.md`. Each change must
  match the canonical form exactly — the Reviewer enforces consistency.md.
- **TEST-FIRST per `testing.md`** ("When you're changing existing untested code"): for every unit in
  every cluster, write a **characterization test first** that pins current behavior and confirm it is
  green, *then* refactor on top and keep it green (red→green not applicable — behavior is unchanged;
  the test is the safety net). Status log must note the characterization test landing before each
  refactor; PR with implementation-first for these units fails Gate 6.
- **Serialization (collision map, `TICKET-MAP.md`):** none of the files here are in a shared-file
  cluster, but this is an L sweep across many files — run the clusters as **independent sub-streams**
  (A* backend, B1 backend, B3 backend, C* frontend, E1/E2 android) with a reviewer per developer
  instance, and **never run two instances editing the same file**. The three backend clusters
  (A*/B1/B3) touch disjoint Features folders, so they can run concurrently; serialize only if any two
  land in the same file.
- **Serialization note (NSwag):** B1 adds a `Response` record to three commands. If the generated
  client's response *type* changes for `CreateDispute`/`UpdateDisputeStatus`/`DeleteSavedAddress`,
  flag `manual_step: nswag-regen` to the owner and hold the consuming frontend/mobile until confirmed.
  Prefer shaping the `Response` so the serialized JSON is unchanged (wrap the existing id/flag under
  the same property name) to avoid a regen; the Backend dev confirms the OpenAPI diff in review. This
  ticket's frontmatter lists no manual_steps on the assumption of wire-compatible Response shapes —
  the PM adds `nswag-regen` and holds dependents if the OpenAPI diff shows a change.
- **A\*:** introduce the missing `XxxSpecification` / `XxxSort` for promo-codes and referrals if absent
  (mirroring `GetPagedDisputes`); the bespoke `repo.GetPagedAdminAsync` call sites become the
  canonical `repository.GetPagedSort<XxxSort>(...)` + `GetCountAsync(...)`. Keep the exact filter
  semantics the characterization test pins.
- **E2:** reuse the existing `customer-app/.../ui/state/ActionState.kt`; do not introduce a parallel
  type. For partner, replace the `enum OrderAction inFlight` with `ActionState` + `SharedFlow`.
- This is an **L** ticket — it must be split before it goes `ready` (Definition of Ready / PM
  constraint: no L runs). The natural split is one ticket per cluster (the original
  T-0001/T-0002/T-0003/T-0004/T-0006/T-0010/T-0011/T-0013/T-0014), each S/M and independently
  reviewable; this file is the consolidating epic that scopes and grounds them.
