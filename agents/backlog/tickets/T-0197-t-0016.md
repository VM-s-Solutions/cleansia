## Context

`consistency.md` **E5** fixes the canonical mobile repository contract: a repository **returns
`ApiResult<T>`** (the sealed `Success`/`Error` type) and the **ViewModel** surfaces the snackbar.
The rule is an explicit *judgment call* (see consistency.md "Judgment calls" → "E5 `ApiResult<T>`"):
the two Android apps disagree, and we canonicalize on the more explicit contract because it carries
the error, enables retry, and keeps UI concerns out of the data layer. **Judgment calls are
Architect-owned; this is why the ticket needs its own ADR rather than an ad-hoc edit.**

Today the contract is split across the two apps:

- **partner-app (canonical):** `ApiResult<T>` is defined at
  `src/cleansia_android/partner-app/src/main/java/cz/cleansia/partner/core/network/ApiResult.kt`
  (`sealed class ApiResult<out T>` with `Success`/`Error(ApiError)`, plus `map`/`onSuccess`/`onError`).
  Repos like `partner/data/orders/OrdersRepository.kt` already return `ApiResult<…>` and use
  `safeApiCall { … }`.
- **customer-app (legacy `T?`-with-snackbar-in-repo):** every repo under
  `src/cleansia_android/customer-app/src/main/java/cz/cleansia/customer/core/**/*Repository.kt`
  returns a nullable/sentinel value and **surfaces the snackbar inside the repo** via
  `SnackbarController`. e.g. `core/orders/OrderRepository.kt` (`getById(): OrderDetailDto?`,
  `cancel(): CancelOrderResponse?`, `refresh(): String?`) and `core/disputes/DisputeRepository.kt`
  (`create(): String?`, `addMessage(): Boolean`) parse errors with
  `ApiErrorParser.parseToUserMessage(...)` then call `snackbar.showError(msg)` and return `null`/
  `false`/`emptyString`/`emptyList`.

`check-consistency.mjs` flags this with rule **E5** (the tool's mobile pass, the
`suspend fun …): T?` → `'Repository returns a nullable body (legacy) — prefer ApiResult<T>'` check):
every legacy customer-app repo method trips it. This ticket is the cross-cutting refactor the tool's
message calls a "tracked migration".

Two structural prerequisites the ADR must settle, because they make this cross-app and not a
same-day edit:
1. **`ApiResult<T>`/`ApiError`/`safeApiCall` must move from `partner-app` to the shared `:core`
   module** (alongside the already-shared `cz.cleansia.core.network.NetworkCall.kt`) so customer-app
   **and the incoming iOS app** consume one contract. partner-app then re-points its imports.
2. **iOS** has no repos yet, but the ADR fixes the equivalent Swift result contract so the iOS app is
   born on the canonical shape rather than re-deriving customer-app's legacy form.

This is a **behavior-preserving refactor** (the same calls succeed/fail the same way; only *where* the
error is turned into a snackbar moves from repo → ViewModel). It is `L` and must be **split before it
goes `ready`** (architect ADR + `:core` type move; then one child ticket per customer-app repo +
its ViewModels, serialized so no two children edit the same files; then iOS contract). Each child
ticket is sized `S`/`M` and characterization-tested.

## Acceptance criteria

> This is a Wave-3 consistency refactor: **behavior is unchanged**; the smell (legacy `T?`/sentinel
> repo contract with in-repo snackbar) is removed; `check-consistency.mjs` is clean for the touched
> area. Per `testing.md`, the change is **characterization-test-first** (pin current behavior, see it
> green, then refactor under it).

- [ ] **AC1 (ADR — architect)** — Given the E5 judgment call is Architect-owned, When this ticket is
  worked, Then a new ADR is authored and accepted that (a) canonicalizes `ApiResult<T>` as the mobile
  repository contract, (b) decides the move of `ApiResult`/`ApiError`/`safeApiCall` from partner-app
  into `:core` (`cz.cleansia.core.network`), and (c) states the equivalent iOS Swift contract. The
  ADR number is recorded in this ticket's `adrs:` frontmatter and in `consistency.md`'s E5 reference.
- [ ] **AC2 (shared type move)** — Given `ApiResult` lives in `partner-app/core/network`, When the
  `:core` move lands, Then `ApiResult`/`ApiError`/`safeApiCall` resolve from
  `cz.cleansia.core.network.*`, partner-app compiles against the moved type (imports re-pointed, **no
  behavior change** to partner repos/VMs), and both Android apps build green.
- [ ] **AC3 (characterization first)** — Given a customer-app repo method currently returns a
  nullable/sentinel and surfaces a snackbar in the repo, When the migration child ticket starts, Then
  a characterization test pins the *current* observable behavior (success returns the body; failure
  fires exactly one snackbar with the `ApiErrorParser`-derived message and the call yields the failure
  sentinel) and is **green before** any production code changes. (`OrderRepositoryTest.kt` and
  `AuthRepositoryTest.kt` already exist as the harness pattern to follow.)
- [ ] **AC4 (contract migrated, behavior identical)** — Given the characterization test is green, When
  each customer-app `*Repository.kt` method is migrated, Then it returns `ApiResult<T>` (with `Unit`
  for fire-and-forget, e.g. former `addMessage(): Boolean`), the snackbar call is **moved into the
  consuming ViewModel** (per E3's "localize at the layer that surfaces the snackbar"), and the
  characterization test still asserts the **same** success/failure outcome and the **same** single
  snackbar message (now raised by the VM) — i.e. behavior is observably identical end-to-end.
- [ ] **AC5 (ViewModels updated, no flag-bag regressions)** — Given repos now return `ApiResult<T>`,
  When the consuming customer-app ViewModels are updated to branch on `Success`/`Error`, Then they map
  `Error` to the snackbar via `onError`/`errorOrNull()` and do not reintroduce any E1/E2 flag-bag
  state; existing UiState shapes are preserved (this ticket does **not** also do the E1/E2 sealed-state
  migration — see Out of scope).
- [ ] **AC6 (smell removed; tool clean)** — Given all in-scope customer-app repos are migrated, When
  `node agents/tools/check-consistency.mjs mobile --paths=src/cleansia_android/customer-app` is run,
  Then there are **zero E5 violations** for the touched area, and the entry for these repos is cleared
  from `agents/backlog/audits/consistency-violations.md`.
- [ ] **AC7 (build + tests green)** — Given the migration is complete, When the Android build and unit
  tests run, Then both apps compile and all characterization + existing repo tests pass (red→green
  recorded in the status log per testing.md).

## Out of scope

- **E1/E2 sealed-UiState / `ActionState` migration** of customer-app ViewModels (the flag-bag fixes
  for `CreateDisputeViewModel`, `MembershipViewModel`, `ProfileViewModel`, partner UiStates) — those
  are their own Wave-3 tickets; here we only change the **repo→VM error channel**, not the UiState shape.
- **E7 directory/naming unification** (inline-singular `features/<name>/` convention) — separate ticket.
- Any **behavior change**: no new error messages, no retry UX, no caching/pagination changes, no new
  endpoints. Same successes, same failures, same single snackbar per failure.
- partner-app repo logic (it is already canonical) — partner is touched **only** to re-point imports
  after the `:core` type move (AC2).
- Writing the iOS repos themselves — iOS scope here is **only** fixing the Swift result contract in
  the ADR so the future app is born canonical (no iOS source exists yet).

## Implementation notes

- **Canonical pattern:** `consistency.md` **E5** (repo returns `ApiResult<T>`, VM surfaces snackbar)
  and **E3** (localize the error at the snackbar-surfacing layer). The judgment-call rationale is in
  consistency.md's "Judgment calls → E5 `ApiResult<T>`" — **do not** reverse it ad hoc; AC1's ADR is
  the only sanctioned way to revisit it.
- **TEST-FIRST (testing.md):** this touches existing, partly-untested code → write a
  **characterization test first** that pins current behavior, confirm it passes, then TDD the
  refactor on top (red→green→refactor, green throughout). The status log must note
  `characterization green → migrate → green`. Use the existing
  `customer-app/src/test/.../OrderRepositoryTest.kt` / `AuthRepositoryTest.kt` as the harness shape;
  don't hand-roll fixtures. Pure-mapping helpers (e.g. `safeApiCall` error→`ApiError`) get strict
  red-green unit tests.
- **Serialization / collision-avoidance:** this is `L` and **must be split before `ready`**.
  Suggested split (each child is `S`/`M`, none editing the same files concurrently):
  (1) ADR (architect) — AC1; (2) `:core` type move + partner import re-point — AC2; then one child
  **per customer-app repo + its ViewModels**, run **serially** (each repo file + its VM files is a
  distinct cluster): `core/orders/OrderRepository.kt` (+ `OrderDetailViewModel`,
  `OrderPhotosViewModel`, booking/recurring consumers), `core/disputes/DisputeRepository.kt`
  (+ `CreateDisputeViewModel`, `DisputeDetailViewModel`, `DisputesListViewModel`),
  `core/data/AddressRepository.kt`, `core/memberships/MembershipRepository.kt`,
  `core/loyalty/LoyaltyRepository.kt`, `core/referral/ReferralRepository.kt`,
  `core/recurring/RecurringBookingRepository.kt`, `core/notifications/*Repository.kt`,
  `core/payments/PaymentRepository.kt`, `core/catalog/CatalogRepository.kt`,
  `core/user/UserRepository.kt`, `core/auth/AuthRepository.kt`, `core/settings/AppSettingsRepository.kt`;
  finally (N) iOS contract note. The PM never spawns two repo children in parallel.
- **No serialization (wire) impact:** the DTOs and the `networkCall`/Retrofit `Response<T>` layer are
  unchanged; `ApiResult` is constructed *from* the existing `networkCall { … }` result (mirroring
  partner-app's `safeApiCall`). No JSON/serialization changes, **no `manual_steps`** (no
  ef-migration, no nswag-regen — this is mobile-only client refactoring).
- **Snackbar move:** the repo stops calling `snackbar.showError(...)`; it returns
  `ApiResult.Error(ApiError(...))` built from `ApiErrorParser.parseToUserMessage(...)`. The VM calls
  `.onError { snackbar.showError(it.message) }` (or equivalent), preserving the **same single
  message**. Background/silent loaders (e.g. `OrderRepository.loadNextPage`,
  `getMyServingCleaners`) keep their silent behavior — map their `Error` to a no-op in the VM so the
  observable behavior is identical.
- **iOS:** no repos exist; the ADR (AC1) fixes the Swift equivalent result type so the incoming iOS
  app starts on the canonical contract. No iOS source is written in this ticket.
