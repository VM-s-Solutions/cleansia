## Context

`src/Cleansia.Core.AppServices/Features/Orders/CreateOrder.cs` is a 484-line god-unit flagged by
**AUD-06**. The `Handler` constructor injects **15 dependencies** (`addressRepository`,
`savedAddressRepository`, `currencyRepository`, `countryRepository`, `serviceCityRepository`,
`stripeClientFactory`, `queueClient`, `promoCodeService`, `referralService`, `referralRepository`,
`userSessionProvider`, `pricingCalculator`, `orderFactory`, `addressGeocoder`, `logger`) and its
`Handle` method braids at least four unrelated concerns inline:

1. **Late referral acceptance** — lines ~240-265 (`referralRepository.GetByReferredUserIdAsync` →
   `referralService.AcceptAsync` with retry/try-catch/log, never blocks the booking).
2. **Address resolution + serviced-area gating** — `ResolveAddressAsync` (saved-vs-inline,
   ownership check, country `IsServicedAsync`, single-serviced-country fallback), plus the inline
   `serviceCityRepository.CityIsServicedAsync` city check and `addressGeocoder.PopulateCoordinatesAsync`.
3. **Promo preview + post-persist apply** — lines ~317-328 (`promoCodeService.PreviewAsync`) and
   ~387-404 (`promoCodeService.ApplyAsync`, failure logs but does not roll back).
4. **Payment-type side effects** — the `switch (command.PaymentType)` (Stripe checkout session with a
   narrow `StripeException` catch → `PaymentGatewayUnavailable`; Cash → `queueClient.SendAsync(QueueNames.GenerateReceipt, ...)`).

This is a **Wave-3 consistency/quality refactor, not a behavior change**: the goal is to lift these
collaborators out of the handler so each is independently testable and the handler reads as
orchestration, while the externally observable behavior (success `Response`, every `BusinessResult.Failure`
error code, ordering of side effects) stays **identical**.

**Sequencing note (TICKET-MAP `CreateOrder.cs` cluster):** F2 (T-0118) → AUD-06 (this) → TC-4. The
queue enqueue site in the Cash branch is the **F2 dispatch site** — this ticket **rebases on the
post-F2 `CreateOrder.cs`** and must not undo T-0118's post-commit-dispatch shape. **TC-4 (the
CreateOrder characterization suite) is written BEFORE this refactor** (Wave 4, no dep on AUD-06) and
is the green regression net this ticket refactors under — do not advance this ticket until TC-4 is
green against the pre-refactor handler.

- **TEST-FIRST (mandatory).** Per `agents/knowledge/testing.md` → "When you're changing existing
  untested code": write/confirm the **characterization test first** (TC-4) pinning current behavior,
  see it green, then refactor under green (red→green→refactor). The status log must show the
  characterization green **before** the decomposition commit; a diff where the refactor lands first
  fails Gate 6.

  `agents/knowledge/consistency.md` section **B** (commands): keep B2 session-derived identity, B5
  `nameof(field)` error construction, **B7** rich domain methods (no direct entity property sets in
  the new collaborators), **B8** narrow provider-specific `try/catch` (the `StripeException` catch and
  the best-effort referral/promo log-on-failure stay exactly as-is), and **B9** `entity.MapToDto()`
  mapping. The extracted services are plain injected collaborators behind interfaces, registered in DI
  alongside the existing Orders services.
- **Rebase target.** Branch from post-T-0118 `master`; the Cash-branch `queueClient.SendAsync(
  QueueNames.GenerateReceipt, new GenerateReceiptMessage(order.Id, command.Language), ...)` is the F2
  dispatch site — keep it where F2 placed it.
- **Serialization.** `CreateOrder.cs` is a serialized shared-file cluster (TICKET-MAP): F2 → AUD-06 →
  TC-4. Do not run this concurrently with any other ticket editing `CreateOrder.cs`. Newly created
  collaborator files are fresh files (no collision), but their DI registration touches the Orders
  feature registration — serialize against any concurrent Orders-feature DI edit.
- **Size note.** Sized **L** at the source. Per the lifecycle a ticket must be split before it goes

  (b) extract promo-application collaborator, (c) extract payment-side-effect dispatcher + late-referral

  of `draft`.
