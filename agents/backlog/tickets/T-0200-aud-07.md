## Context

`src/Cleansia.App/libs/cleansia-customer-features/order-wizard/src/lib/order-wizard/order-wizard.facade.ts`
is the order-wizard god-facade flagged by **AUD-07** (the master ticket-map cites ~1048 lines; the
file measures **977 lines** today — same god-unit, the figure drifted slightly post-Wave-0). One
`OrderWizardFacade` braids at least seven unrelated concerns inline:

1. **Live server-quote engine** — the `toObservable(this.quoteInputs)` →
   `debounceTime → distinctUntilChanged → switchMap(QuoteOrder) → catchError → subscribe` stream
   (~lines 290-346) plus `refreshQuoteNow()` (~line 355), `quote`/`quoting`/`lastQuotedInputs` signals,
   and the express-surcharge `isExpressSlot`/`displayedTotalPrice` computeds.
2. **Promo + referral validation** — `setPromoCode`/`setReferralCode`/`validatePromoCodeNow` (~line 482)
   /`validateReferralCodeNow` (~line 526)/`clearPromoCode`/`clearReferralCode` with their
   `firstValueFrom` calls and `PromoCodeUiState`/`ReferralUiState`.
3. **Service-area (city-serviced) check** — the `cityServiced` signal + `lastCityCheckKey` cache.
4. **Catalog/category filtering** — `services`/`packages`/`extras`/`countries` loading, `categories`
   and `filteredServices` computeds, `setCategory`.
5. **Saved-address management** — `savedAddresses`/`selectedSavedAddressId`, `selectSavedAddress`,
   `isSavedAddressSelected`, `saveCurrentAddressAsSaved`, `updateAddressFromForm`.
6. **Step navigation + rebook prefill** — `activeStep`/`steps`/`stepIcons`, `nextStep`/`prevStep`/
   `goToStep`/`canProceed`, `prefillFromRebook`, `initialize`.
7. **Order submission** — `submitOrder(...)` (~line 909) with the Card vs Cash branches
   (`paymentClient.createOrder` / `orderClient.createOrder`).

On top of the size, the facade has **two consistency-rule violations** in `agents/knowledge/consistency.md`:

- **C1 (cleanup paradigm).** The facade uses `DestroyRef` + `takeUntilDestroyed(this.destroyRef)`
  (imports/field at lines 1, 35, 78; usages at lines 344, 578, 596, 608, 679, 1007, 1029) and bare
  `firstValueFrom(...)` (lines 364, 496, 535) instead of extending `UnsubscribeControlDirective` and
  using `takeUntil(this.destroyed$)`. This is the **exact deviation named in `consistency.md` C1 line
  86** ("✗ `OrderWizardFacade` …"). One cleanup paradigm codebase-wide is the rule.
- **C3 (the canonical async pipe).** Client calls do **not** use the canonical
  `takeUntil(this.destroyed$) → catchError(() => of(null)) → finalize(() => this.loading.set(false))`
  pipe: the quote stream resets state with `.subscribe(() => this.quoting.set(false))` (line 346)
  rather than `finalize`, and the two submit branches (`paymentClient.createOrder` line 1007,
  `orderClient.createOrder` line 1029) subscribe with no `catchError`/`finalize`.

This is a **Wave-3 consistency/quality refactor, not a behavior change.** The goal is to split the
god-facade into focused collaborators (the facade reads as orchestration) and migrate every async call
onto the C3 pipe, while the externally observable behavior — wizard step flow, the live quote/total
shown, promo/referral validation results, the city-serviced banner, saved-address handling, and the
Card/Cash submit outcomes and navigation — stays **identical**.

**Dependency.** `depends_on: [T-0196]` — the C1 `UnsubscribeControlDirective` base sweep. The C1
migration here builds on the canonical base that T-0196 establishes/lands across customer-features; do
not start until T-0196 is `done`.

## Acceptance criteria

- [ ] **AC1 (characterization first / red baseline)** — Given the current `OrderWizardFacade`, When the
  facade Jest spec (per `agents/knowledge/testing.md` "changing existing untested code" + the
  "pragmatic test-alongside … test the facade first" rule for UI) runs against the **unchanged**
  facade, Then it is green and pins the current behavior: the three data states; the debounced live
  quote populating `quote()`/`totalPrice()` and the `isExpressSlot`/`displayedTotalPrice` math;
  `validatePromoCodeNow`/`validateReferralCodeNow` returning the current `PromoCodeUiState`/
  `ReferralUiState`; the `cityServiced` `idle→pending→ok|rejected|error` transitions; `canProceed()`
  per step; `prefillFromRebook` outputs; and `submitOrder` taking the Card branch
  (`paymentClient.createOrder`) vs the Cash branch (`orderClient.createOrder`) with its
  success-navigation and error→`SnackbarService` mapping. **This spec must predate the refactor**
  (visible in commit order / the status log per `testing.md` "characterization test first").

- [ ] **AC2 (split — the smell removed)** — Given the refactor, When the facade is reviewed, Then the
  god-facade is decomposed into focused, separately-providable collaborators (e.g. a quote/pricing
  facade or service, a promo+referral validation service, a city-serviced/service-area service, a
  saved-address service, and the slimmed `OrderWizardFacade` retaining step-navigation + submit
  orchestration). No single new collaborator reconstitutes the god-unit; the orchestrating facade's
  line count and direct-concern count are materially reduced. Each non-trivial extracted unit has its
  own Jest spec.

- [ ] **AC3 (C1 + C3 migration)** — Given the change, When the cleanup/async paradigm is reviewed, Then
  the facade (and every extracted collaborator that makes client calls) **extends
  `UnsubscribeControlDirective`** and uses **`takeUntil(this.destroyed$)`** — no `DestroyRef`,
  `takeUntilDestroyed`, or bare `firstValueFrom` remain (C1, the deviation named at `consistency.md`
  line 86 is gone); and **every** client call uses the exact C3 pipe
  `takeUntil(this.destroyed$) → catchError(() => of(null)) → finalize(() => this.<loadingSignal>.set(false))`
  — the quote stream resets `quoting` via `finalize` (not inline in `subscribe`), and both submit
  branches gain the `catchError`/`finalize` pipe.

- [ ] **AC4 (behavior identical)** — Given the split + migrated facade, When AC1's characterization
  spec re-runs unchanged, Then it is **still green** — same quote/total values, same promo/referral
  states, same city-serviced transitions, same `canProceed` gating, same Card/Cash submit outcomes and
  navigation, same error→snackbar mapping. No generated-client (NSwag) edits, so **no nswag-regen** is
  required.

- [ ] **AC5 (consistency clean for the touched area)** — Given the change, When
  `node agents/tools/check-consistency.mjs frontend --paths=src/Cleansia.App/libs/cleansia-customer-features/order-wizard`
  runs, Then it reports **zero** violations for the order-wizard files — specifically C1 and C3 clear,
  and no new C2 (signal-state) / C4 (`SnackbarService` errors) / C8 (no NgRx mixed into single-feature
  state beyond the legitimate shared `selectCustomerServices`/`selectCustomerPackages`/
  `SavedAddressStore` catalog reads) deviations are introduced by the split.

## Implementation notes

- **TEST-FIRST (mandatory).** Per `agents/knowledge/testing.md` → "When you're changing existing
  untested code" and the UI rule "test the facade first (state transitions, error→snackbar mapping, the
  three data states)": write/confirm the **characterization Jest spec first** (AC1) pinning current
  behavior, see it green, then refactor under green (red→green→refactor). The status log must show the
  characterization green **before** the split commit; a diff where the refactor lands first fails Gate 6.
- **Canonical pattern.** Follow `agents/knowledge/consistency.md` section **C** (frontend list/feature
  facades): **C1** facade extends `UnsubscribeControlDirective`, `@Injectable()`, provided on the
  component — "One cleanup paradigm codebase-wide: `UnsubscribeControlDirective` +
  `takeUntil(this.destroyed$)`" (this ticket removes the C1-line-86 deviation explicitly);
  **C3** every client call uses the exact `takeUntil(this.destroyed$) → catchError(() => of(null)) →
  finalize(() => this.loading.set(false))` pipe (reset state in `finalize`, never inline in
  `catchError`/`subscribe`); **C2** signal state; **C4** errors via `SnackbarService`; **C8** keep NgRx
  for genuinely cross-feature state only (the existing `selectCustomerServices`/
  `selectCustomerPackages`/`SavedAddressStore` reads are legitimate shared catalog/state — do not add
  per-feature state into NgRx during the split). New collaborators that make client calls also follow
  C1/C3/C4.
- **Async-pipe note.** "Migrate to the C3 async pipe" = the canonical **C3 RxJS pipe** above (this is
  not a markup change to Angular's `async` pipe). The `firstValueFrom(...)` calls in
  `validatePromoCodeNow`/`validateReferralCodeNow`/`refreshQuoteNow` are converted to the
  `UnsubscribeControlDirective` + `takeUntil(this.destroyed$)` observable form (or kept as
  `firstValueFrom` only where an imperative await is genuinely required and the C1 cleanup still
  holds — reviewer's call, but C1 forbids it as the default per line 86).
- **Serialization.** The order-wizard facade is a single-feature surface; the only shared dependency is
  the `UnsubscribeControlDirective` base introduced by **T-0196** — which is why this ticket
  `depends_on: [T-0196]` rather than runs concurrently. Do not run this concurrently with any other
  ticket editing files under
  `src/Cleansia.App/libs/cleansia-customer-features/order-wizard/`; newly created collaborator files
  are fresh files (no collision).
- **Size note.** Sized **L** at the source. Per `agents/process/ticket-lifecycle.md` an L ticket must be
  **split before it goes `ready`** — the PM splits this into (a) extract the quote/pricing collaborator
  + C3-migrate its stream, (b) extract the promo+referral validation + city-serviced collaborators +
  C3-migrate (drop `firstValueFrom`), (c) extract saved-address + slim the orchestrating facade
  (step-nav + submit) + C1/C3-migrate the submit branches — each landing under the shared AC1
  characterization net — before transitioning out of `draft`.
