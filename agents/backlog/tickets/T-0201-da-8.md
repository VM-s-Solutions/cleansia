## Context
Audit finding **DA-8**: `AddSavedAddress.Handler.Handle`
(`src/Cleansia.Core.AppServices/Features/SavedAddresses/AddSavedAddress.cs:89-147`) is a ~58-line
god-method that interleaves several distinct responsibilities in one flow:

1. Identity resolution â€” `userSessionProvider.GetUserId()!` (line 91).
2. Default-country fallback â€” resolve missing `CountryId` to ISO `"CZE"`, else first configured
   country, else throw (lines 92-99).
3. Address dedup/reuse â€” `addressRepository.GetAddressAsync(...)` or `Address.Create(...)`
   (lines 101-103).
4. Duplicate-saved-address guard â€” load the user's saved set and fail with
   `BusinessErrorMessage.SavedAddressAlreadyExists` if the address is already saved (lines 105-114).
5. Conditional add of a new `Address` row (lines 116-119).
6. Default-clearing â€” `ClearDefaultForUserAsync` when `SetAsDefault` (lines 121-124).
7. Persist the `SavedAddress` (lines 126-132).
8. **Inline DTO projection** â€” a hand-built `new SavedAddressDto(...)` (lines 135-146).

This is a long, multi-concern handler. The inline projection at step 8 is also a `consistency.md`
**B9** violation and is **triplicated**: the same `new SavedAddressDto(...)` projection appears in
`AddSavedAddress.cs:135`, `UpdateSavedAddress.cs:150`, and `GetSavedAddresses.cs:23`. There is no
`Mappers/` folder and no `SavedAddressMapper` in this feature today, and no SavedAddress test exists
in `Cleansia.Tests`.

This is a **Wave 3 consistency/quality refactor â€” behavior must not change.** The fix is to pin the
current behavior with a characterization test, then decompose: extract the cohesive steps into named
private helpers (and/or a small domain/app service) and replace the inline projection with the
canonical `address`/`saved` â†’ `SavedAddressDto` mapper extension, leaving the handler a readable
orchestration of well-named steps.
