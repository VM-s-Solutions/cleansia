## Context

Wave-3 consistency/quality cleanup. This ticket removes four pieces of dead-or-unsafe code uncovered
by the "blind-spot" slice of the audit (`agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md`

behavior â€” the goal is to remove the smell with **zero behavior change** for any current caller. The
title also folds in the sibling mobile dead-code finding **IA-11** ("dead scrap/auth"), which is the
reason `mobile` is a layer here.

The four (+1) smells, all confirmed against the real code:

1. **BLIND-4 (latent XSS / dead code)** â€” `Cleansia.Infra.Services/Templates/HandlebarsTemplateEngine.cs`
   is registered at `Cleansia.Infra.Services/ServiceCollectionExtensions.cs:13`
   (`AddScoped<ITemplateEngine, HandlebarsTemplateEngine>()`) but has **zero production consumers** â€”
   grep confirms `ITemplateEngine`/`CompileAsync` is referenced only inside its own
   `Templates/*` files; all real email goes through SendGrid **dynamic templates** (`EmailService`),
   so rendering happens in SendGrid's cloud, not here. Its helpers (`formatNumber`, `add`, `multiply`,
   `eq`) call `writer.WriteSafeString(...)` with **no HTML-encoding** on the result, and the engine
   is built with `Handlebars.Create()` defaults â€” a latent stored/reflected-XSS trap the moment
   anyone wires it.
2. **BLIND-9 (FormatException trap)** â€” same file, `HandlebarsTemplateEngine.cs:33,84-85,104-105,117-118`:
   the numeric helpers use `Convert.ToDecimal(parameters[0])` / `Convert.ToInt32(...)` with no
   try-parse and **current-culture** parsing (itself a bug in a 5-language app). A single non-numeric
   value would throw `FormatException` and crash the whole render. Latent (gated by BLIND-4's "unused").
3. **BLIND-10 (divergent dead abstraction)** â€” `Cleansia.Infra.Clients/SendGrid/SendGridClientFactory.cs`
   exposes a `BusinessResult`-returning `SendTemplateEmailAsync(...)` and an `EmailNotSentError =
   "email.sending_failed"` constant that have **no production consumer**. The real send path
   (`Cleansia.Core.AppServices/Services/EmailService.cs:348,390`) ignores the factory's send method,
   does `new SendGridClient(...)` and throws `EmailDeliveryException` â€” a second, divergent contract
   for the same operation. The only live use of the factory is `HealthController.cs`, which calls
   `CreateClient()` only. Two contracts â†’ the unused one rots; violates `conventions.md` "one way to
   do each thing".
4. **BLIND-11 (log-spam)** â€” `Cleansia.Infra.Clients/Fcm/FcmPushDispatcher.cs:48-53`: when
   `ServiceAccountJson` is empty the dispatcher correctly no-ops, but logs a `Warning` (with token
   count + event key) on **every** dispatch. In an environment with push intentionally disabled this
   buries real signals. (The terminal config-missing branch at `:152` already latches `_initAttempted`
   correctly â€” keep that.)
5. **IA-11 (dead mobile code, the "scrap/auth")** â€” `src/cleansia_android/scrap/partner-app-pre-rebuild/`
   ships **13 `.kt` files** under `kotlin/auth/{viewmodels,screens}/` that declare the **same package**
   as the live partner-app auth code (`cz.cleansia.partner.features.auth.*`). They are an unreferenced
   pre-rebuild backup that risks duplicate-class confusion / accidental edits to the wrong copy.
