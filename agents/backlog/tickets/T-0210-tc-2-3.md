## Context
This is **TC-2/3** from the Wave-4 test backlog (`agents/backlog/TICKET-MAP.md` Wave 4 ‚Äî "TC-2/3
(webhook integration incl. signature-stays-on regression)"). It is the **end-to-end** counterpart to
the Wave-0 unit/handler idempotency suite **T-0127 (TC-IDEMP-0)**: where T-0127 proves the dispatch
behavior and the receipt consumer are safe to run twice in isolation, this ticket drives the **real
Stripe webhook routes through the host** (`Cleansia.IntegrationTests`, WebApplicationFactory) and
asserts the observable money/membership outcome of a re-delivered event.

Two reasons it exists:

1. **`agents/knowledge/testing.md` must-cover #6 (Idempotency / S7)** ‚Äî "Simulate the webhook
   re-delivery." The catalog requires the side-effecting Stripe path to be proven safe to run twice at
   the layer the bug actually lives ‚Äî the controller ‚Üí MediatR ‚Üí validator ‚Üí handler ‚Üí DB ‚Üí post-commit
   dispatch chain ‚Äî not only at the mocked-handler layer. (`testing.md:96`, must-cover #6.)
2. **The signature-stays-on regression.** The audit's auth-takeover cluster removed the
   `IsDevelopment()` verification bypass from server-side token paths (IDA-SEC-01, T-0105). The Stripe
   webhooks have the **same** failure mode: a webhook that skips `EventUtility.ConstructEvent`
   signature verification (or gates it behind an env/config flag) lets an attacker forge
   `checkout.session.completed` / `customer.subscription.created` events and mint paid orders and free
   memberships. This ticket installs the **regression lock**: an integration test that a request with a
   missing/invalid `Stripe-Signature` is rejected (no event is processed, no side effect fires) in
   **every** environment, so a future "make local testing easier" change cannot silently turn signature
   verification off.

The fixes this suite exercises already landed in Wave-0 ‚Äî that is why this ticket **depends on**
**T-0114 (SEC-W2: 2nd-active-membership active-check + filtered unique index)** and **T-0118
(F2/SEC-W1: post-commit dispatch + idempotent `generate-receipt` consumer)**. ADR-0002
(`agents/backlog/adr/0002-outbox-dispatch-contract.md`) governs the dispatch/idempotency contract that
these integration tests assert end-to-end (post-commit dispatch fires once per committed success;
re-delivery does not double the effect; `ProcessedStripeEvent` dedups the order webhook; the active
membership check + filtered unique index dedup the subscription webhook).

## Acceptance criteria
Each case is an **integration** test in `Cleansia.IntegrationTests` driving the real webhook route
through the host, authenticated/posted as Stripe would. Assertions are on **observable** state (HTTP
status, persisted rows, effect-count via the seam the dispatch behavior calls), never "a method
exists". Builders come from `Cleansia.TestUtilities`.

- [ ] **AC1 ‚Äî Order webhook re-delivery is idempotent (S7, exercises T-0118 + `ProcessedStripeEvent`)** ‚Äî
  Given a valid `checkout.session.completed` event for an order in `Pending`, When the event is POSTed
  to the order webhook route **twice** (same Stripe event id, valid signature ‚Äî the at-least-once
  re-delivery case), Then the order ends `Confirmed`/`Paid` exactly once, **exactly one**
  `ProcessedStripeEvent` row exists, and the post-commit dispatch realizes the `generate-receipt`
  enqueue and the `OrderConfirmed` push **exactly once each** (asserted at the `IQueueClient.SendAsync`
  seam). The second POST returns success (the already-processed short-circuit, ADR-0002 D1.2) and
  dispatches nothing.

- [ ] **AC2 ‚Äî Subscription webhook re-delivery does not create a 2nd active membership (S7, exercises
  T-0114)** ‚Äî Given a user with an existing `Active` `UserMembership`, When a
  `customer.subscription.created` event for that user (new `subscriptionId`, valid signature) is POSTed
  to the subscription webhook route, Then `ProvisionFromCreatedEventAsync` finds the active row via
  `GetActiveForUserAsync` and **does not** insert a second active row ‚Äî the route returns success and
  the membership count for `(TenantId, UserId)` stays exactly one. Given a **clean** user, the same
  event creates **exactly one** `UserMembership` (happy path unchanged).

- [ ] **AC3 ‚Äî Filtered-unique-index backstop holds end-to-end (exercises T-0114 AC3)** ‚Äî Given the
  active-check is bypassed by a race (two near-simultaneous `customer.subscription.created` deliveries
  for the same clean user), When both are processed, Then **exactly one** `Active` `UserMembership`
  lands and the loser surfaces the PG 23505 filtered-unique-index violation (not a second row); a
  `Cancelled`/expired row plus a new `Active` row for the same user is still permitted (the index is
  *filtered* on `Status = Active`).

- [ ] **AC4 ‚Äî Signature-stays-on regression: missing signature rejected (the regression lock)** ‚Äî
  Given a well-formed `checkout.session.completed` body but **no** `Stripe-Signature` header, When it is
  POSTed to the order webhook route, Then the route rejects it (4xx, no event constructed), **no**
  order state changes, **no** `ProcessedStripeEvent` row is written, and **nothing** is dispatched
  (zero `IQueueClient.SendAsync`). Same assertion for the subscription webhook route.

- [ ] **AC5 ‚Äî Signature-stays-on regression: invalid/forged signature rejected** ‚Äî Given a forged body
  signed with the **wrong** webhook secret (an attacker-minted event), When POSTed to each webhook
  route, Then `EventUtility.ConstructEvent` throws / the route returns 4xx, **no** side effect fires
  (no membership, no order confirmation, no enqueue), in the **test host's configured environment**.
  This is the test that fails if signature verification is ever short-circuited.

- [ ] **AC6 ‚Äî Signature verification is not environment-gated (the structural lock)** ‚Äî A test (or a
  mechanical assertion) proves the webhook signature check has **no `IsDevelopment()` / config-flag
  bypass**: with the host configured as Development, an unsigned/forged event is **still** rejected
  exactly as in AC4/AC5. (Mirrors the IDA-SEC-01 / T-0105 "remove IsDevelopment bypass" lock for the
  webhook surface.)

- [ ] **AC7 ‚Äî Valid happy path still passes (no false lock)** ‚Äî Given a correctly Stripe-signed,
  first-delivery `checkout.session.completed` and `customer.subscription.created`, When POSTed, Then
  each is processed exactly once with its full effect ‚Äî proving AC4‚ÄìAC6 reject only bad signatures, not
  legitimate Stripe traffic.

- [ ] **AC8 ‚Äî Built test-first, no theater** ‚Äî Per `agents/knowledge/testing.md`, each case was
  committed red-first against the Wave-0 fixes it exercises (visible in commit order / this ticket's
  status log), each AC maps to a named integration test, and every assertion checks an observable
  effect/effect-count ‚Äî not method presence or non-null (testing.md anti-patterns). Money and
  signature-rejection sad paths are covered, not only happy paths.

## Implementation notes
- **Built TEST-FIRST per `agents/knowledge/testing.md`.** Webhook idempotency ("safe to run twice",
  must-cover #6, `testing.md:96`) and the signature-rejection sad paths are the strict category ‚Äî write
  each case **failing-first** and confirm it fails *for the right reason* (re-delivery doubles the
  effect / an unsigned event is processed) before relying on the Wave-0 fix to make it green. The status
  log records the red‚Üígreen note per case. Implementation-first with tests bolted on fails Gate 6.
- **Governing ADR: ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`). These tests are
  the **integration-layer** proof of the same guarantees the ADR's verification ¬ß#5/#7 prove at the
  unit layer: post-commit dispatch fires **once per committed success** and never on a non-success
  (D1/D1.2); the deterministic `MessageKey` (D2.1) collapses a duplicate-enqueue and a redelivery onto
  one effect; the `ProcessedStripeEvent` stamp dedups the order webhook (Context ¬ß1a parallel-retry
  trace). Honor the ADR's honest semantics ‚Äî the receipt **email** is the terminal effect that must be
  exactly-once; do **not** assert a mythical exactly-once on the push (AC1 asserts the *enqueue* count,
  the push-consumer's own dedup is F4/F3, out of scope). ADR-0002 is `accepted` and **immutable**.
- **Test project:** `Cleansia.IntegrationTests` (WebApplicationFactory-style host, per `testing.md`
  layer table ‚Äî "A route end-to-end ‚Üí Integration ‚Üí incl. auth"). Drive the actual webhook controllers;
  do not call the handlers directly (that is T-0127's layer). Use `Cleansia.TestUtilities` builders for
  the order/membership/event fixtures; sign test bodies with a known test webhook secret so AC4/AC5 can
  flip the secret/header to forge.
- **Cited code under test (verified via the paired tickets / ADR-0002 against current source ‚Äî this is
  a test ticket so it adds files, edits none):**
  - Order webhook: `HandlePaymentNotification.cs:144-159` (already-processed short-circuit +
    state change), `:241-257,278` (the `generate-receipt` + `OrderConfirmed` push enqueue, post-T-0118
    behind `IPendingDispatch`), and the `ProcessedStripeEvent` dedup
    (`ProcessedStripeEventEntityConfiguration.cs:37-38`).
  - Subscription webhook: `StripeSubscriptionWebhookHandler.ProvisionFromCreatedEventAsync`
    (`:102-167`, the active-check inserted at T-0114 before `UserMembership.Create` at `:154`) and the
    filtered unique index on `(TenantId, UserId) WHERE Status = Active`
    (`UserMembershipEntityConfiguration.cs:59-61`, added by T-0114).
  - Signature surface: the `EventUtility.ConstructEvent(...)` call in each webhook controller (the
    AC4/AC5/AC6 lock) ‚Äî assert it has **no** `IsDevelopment()`/config bypass, mirroring T-0105
    (IDA-SEC-01) for tokens.
- **Serialization cluster:** this ticket adds **only test files** under `Cleansia.IntegrationTests` /
  `Cleansia.TestUtilities`, which are in **no** shared-file cluster in `agents/backlog/TICKET-MAP.md`
  (the clusters there are production files ‚Äî `UnitOfWorkPipelineBehavior.cs` + queue call sites,
  `PolicyBuilder.cs`, `CleansiaStartupBase.cs`, etc.). So **no serialization constraint** with other
  tickets on the test files themselves. The ordering constraint is purely the **dependency edge**: it
  cannot complete before its Wave-0 fixes ‚Äî `T-0114` and `T-0118` (and, transitively, the F2/F11
  cluster `F11 ‚Üí F2/SEC-W1 ‚Üí F4 ‚Üí F3` that T-0118 sits in) must be `done` first.
- **Dependencies:** **T-0114 (SEC-W2)** ‚Äî the subscription active-check + filtered unique index must
  exist for AC2/AC3 to pass. **T-0118 (F2/SEC-W1)** ‚Äî the post-commit dispatch behavior +
  `QueueEnvelope`/key + idempotent receipt consumer must exist for AC1's effect-count assertions. Hold
  this ticket until both are `done`. (The order webhook consumer reachability rides on the same
  FUNC-CORE/T-0121 extraction that T-0118/T-0127 rely on for the Functions side.)
- **Routing** (`agents/process/routing.md`): qa/backend authors the integration cases; spawn a
  **reviewer** in parallel (Gate 6 enforces test-first + no-theater + the signature-lock per
  `testing.md`). `security_touching: false` ‚Äî the tests add no auth/ownership *surface*; the webhook
  fixes carry their own Security gate in T-0114/T-0118. (The signature-stays-on assertion is a
  security-relevant *regression lock* but it tests existing behavior, it does not change a policy.)
- `manual_steps: []` ‚Äî no migration/NSwag from this ticket; the SEC-W2 migration is owned by T-0114.
