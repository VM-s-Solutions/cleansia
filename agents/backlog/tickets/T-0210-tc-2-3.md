## Context
This is **TC-2/3** from the Wave-4 test backlog (`agents/backlog/TICKET-MAP.md` Wave 4 — "TC-2/3
(webhook integration incl. signature-stays-on regression)"). It is the **end-to-end** counterpart to
the Wave-0 unit/handler idempotency suite **T-0127 (TC-IDEMP-0)**: where T-0127 proves the dispatch
behavior and the receipt consumer are safe to run twice in isolation, this ticket drives the **real
Stripe webhook routes through the host** (`Cleansia.IntegrationTests`, WebApplicationFactory) and
asserts the observable money/membership outcome of a re-delivered event.

Two reasons it exists:

1. **`agents/knowledge/testing.md` must-cover #6 (Idempotency / S7)** — "Simulate the webhook
   re-delivery." The catalog requires the side-effecting Stripe path to be proven safe to run twice at
   the layer the bug actually lives — the controller → MediatR → validator → handler → DB → post-commit
   dispatch chain — not only at the mocked-handler layer. (`testing.md:96`, must-cover #6.)
2. **The signature-stays-on regression.** The audit's auth-takeover cluster removed the
   `IsDevelopment()` verification bypass from server-side token paths (IDA-SEC-01, T-0105). The Stripe
   webhooks have the **same** failure mode: a webhook that skips `EventUtility.ConstructEvent`
   signature verification (or gates it behind an env/config flag) lets an attacker forge
   `checkout.session.completed` / `customer.subscription.created` events and mint paid orders and free
   memberships. This ticket installs the **regression lock**: an integration test that a request with a
   missing/invalid `Stripe-Signature` is rejected (no event is processed, no side effect fires) in
   **every** environment, so a future "make local testing easier" change cannot silently turn signature
   verification off.

The fixes this suite exercises already landed in Wave-0 — that is why this ticket **depends on**
**T-0114 (SEC-W2: 2nd-active-membership active-check + filtered unique index)** and **T-0118
(F2/SEC-W1: post-commit dispatch + idempotent `generate-receipt` consumer)**. ADR-0002
(`agents/backlog/adr/0002-outbox-dispatch-contract.md`) governs the dispatch/idempotency contract that
these integration tests assert end-to-end (post-commit dispatch fires once per committed success;
re-delivery does not double the effect; `ProcessedStripeEvent` dedups the order webhook; the active
membership check + filtered unique index dedup the subscription webhook).

## Acceptance criteria
Each case is an **integration** test in `Cleansia.IntegrationTests` driving the real webhook route
through the host, authenticated/posted as Stripe would. Assertions are on **observable** state (HTTP
status, persisted rows, effect-count via the seam the dispatch behavior calls), never "a method
exists". Builders come from `Cleansia.TestUtilities`.

- [ ] **AC1 — Order webhook re-delivery is idempotent (S7, exercises T-0118 + `ProcessedStripeEvent`)** —
  Given a valid `checkout.session.completed` event for an order in `Pending`, When the event is POSTed
  to the order webhook route **twice** (same Stripe event id, valid signature — the at-least-once
  re-delivery case), Then the order ends `Confirmed`/`Paid` exactly once, **exactly one**
  `ProcessedStripeEvent` row exists, and the post-commit dispatch realizes the `generate-receipt`
  enqueue and the `OrderConfirmed` push **exactly once each** (asserted at the `IQueueClient.SendAsync`
  seam). The second POST returns success (the already-processed short-circuit, ADR-0002 D1.2) and
  dispatches nothing.

- [ ] **AC2 — Subscription webhook re-delivery does not create a 2nd active membership (S7, exercises
  T-0114)** — Given a user with an existing `Active` `UserMembership`, When a
  `customer.subscription.created` event for that user (new `subscriptionId`, valid signature) is POSTed
  to the subscription webhook route, Then `ProvisionFromCreatedEventAsync` finds the active row via
  `GetActiveForUserAsync` and **does not** insert a second active row — the route returns success and
  the membership count for `(TenantId, UserId)` stays exactly one. Given a **clean** user, the same
  event creates **exactly one** `UserMembership` (happy path unchanged).

- [ ] **AC3 — Filtered-unique-index backstop holds end-to-end (exercises T-0114 AC3)** — Given the
  active-check is bypassed by a race (two near-simultaneous `customer.subscription.created` deliveries
  for the same clean user), When both are processed, Then **exactly one** `Active` `UserMembership`
  lands and the loser surfaces the PG 23505 filtered-unique-index violation (not a second row); a
  `Cancelled`/expired row plus a new `Active` row for the same user is still permitted (the index is
  *filtered* on `Status = Active`).

- [ ] **AC4 — Signature-stays-on regression: missing signature rejected (the regression lock)** —
  Given a well-formed `checkout.session.completed` body but **no** `Stripe-Signature` header, When it is
  POSTed to the order webhook route, Then the route rejects it (4xx, no event constructed), **no**
  order state changes, **no** `ProcessedStripeEvent` row is written, and **nothing** is dispatched
  (zero `IQueueClient.SendAsync`). Same assertion for the subscription webhook route.

- [ ] **AC5 — Signature-stays-on regression: invalid/forged signature rejected** — Given a forged body
  signed with the **wrong** webhook secret (an attacker-minted event), When POSTed to each webhook
  route, Then `EventUtility.ConstructEvent` throws / the route returns 4xx, **no** side effect fires
  (no membership, no order confirmation, no enqueue), in the **test host's configured environment**.
  This is the test that fails if signature verification is ever short-circuited.

- [ ] **AC6 — Signature verification is not environment-gated (the structural lock)** — A test (or a
  mechanical assertion) proves the webhook signature check has **no `IsDevelopment()` / config-flag
  bypass**: with the host configured as Development, an unsigned/forged event is **still** rejected
  exactly as in AC4/AC5. (Mirrors the IDA-SEC-01 / T-0105 "remove IsDevelopment bypass" lock for the
  webhook surface.)

- [ ] **AC7 — Valid happy path still passes (no false lock)** — Given a correctly Stripe-signed,
  first-delivery `checkout.session.completed` and `customer.subscription.created`, When POSTed, Then
  each is processed exactly once with its full effect — proving AC4–AC6 reject only bad signatures, not
  legitimate Stripe traffic.

- [ ] **AC8 — Built test-first, no theater** — Per `agents/knowledge/testing.md`, each case was
  committed red-first against the Wave-0 fixes it exercises (visible in commit order / this ticket's
  status log), each AC maps to a named integration test, and every assertion checks an observable
  effect/effect-count — not method presence or non-null (testing.md anti-patterns). Money and
  signature-rejection sad paths are covered, not only happy paths.

### Security — 2026-06-13 (PASS-WITH-NOTES)

Scope: the signature-stays-on lock (AC4-AC6) and the membership isolation boundary (AC2/AC3). Verified
the tests drive the REAL signature-verification + isolation code paths, not a mock that skips them.

Verified GOOD (genuine boundary coverage):
- Real pipeline, real verification. `AddCoreBindings` wires PostCommitDispatch -> Validation -> UnitOfWork
  -> Handler (`FluentValidationExtensions.cs:21-23`) and the real `OutboxPendingDispatch`
  (`RepositoryExtensions.cs:27`). The tests `IMediator.Send(HandlePaymentNotification.Command)`, so the
  validator's `NotEmpty` signature rule and the handler's `EventUtility.ConstructEvent`
  (`HandlePaymentNotification.cs:125-135`) execute for real. Configured secret matches
  (`appsettings.IntegrationTests.json` `whsec_your_webhook_secret` == `ConfiguredWebhookSecret`).
- Forged-signature tests are STRONG. `ForgedSignature_WrongSecret_*` on BOTH routes sign a real
  `t=,v1=HMAC` over `WrongWebhookSecret`, so the body reaches `ConstructEvent` and is rejected there. If
  verification were broken (any-signature-accepted), the order would go Paid + stamp + outbox rows would
  land and `AssertNoEffectAsync` / `Assert.Empty(memberships)` would fail. These would catch a broken
  signature check.
- AC3 concurrency test seeds a NON-NULL distinct tenant (`tenant-sub-webhook`). The filtered unique index
  is `(TenantId, UserId) WHERE Status=1` and Postgres treats NULL tenants as DISTINCT
  (`UserMembershipEntityConfiguration.cs:80-88`), so a null-tenant seed would hit the single-tenant escape
  branch and prove nothing. The non-null tenant means the 23505 backstop actually fires —
  `TwoConcurrentSubscriptionCreated_...` would catch a removed/unfiltered index. `transactional:false` is
  correctly set so the two scopes genuinely race.
- AC2 active-row guard is tenant-scoped: `GetActiveForUserAsync` uses `GetDbSet()` (no IgnoreQueryFilters)
  under the user-tenant override (`StripeSubscriptionWebhookHandler.cs:141-143,165`), so the count assert
  is in-scope.

NOTES (weaknesses — not blockers; suite still verifies the security guarantee, but two cases are weaker
than the AC claims):

1. AC6 BEHAVIORAL test does not exercise the bypass it claims to lock. `InDevelopmentEnvironment_
   UnsignedEvent_IsStillRejected` sends an EMPTY signature, which the validator's
   `RuleFor(SignatureHeader).NotEmpty()` rejects BEFORE any `ConstructEvent` call — environment-independent
   and unrelated to a verification bypass. A Development bypass that skips `ConstructEvent` for a FORGED
   (non-empty) signature would still pass this test. The real attack always carries some signature header.
   The structural `SignaturePath_HasNoEnvironmentOrConfigFlagBypass` source-grep is the actual AC6 lock and
   it is sound (covers the sole `ConstructEvent` surface, no env-gating in StripeConfig/AddStripe). RECOMMEND
   (test-only follow-up, no prod change): add `InDevelopmentEnvironment_ForgedEvent_IsStillRejected` using
   `Sign(body, WrongWebhookSecret)` so the behavioral Development assertion actually drives `ConstructEvent`.

2. AC2 `AlreadyActiveUser_..._DoesNotCreateSecondActiveMembership` cannot distinguish the app-level
   active-check from the DB index backstop. Because the seed tenant is non-null, removing the
   `GetActiveForUserAsync` check entirely would still land exactly one active row (the filtered index throws
   23505, the handler catch reconciles to the winner) — so this test stays green even with the T-0114
   active-check deleted. The security invariant (no 2nd active row) holds either way and AC3 proves the
   index, so this is a coverage-attribution gap, not a leak. RECOMMEND: a unit-level assertion (or asserting
   the second delivery did NOT throw/short-circuit via the check) to pin the app-check distinctly; out of
   scope to fix here.

No production code edits in this lane; no fixes attempted (test-only ticket). The pre-existing
