## Context
`knowledge/testing.md` #3 (Pricing & money) names refund-on-cancellation math — "the fee-rate ×
refund-amount math" — and dispute refunds as **must-cover before PROD**, with the explicit bar "no
floating-point surprises". This logic currently has **no tests** (coverage is "minimal" per the
audit), yet it moves real money.

The math lives in two real paths today:
- **Cancellation refund** — `BookingPolicy.CalculateCancellationFeeRate(...)` returns the tiered fee
  rate (0.0 / 0.25 / 0.50, plus acceptance-aware and oops-window 0.0 short-circuits), and
  `CancelOrder.Handler` computes `refundAmount = order.TotalPrice * (1m - feeRate)` then persists it
  via `Order.Cancel(...)`.
- **Dispute refund** — `ResolveDispute.Validator` rejects a negative `RefundAmount`
  (`BusinessErrorMessage.InvalidRefundAmount`), and `Dispute.Resolve(...)` records the amount.

This is a **test-only** ticket (TC-7, Wave 4, TDD landing with the money paths). It depends on
T-0172 and T-0140 (the refund/dispute money-path fixes it pins) so the tests assert post-fix behavior,
not behavior that is about to change.

## Acceptance criteria
- [ ] **AC1 (fee-rate tiers)** — Given an **accepted** order, When `CalculateCancellationFeeRate`
  is called with cancel times ≥ free window, in the partial band, and in the last-minute band, Then
  it returns `0m`, `0.25m` (`PartialCancellationFeeRate`), and `0.50m`
  (`LastMinuteCancellationFeeRate`) respectively. Boundary cases (exactly at `FreeCancellationHours`
  and exactly at `PartialCancellationHours`) are asserted, not interpolated.
- [ ] **AC2 (free short-circuits)** — Given a **not-yet-accepted** order (no `Confirmed` history),
  Then the rate is `0m` regardless of timing; and Given an accepted order cancelled within the oops
  window (`OopsWindowMinutesStandard` / `OopsWindowMinutesFirstTime` for first-timers), Then the rate
  is `0m`.
- [ ] **AC3 (Plus free-window override)** — Given a `freeCancellationHoursOverride` (Plus member), When
  cancelling inside the widened free window but inside the default one, Then the rate is `0m`; the
  partial/last-minute thresholds and rates are unchanged by the override.
- [ ] **AC4 (refund-amount math)** — Given a `TotalPrice` and a resolved `feeRate`, Then
  `refundAmount == TotalPrice * (1m - feeRate)` is exact for representative CZK amounts; full-fee
  (`feeRate == 1m`) yields `0m` refund, free (`0m`) yields the full `TotalPrice`. Asserted in `decimal`
  with no float coercion (no floating-point drift).
- [ ] **AC5 (CancelOrder handler wiring)** — Given a paid card order, When `CancelOrder.Handler.Handle`
  runs (mocked repos/Stripe/loyalty/queue), Then the `Response.FeeRate`/`RefundAmount`/`TotalPrice`
  match the formula and `Order.Cancel` is invoked with the same `feeRate`/`refundAmount`; the refund
  branch only fires when `PaymentType==Card && PaymentStatus==Paid && refundAmount>0 && StripeSessionId`
  is set.
- [ ] **AC6 (illegal-state rejection)** — Given an order whose latest status is `Cancelled` /
  `Completed` / `InProgress`, Then `CancelOrder.Handler` returns the matching
  `BusinessErrorMessage` (`OrderAlreadyCancelled` / `OrderAlreadyCompleted` /
  `OrderInProgressCannotCancel`) and computes **no** refund.
- [ ] **AC7 (dispute refund validation)** — Given `ResolveDispute.Command` with a negative
  `RefundAmount`, Then the validator fails with `BusinessErrorMessage.InvalidRefundAmount`; Given a
  null or `≥ 0` amount, Then it passes and `Dispute.Resolve` records the amount and sets
  `DisputeStatus.Resolved`.
- [ ] **AC8 (evidence + TDD order)** — All cases live in `Cleansia.Tests`, use the
  `Cleansia.TestUtilities` builders (no hand-rolled entity graphs), and the status log records the
  red→green sequence per `knowledge/testing.md` (test predates the assertion of fixed behavior).

## Out of scope

- Loyalty revoke-on-cancel correctness, admin refund UI, and chargeback linkage (D-06).
