Source: **TC-4** (Wave 4) — `agents/backlog/TICKET-MAP.md` Wave 4 line + the `CreateOrder.cs`
serialization cluster row. `AUD-06` will decompose the `CreateOrder.Handler` god-unit in Wave 3
(`CreateOrder.cs` cluster: `F2 → AUD-06 → TC-4`). Per `agents/knowledge/testing.md` "When you're
changing existing untested code", we write a **characterization test first** that pins the *current*
observable behavior of the handler/validator so the decomposition can be verified as behavior-

These tests are deliberately written **before** AUD-06 and have **no dependency on it** — they

## Acceptance criteria
- [ ] **AC1 — Validator: happy path passes.** Given a valid `Command` (non-empty name/email/phone,
  future `CleaningDate` above lead time, positive `TotalPrice`, exactly one of `CustomerAddress` /
  `SavedAddressId`, existing service+package ids, price matches the calculator), When validated, Then
  it passes with no errors. (`CreateOrder.cs:45-148`)
- [ ] **AC2 — Address XOR rule.** Given both `CustomerAddress` and `SavedAddressId` set, OR neither
  set, When validated, Then the failure carries `BusinessErrorMessage.OrderAddressExactlyOneRequired`
  on `CustomerAddress`. (`CreateOrder.cs:121-124`)
- [ ] **AC3 — Empty order rejected.** Given empty `SelectedServiceIds` AND `SelectedPackageIds`,
  When validated, Then it fails with `BusinessErrorMessage.EmptyOrder` (cascade-stop before the price
  check). (`CreateOrder.cs:134-139,157-158`)
- [ ] **AC4 — Price mismatch rejected.** Given a `TotalPrice` that differs from
  `IOrderPricingCalculator.CalculateAsync(...).TotalPrice` (with `CleaningDate` passed so the express
  surcharge is folded in), When validated, Then it fails with
  `BusinessErrorMessage.TotalPriceNotMatch`. (`CreateOrder.cs:138-139,160-177`)
- [ ] **AC5 — Cleaning-date guards.** Given a past `CleaningDate`, Then
  `BusinessErrorMessage.CleaningDateInFuture`; given a future date below the booking lead time, Then
  `BusinessErrorMessage.CleaningDateBelowLeadTime`. (`CreateOrder.cs:103-108`)
- [ ] **AC6 — Preferred-employee eligibility.** Given a `PreferredEmployeeId` set AND a logged-in user
  who has **no** completed order with that employee
  (`IOrderRepository.UserHasCompletedOrderWithEmployeeAsync` → false), When validated, Then it fails
  with `BusinessErrorMessage.PreferredEmployeeNotEligible`; when the rule's `When` guard is not met
  (no user id), Then the rule does not fire. (`CreateOrder.cs:141-155`)
- [ ] **AC7 — Handler: saved-address ownership.** Given a `SavedAddressId` whose row's `UserId` differs
  from the session user, When handled, Then the result is `Failure` with
  `BusinessErrorMessage.NotFound` (cross-user leak is masked as NotFound, not surfaced).
  (`CreateOrder.cs:415-427`)
- [ ] **AC8 — Handler: city/country serviced gates.** Given a resolved address in a city that is not
  serviced (`IServiceCityRepository.CityIsServicedAsync` → false), Then `Failure` with
  `BusinessErrorMessage.CityNotServiced` on `City`; and given a saved address whose country is no
  longer serviced, Then `Failure` with `BusinessErrorMessage.CountryNotServiced`.
  (`CreateOrder.cs:280-285,437-441`)
- [ ] **AC9 — Handler: Cash path enqueues GenerateReceipt (the F2 site).** Given `PaymentType.Cash`
  and a successful create, When handled, Then exactly one `GenerateReceiptMessage(order.Id, language)`
  is sent to `QueueNames.GenerateReceipt` via `IQueueClient.SendAsync`, and the response carries a
  null `StripeSessionId`. (`CreateOrder.cs:375-378,406-409`)
- [ ] **AC10 — Handler: Card path + Stripe failure mapping.** Given `PaymentType.Card`, When the
  checkout session is created, Then `Response.StripeSessionId` is populated and **no** receipt message
  is enqueued; and given the Stripe client throws `StripeException`, Then `Failure` with
  `BusinessErrorMessage.PaymentGatewayUnavailable` (non-`StripeException` is **not** caught — it bubbles
  as a 500, characterize that the catch is narrow). (`CreateOrder.cs:354-373`)
- [ ] **AC11 — Tests predate AUD-06 and run green.** All cases live in `Cleansia.Tests`, use the
  `Cleansia.TestUtilities` builders/mocks, run green against the **current** handler, and the status
  log records "red → green" per `agents/knowledge/testing.md`. No production code in `CreateOrder.cs`
  is modified by this ticket.

## Implementation notes
- **Built TEST-FIRST** per `agents/knowledge/testing.md` ("changing existing untested code" →
  characterization test first). Confirm each case fails for the *right* reason if the behavior were
  absent, then pin the current behavior green. Map each AC item to a named test case.
- **Unit under test:** `src/Cleansia.Core.AppServices/Features/Orders/CreateOrder.cs`. Validator deps
  to mock: `IPackageRepository`, `IServiceRepository`, `ICurrencyRepository`,
  `IOrderPricingCalculator`, `IOrderRepository`, `IUserSessionProvider` (lines 33-39). Handler deps to
  mock: the ctor list at lines 216-231 (notably `ISavedAddressRepository`, `ICountryRepository`,
  `IServiceCityRepository`, `IStripeClientFactory`, `IQueueClient`, `IOrderFactory`,
  `IUserSessionProvider`, `IOrderPricingCalculator`).
- **Test location:** `Cleansia.Tests` (xUnit). Use `Cleansia.TestUtilities` builders for the
  `Command`/entity graphs — do not hand-roll graphs inline (testing.md anti-pattern). Assert against
  the `BusinessErrorMessage` **constants**, never hardcoded strings.
- **Serialization cluster:** `CreateOrder.cs` cluster — strict order `F2 (Wave 0) → AUD-06 (Wave 3) →
  TC-4 (Wave 4)`. This is **cross-wave**; ordering handles the collision. TC-4 has **no `depends_on`**
  by design — it characterizes the post-F2 handler and AUD-06 rebases onto these tests. The test file
  is new and does not edit `CreateOrder.cs`, so it does not contend for the shared production file.
- **ADR in force:** **ADR-0002** (outbox-dispatch-contract) governs the Cash-path enqueue at
  `CreateOrder.cs:376-377` (the F2 dispatch site, `QueueNames.GenerateReceipt`). AC9 characterizes that
  the enqueue happens; it does not re-test idempotency (that is TC-IDEMP-0).
- Spawn a `reviewer` in parallel with the `backend` developer (same ticket) per the
  reviewer-per-developer invariant; Reviewer enforces Gate 6 / testing.md (no theater, sad paths
  covered, error-code constants asserted, test-first visible in commit/status order).
