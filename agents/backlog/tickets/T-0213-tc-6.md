## Context
Source: `agents/knowledge/testing.md` must-cover item **#7 ‚Äî "Pay periods & invoices"** (TC-6 in the
Wave-4 test plan, `audits/AUDIT-2026-06-01-execution-plan.md`). Payroll invoice generation,
numbering, and the pay-period open/close lifecycle are money paths with **no coverage today**
(testing.md: "Coverage today is minimal"). A defect here pays an employee twice, closes a period over
unpaid work, or collides invoice numbers ‚Äî all costly and hard to detect after the fact.

This ticket adds the executable evidence for that surface. The behavior under test already exists:
- `GenerateInvoice` command (handler + validator) ‚Äî `Cleansia.Core.AppServices/Features/EmployeePayroll/GenerateInvoice.cs`
- `ClosePayPeriod` command (handler + validator) ‚Äî `Cleansia.Core.AppServices/Features/PayPeriods/ClosePayPeriod.cs`
- `EmployeeInvoice` domain entity (Create/numbering/status transitions) ‚Äî `Cleansia.Core.Domain/EmployeePayroll/EmployeeInvoice.cs`
- `PayPeriod` domain entity (Create/Close/MarkAsPaid/Reopen) ‚Äî `Cleansia.Core.Domain/EmployeePayroll/PayPeriod.cs`

Depends on **T-0171** and **T-0180** (the Wave-4 invoice/pay-period fix + characterization
predecessors) so the cases pin the corrected behavior, not the pre-fix behavior. Per
testing.md, all of this is built **test-first** (these are pure logic + handler-contract cases ‚Äî the
strict/contract TDD tiers).

## Acceptance criteria
Each AC is a test case (or small group) in `Cleansia.Tests`; assert on the `BusinessErrorMessage`
**constant**, never a hardcoded string; use `Cleansia.TestUtilities` builders, not inline graphs.

**Invoice generation (`GenerateInvoice.Handler`, mocked repos)**
- [ ] **AC1** ‚Äî Given an employee with N unassigned `OrderEmployeePay` rows in a period, When the
  handler runs, Then it returns `IsSuccess`, creates exactly one `EmployeeInvoice`, and
  `SubTotal == Œ£(BasePay + ExtrasPay + ExpensesPay)`, `BonusAmount == Œ£ BonusPay`,
  `DeductionAmount == Œ£ DeductionPay`, `TotalOrders == N` ‚Äî matching the sums at
  `GenerateInvoice.cs:78-80,88-95`. Money is exact (no float drift).
- [ ] **AC2** ‚Äî Given the generated invoice, When created, Then **every** fetched `OrderEmployeePay`
  has `AssignToInvoice(invoice.Id)` applied (`GenerateInvoice.cs:99-102`) ‚Äî no order pay is left
  unassigned (prevents double-billing on a second generation).
- [ ] **AC3** ‚Äî Given `Œ£ pays` is negative (deductions exceed base+bonus), When the invoice is
  created, Then `TotalAmount` is clamped to `0`, never negative (`EmployeeInvoice.Create` :104-109).

**Invoice generation validator (`GenerateInvoice.Validator`, mocked repos) ‚Äî every error path**
- [ ] **AC4** ‚Äî empty/whitespace `EmployeeId` or `PayPeriodId` ‚Üí `BusinessErrorMessage.Required`.
- [ ] **AC5** ‚Äî unknown employee ‚Üí `EmployeeNotFound`; unknown pay period ‚Üí `PayPeriodNotFound`.
- [ ] **AC6** ‚Äî an invoice already exists for that employee+period ‚Üí `InvoiceAlreadyExists`
  (`Validator.ExistsForPayPeriodAsync`, :57-59) ‚Äî the idempotency guard against double-generation.
- [ ] **AC7** ‚Äî there are no unassigned order pays for the employee+period ‚Üí `NoUnpaidOrderPays`
  (`Validator.NoUnpaidOrderPaysExist`, :61-63).
- [ ] **AC8** ‚Äî a valid command (employee + period exist, no existing invoice, unassigned pays
  present) ‚Üí validator passes clean.

**Invoice numbering & symbols (`EmployeeInvoice`, pure)**
- [ ] **AC9** ‚Äî `Create` sets `InvoiceNumber` to the `INV-yyyyMM-XXXXX` shape and `PaymentReference`
  equal to it (`EmployeeInvoice.cs:111,117,126`); two invoices generated in the same call produce
  **distinct** `InvoiceNumber`s (the DB unique index on `InvoiceNumber` is asserted as the backstop ‚Äî
  `EmployeeInvoiceEntityConfiguration.cs:113-114`). Note in evidence that numbering is GUID-suffixed,
  not gap-free sequential.
- [ ] **AC10** ‚Äî `GenerateVariableSymbol(employeeId, payPeriodId)` is deterministic for the same
  inputs and produces a `D4`+`D6` (10-char) string (`EmployeeInvoice.cs:287-292`).

**Invoice status transitions (`EmployeeInvoice`, pure ‚Äî legal + rejected illegal)**
- [ ] **AC11** ‚Äî `Approve` succeeds from `Pending` and from `Disputed` and sets
  `Status=Approved`/`ApprovedAt`/`ApprovedBy`; from any other status it **throws**
  `InvalidOperationException` (:196-209).
- [ ] **AC12** ‚Äî `MarkAsPaid` succeeds only from `Approved` (sets `Status=Paid`/`PaidAt`); from any
  other status it **throws** (:211-228).
- [ ] **AC13** ‚Äî `Dispute`, `Reject`, `UpdateAmounts`, and `Cancel` each **throw** when `Status==Paid`
  (:230-318); `Cancel` also throws when already cancelled (:306-309).

**Pay-period close (`ClosePayPeriod` + `PayPeriod`)**
- [ ] **AC14** ‚Äî Given an `Open` period with all invoices `Paid`, When `ClosePayPeriod.Handler` runs,
  Then it returns `IsSuccess`, `Status==Closed`, `ClosedAt` set, `ClosedBy == session email`
  (`ClosePayPeriod.cs:67-73`; `PayPeriod.Close` :105-122).
- [ ] **AC15** ‚Äî Validator rejects: unknown period ‚Üí `PayPeriodNotFound`; period not `Open` ‚Üí
  `PayPeriodNotOpen` (`BeOpenStatusAsync` :50-54); a non-`Paid` invoice exists in the period ‚Üí
  `UnpaidInvoicesExist` (`AllInvoicesPaidInPeriodAsync` inverse, `EmployeeInvoiceRepository.cs:64-72`);
  `Notes` over 1000 chars ‚Üí `MaxLength`.
- [ ] **AC16** ‚Äî `PayPeriod` lifecycle guards (pure): `Close` throws unless `Open`; `MarkAsPaid`
  throws unless `Closed`; `Reopen` throws when `Paid` (:105-154).
- [ ] **AC17** ‚Äî Each test predates its assertion target in commit order and the status log records
  the **red ‚Üí green** per testing.md "How the ticket shows TDD happened". No theater asserts (a
  method merely returning non-null), no happy-path-only on the money/transition cases.
