## Context
Source: `agents/knowledge/testing.md` must-cover item **#7 — "Pay periods & invoices"** (TC-6 in the
Wave-4 test plan, `audits/AUDIT-2026-06-01-execution-plan.md`). Payroll invoice generation,
numbering, and the pay-period open/close lifecycle are money paths with **no coverage today**
(testing.md: "Coverage today is minimal"). A defect here pays an employee twice, closes a period over
unpaid work, or collides invoice numbers — all costly and hard to detect after the fact.

This ticket adds the executable evidence for that surface. The behavior under test already exists:
- `GenerateInvoice` command (handler + validator) — `Cleansia.Core.AppServices/Features/EmployeePayroll/GenerateInvoice.cs`
- `ClosePayPeriod` command (handler + validator) — `Cleansia.Core.AppServices/Features/PayPeriods/ClosePayPeriod.cs`
- `EmployeeInvoice` domain entity (Create/numbering/status transitions) — `Cleansia.Core.Domain/EmployeePayroll/EmployeeInvoice.cs`
- `PayPeriod` domain entity (Create/Close/MarkAsPaid/Reopen) — `Cleansia.Core.Domain/EmployeePayroll/PayPeriod.cs`

Depends on **T-0171** and **T-0180** (the Wave-4 invoice/pay-period fix + characterization
predecessors) so the cases pin the corrected behavior, not the pre-fix behavior. Per
testing.md, all of this is built **test-first** (these are pure logic + handler-contract cases — the
strict/contract TDD tiers).

## Acceptance criteria
Each AC is a test case (or small group) in `Cleansia.Tests`; assert on the `BusinessErrorMessage`
**constant**, never a hardcoded string; use `Cleansia.TestUtilities` builders, not inline graphs.

**Invoice generation (`GenerateInvoice.Handler`, mocked repos)**
- [ ] **AC1** — Given an employee with N unassigned `OrderEmployeePay` rows in a period, When the
  handler runs, Then it returns `IsSuccess`, creates exactly one `EmployeeInvoice`, and
  `SubTotal == Σ(BasePay + ExtrasPay + ExpensesPay)`, `BonusAmount == Σ BonusPay`,
  `DeductionAmount == Σ DeductionPay`, `TotalOrders == N` — matching the sums at
  `GenerateInvoice.cs:78-80,88-95`. Money is exact (no float drift).
- [ ] **AC2** — Given the generated invoice, When created, Then **every** fetched `OrderEmployeePay`
  has `AssignToInvoice(invoice.Id)` applied (`GenerateInvoice.cs:99-102`) — no order pay is left
  unassigned (prevents double-billing on a second generation).
- [ ] **AC3** — Given `Σ pays` is negative (deductions exceed base+bonus), When the invoice is
  created, Then `TotalAmount` is clamped to `0`, never negative (`EmployeeInvoice.Create` :104-109).

**Invoice generation validator (`GenerateInvoice.Validator`, mocked repos) — every error path**
- [ ] **AC4** — empty/whitespace `EmployeeId` or `PayPeriodId` → `BusinessErrorMessage.Required`.
- [ ] **AC5** — unknown employee → `EmployeeNotFound`; unknown pay period → `PayPeriodNotFound`.
- [ ] **AC6** — an invoice already exists for that employee+period → `InvoiceAlreadyExists`
  (`Validator.ExistsForPayPeriodAsync`, :57-59) — the idempotency guard against double-generation.
- [ ] **AC7** — there are no unassigned order pays for the employee+period → `NoUnpaidOrderPays`
  (`Validator.NoUnpaidOrderPaysExist`, :61-63).
- [ ] **AC8** — a valid command (employee + period exist, no existing invoice, unassigned pays
  present) → validator passes clean.

**Invoice numbering & symbols (`EmployeeInvoice`, pure)**
- [ ] **AC9** — `Create` sets `InvoiceNumber` to the `INV-yyyyMM-XXXXX` shape and `PaymentReference`
  equal to it (`EmployeeInvoice.cs:111,117,126`); two invoices generated in the same call produce
  **distinct** `InvoiceNumber`s (the DB unique index on `InvoiceNumber` is asserted as the backstop —
  `EmployeeInvoiceEntityConfiguration.cs:113-114`). Note in evidence that numbering is GUID-suffixed,
  not gap-free sequential.
- [ ] **AC10** — `GenerateVariableSymbol(employeeId, payPeriodId)` is deterministic for the same
  inputs and produces a `D4`+`D6` (10-char) string (`EmployeeInvoice.cs:287-292`).

**Invoice status transitions (`EmployeeInvoice`, pure — legal + rejected illegal)**
- [ ] **AC11** — `Approve` succeeds from `Pending` and from `Disputed` and sets
  `Status=Approved`/`ApprovedAt`/`ApprovedBy`; from any other status it **throws**
  `InvalidOperationException` (:196-209).
- [ ] **AC12** — `MarkAsPaid` succeeds only from `Approved` (sets `Status=Paid`/`PaidAt`); from any
  other status it **throws** (:211-228).
- [ ] **AC13** — `Dispute`, `Reject`, `UpdateAmounts`, and `Cancel` each **throw** when `Status==Paid`
  (:230-318); `Cancel` also throws when already cancelled (:306-309).

**Pay-period close (`ClosePayPeriod` + `PayPeriod`)**
- [ ] **AC14** — Given an `Open` period with all invoices `Paid`, When `ClosePayPeriod.Handler` runs,
  Then it returns `IsSuccess`, `Status==Closed`, `ClosedAt` set, `ClosedBy == session email`
  (`ClosePayPeriod.cs:67-73`; `PayPeriod.Close` :105-122).
- [ ] **AC15** — Validator rejects: unknown period → `PayPeriodNotFound`; period not `Open` →
  `PayPeriodNotOpen` (`BeOpenStatusAsync` :50-54); a non-`Paid` invoice exists in the period →
  `UnpaidInvoicesExist` (`AllInvoicesPaidInPeriodAsync` inverse, `EmployeeInvoiceRepository.cs:64-72`);
  `Notes` over 1000 chars → `MaxLength`.
- [ ] **AC16** — `PayPeriod` lifecycle guards (pure): `Close` throws unless `Open`; `MarkAsPaid`
  throws unless `Closed`; `Reopen` throws when `Paid` (:105-154).
- [ ] **AC17** — Each test predates its assertion target in commit order and the status log records
  the **red → green** per testing.md "How the ticket shows TDD happened". No theater asserts (a
  method merely returning non-null), no happy-path-only on the money/transition cases.

  methods directly. Use `Cleansia.TestUtilities` builders for `OrderEmployeePay`/`EmployeeInvoice`/

  `EmployeeNotFound`, `PayPeriodNotFound`, `InvoiceAlreadyExists`, `NoUnpaidOrderPays`,
  `PayPeriodNotOpen`, `UnpaidInvoicesExist` (:119-126), `MaxLength`.
- **Serialization cluster:** **none.** TICKET-MAP's serialization map lists no
  invoice/pay-period/`Cleansia.Tests` cluster; this ticket adds new test files only and does not edit
  any shared production file, so it can run concurrently with other Wave-4 test tickets. (Avoid
  touching the same new test class as a sibling test ticket if one is ever added — none is today.)
- **Governing ADR:** none specific (no ADR cited for TC-6). The authoritative spec is
  `agents/knowledge/testing.md` (must-cover #7 + the TDD section); follow its "match the codebase"
  and anti-patterns sections — Reviewer enforces Gate 6 against that doc.
- **Built TEST-FIRST** per `knowledge/testing.md`: pure logic (numbering, symbols, entity
  transitions, clamp) is **strict red-green-refactor**; the handler/validator cases are **test-first
  at the contract**. Because the production code already exists (T-0171/T-0180 land the fixes first),
  for any unit you are pinning rather than driving, write a **characterization test** that confirms
  current behavior first, then assert the corrected behavior — record the red→green in the status log.
- **Dependency reason:** T-0171/T-0180 are the invoice/pay-period predecessors in the Wave-4 plan;
  hold this until both are `done` so the cases encode post-fix behavior.
- **Watch-outs grounded in the code:** numbering is GUID-suffixed (not gap-free); the only gap-free
  guarantee is the DB unique index — call this out in AC9 evidence. `AllInvoicesPaidInPeriodAsync`
  returns the **inverse** (true == zero unpaid) — assert the boundary both ways.
