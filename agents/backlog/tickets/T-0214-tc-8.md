## Context
The `Cleansia.Functions` worker hosts **16 Functions** (queue consumers + timer-triggered sweeps) that
realize the platform's most sensitive side effects — receipt PDFs + fiscal email, pay calculation,
invoice generation, push fan-out, recurring-booking materialization, cleanup/retry sweeps. Today these
bodies are effectively **untested**: `Cleansia.Functions.csproj:8` is `OutputType=Exe` and
`Cleansia.Tests.csproj:25-26` does not reference it, so the consumer logic cannot even be instantiated
from a test (verified in ADR-0002 §"Lead re-verification" and Test-C1, BLOCKING). `testing.md` puts
fiscal-mode routing (must-cover #4), idempotency / "safe to run twice" (#6), and pay periods/invoices
(#7) — all of which live in these Functions — on the non-negotiable PROD list.

This ticket is the **TC-8** slice from Wave 4 (`TICKET-MAP.md` line 117). It writes the per-Function
test suite that exercises the behavior frozen by **ADR-0002** (post-commit dispatch contract,
idempotent consumers, poison/dead-letter floor, failure classification). It is **test-first per
`knowledge/testing.md`** but lands **after** the testability extraction and the consumer fixes it
verifies: it depends on **T-0121** (FUNC-CORE — extract `Cleansia.Functions.Core` as a non-Exe class
library so the Tests project can reference the consumer bodies; ADR-0002 D5 step 1) and **T-0143**
(the F2/F3/F4 consumer-guard + poison + dual-read + reconciliation work whose behavior these tests
assert). It does **not** re-implement TC-IDEMP-0 / TC-POISON-0 / TC-CLASSIFY-0 / TC-DISPATCH-0
(those are Wave-0 slices landing with their fixes) — it is the **broad coverage net across all 16
Functions** that those targeted suites do not individually cover.

## Acceptance criteria
- [ ] **AC1 — Every Function is reachable from a test.** Given `Cleansia.Functions.Core` exists
  (T-0121), When `Cleansia.Tests` references it, Then each of the 16 Function classes
  (`GenerateReceiptFunction`, `GenerateInvoiceFunction`, `SendPushNotificationFunction`,
  `CalculateOrderPayFunction`, `SendSitewidePromoFanoutFunction`, `DataRetentionTimerFunction`,
  `PeriodReminderTimerFunction`, `RetryFailedFiscalRegistrationsFunction`,
  `AutoCancelStaleRecurringOrdersFunction`, `CleanupStalePendingOrdersFunction`,
  `MaterializeRecurringBookingsFunction`, `RefreshTokenCleanupTimerFunction`,
  `SendMembershipLifecycleNotificationsFunction`, `SendRecurringOrderRemindersFunction`,
  `PayPeriodTimerFunction`, `SendNewJobsDigestTimerFunction`) is constructed with mocked dependencies
  and its trigger method (`Run`) invoked in at least one test.
- [ ] **AC2 — `GenerateReceiptFunction` happy path + each branch.** Given an eligible cash/paid order
  with no receipt, When `Run` is invoked, Then `IReceiptService.GenerateReceiptAsync` and
  `IEmailService.SendOrderReceiptEmailAsync` are each called once and the receipt is marked email-sent
  (`GenerateReceiptFunction.cs:72,95,97`). Cover the rejection branches as separate cases: invalid
  `OrderId` format → ack/no-throw (`:36-40`); order not found → **throws** (transient,
  fiscal-queue carve-out per ADR-0002 D3.3 — `:48-52`, MUST NOT be reclassified to ack); ineligible
  `PaymentType`/`PaymentStatus` → ack (`:59-64`); `order.Receipt is not null` → ack, no generate
  (`:66-70`); BlockingOnline country with no `FiscalCode` → email **held**, commit, no send (`:83-90`).
- [ ] **AC3 — `GenerateReceiptFunction` is safe to run twice (terminal effect = email).** Given the
  same receipt message delivered twice, When `Run` runs twice, Then
  `IEmailService.SendOrderReceiptEmailAsync` fires **exactly once** (closes the C6 email re-send gap
  named in ADR-0002 D2.2). This aligns with TC-IDEMP-0 but is asserted here for the full-coverage net.
- [ ] **AC4 — `SendPushNotificationFunction` idempotency + failure classification.** Given a
  notifications-dispatch message, When `Run` runs twice with the same key, Then
  `IPushDispatcher.SendAsync` is called **once** (guard-first / at-most-once-after-marker, ADR-0002
  D2.2) and the second run short-circuits on the `IIdempotencyGuard` claim. And: a **malformed body**
  (missing `UserId`/`EventKey`) is **acked, no throw** (`SendPushNotificationFunction.cs:45-51`); a
  **muted category** acks with no send (`:67-74`); a **simulated infra fault** during dispatch
  **throws** so the queue retries (per ADR-0002 D3.3 — this is the fix to the old throw-on-everything
  at `:115-121`).
- [ ] **AC5 — `CalculateOrderPayFunction` ack-on-reject vs throw-on-infra.** Given a message, When the
  inner `CalculateOrderPay.Command` returns a validation failure (already-calculated, missing config),
  Then `Run` logs warning and **does not throw** (`CalculateOrderPayFunction.cs:55-65`); When it
  succeeds, Then the pay-row effect is asserted once; When `EnsureOpenPeriodAsync`/`mediator.Send`
  raises an infra exception, Then `Run` **throws**.
- [ ] **AC6 — `GenerateInvoiceFunction` stub behavior is pinned (characterization).** Given the
  current stub (`GenerateInvoiceFunction.cs:20-26`), When `Run` is invoked, Then it deserializes,
  logs "not yet implemented", and completes with **no effect** (no invoice created) — characterization
  test per `testing.md` §"changing existing untested code", so its future replacement (T-tbd / F1)
  cannot silently change behavior. It is **excluded** from the idempotency assertion (ADR-0002 D2.2 —
  no effect to dedup).
- [ ] **AC7 — `SendSitewidePromoFanoutFunction` is a producer, not an effect-realizer.** Given a
  fan-out message, When `Run` pages users, Then it enqueues N notifications-dispatch messages via
  direct `IQueueClient.SendAsync` (D2.3 carve-out) and a per-recipient enqueue failure is caught and
  the loop continues (`:137-146`); the test asserts **no inbound idempotency guard is required** and
  one bad recipient does not abort the campaign. Excluded from the "safe twice" assertion.
- [ ] **AC8 — Timer/sweep Functions exercise their happy path + an empty/no-op path.** For each of the
  9 timer-triggered functions (`DataRetention`, `PeriodReminder`, `RetryFailedFiscalRegistrations`,
  `AutoCancelStaleRecurringOrders`, `CleanupStalePendingOrders`, `MaterializeRecurringBookings`,
  `RefreshTokenCleanup`, `SendMembershipLifecycleNotifications`/`SendRecurringOrderReminders`,
  `PayPeriodTimer`, `SendNewJobsDigest`), assert: nothing-due → no side effect; due-rows → the
  expected service method is invoked. (Behavior depth scoped to what the underlying service exposes —
  these are sweeps, not money/lifecycle math.)
- [ ] **AC9 — No theater.** Every test asserts an observable behavior (a mock invocation count, a
  thrown/not-thrown outcome, a domain state), not merely that a method returns non-null or exists
  (`testing.md` §Anti-patterns). Tests assert on `BusinessErrorMessage` constants where an error code
  is checked, never on hardcoded strings or log text.
- [ ] **AC10 — Suite is green and runs in CI.** `dotnet test src/Cleansia.Tests` passes including the
  new Function tests; builders from `Cleansia.TestUtilities` are used for entity graphs (no
  hand-rolled inline graphs).

## Implementation notes
- **Built TEST-FIRST per `knowledge/testing.md`.** For each Function the test encodes the desired
  branch behavior from ADR-0002 / the existing code, fails for the right reason, then is satisfied by
  the (already-merged via deps) consumer code. For the stub (AC6) and untested sweeps (AC8) these are
  **characterization tests** that pin current behavior first (testing.md §"changing existing untested
  code"). The status log must note the red→green order; tests appear with/before any code touched.
- **Governing ADR: ADR-0002** (`agents/backlog/adr/0002-outbox-dispatch-contract.md`). The
  consumer-side obligations being asserted: D2.2 (idempotent terminal effect: receipt email, push via
  `IIdempotencyGuard`), D2.3 (fan-out producer carve-out), D3.3 (permanent=ack / transient=throw, with
  the **fiscal target-not-found-stays-transient** carve-out at `GenerateReceiptFunction.cs:48-52` —
  reviewers must NOT "simplify" that throw to an ack). ADR-0002 verification checks #5/#9 (TC-IDEMP-0)
  overlap AC3/AC4; this ticket re-asserts them within full-Function coverage.
- **Dependency gating:** hold until **T-0121 (FUNC-CORE)** is `done` — the Tests project cannot
  reference the consumers until they live in a non-Exe `Cleansia.Functions.Core` library
  (`Cleansia.Functions.csproj:8` is `OutputType=Exe`; `Cleansia.Tests.csproj:25-26` does not reference
  Functions). Also hold until **T-0143** (the F2/F3/F4 consumer guards + poison + dual-read +
  reconciliation) is `done` — AC3/AC4/AC5 assert behavior those fixes introduce; running before them
  would test the broken state. Both deps must be `done` before this goes `ready`.
- **Test locations / shape:** `Cleansia.Tests` (xUnit unit). Construct each Function with mocked
  `IXxxRepository`/service deps (Moq), invoke `Run`, assert mock invocation counts and thrown/
  not-thrown outcomes. Use `Cleansia.TestUtilities` builders for `Order`/`PayPeriod`/`Device` graphs.
- **Serialization cluster:** this ticket touches **only test files under `Cleansia.Tests`** plus the
  `Cleansia.Tests.csproj` reference add — it is **not** in any shared-file serialization cluster in
  `TICKET-MAP.md` (those clusters are production source: `PolicyBuilder.cs`, `CleansiaStartupBase.cs`,
  `UnitOfWorkPipelineBehavior.cs`, host `ServiceExtensions.cs`, `CreateOrder.cs`, `LoyaltyService.cs`).
  The only coordination is the `depends_on` gate above. If the csproj reference add collides with
  another concurrent test ticket editing `Cleansia.Tests.csproj`, serialize that single edit.
- **Sizing:** marked `L` in the source. **Must be split before it goes `ready`** (PM constraint — no
  `L` runs). Suggested split into runnable `M`/`S` children by surface, e.g.: (a) the two fiscal
  consumers — `GenerateReceiptFunction` + `GenerateInvoiceFunction` stub (AC2/AC3/AC6); (b)
  notifications + pay — `SendPushNotificationFunction` + `CalculateOrderPayFunction` +
  `SendSitewidePromoFanoutFunction` (AC4/AC5/AC7); (c) the 9 timer/sweep Functions (AC1/AC8); with
  AC9/AC10 applying to each child. Each child keeps the reviewer-per-developer invariant.
