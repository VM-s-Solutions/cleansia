## Context
The platform is multi-tenant: tenant-scoped rows carry a `TenantId` and reads are auto-scoped by an
EF Core global query filter built in `Cleansia.Infra.Database/CleansiaDbContext.cs:108-174` from
`ITenantProvider.GetCurrentTenantId()`, which resolves the `tenant_id` JWT claim
(`Cleansia.Infra.Database/TenantProvider.cs:8,18-19`). Per `knowledge/testing.md` must-cover #5
(S2/S3), the cross-user and cross-tenant boundary on resource-by-id **write** endpoints is "a test,
not just a code review" — the rejection path must return **NotFound** (the resource must not leak),
not 403-with-body and not the row. Today this is asserted only by inspection: there is **no**
integration coverage. The `Cleansia.IntegrationTests` project named in `knowledge/testing.md` does
not yet exist on disk (only `Cleansia.Tests` unit specs are present) — the harness/fixture stand-up
is the work of the dependency tickets T-0100 and T-0126; this ticket consumes that harness to add the
authz/cross-tenant write-path suite.

This is the Wave-4 integration counterpart to the Wave-0 unit slice TC-AUTHZ-0; it proves the same
boundary end-to-end (controller → MediatR → validator → handler → DB), through the real tenant
filter and the real authorization policies governed by **ADR-0001 (authz)**.

Source: `knowledge/testing.md` #4 (must-cover #5, "Authorization & ownership boundaries").

## Acceptance criteria
- [ ] **AC1** — Given a tenant-scoped resource owned by tenant A (e.g. an `Order`), When a caller
      authenticated with a `tenant_id` claim for tenant B invokes a write endpoint against that
      resource by id, Then the response is **404 NotFound** and the resource is unmodified in the DB
      (the global query filter never surfaces it to tenant B). A passing integration test exists in
      `Cleansia.IntegrationTests` asserting the status and the unchanged row.
- [ ] **AC2** — Given a resource owned by **user X** within one tenant, When **user Y** (same tenant,
      non-elevated) invokes a write endpoint against it by id, Then the response is **404 NotFound**
      (cross-user ownership rejection), proven by an integration test — not just the cross-tenant case.
- [ ] **AC3** — The suite covers the highest-risk tenant-scoped write paths across hosts, at minimum:
      an `Order` lifecycle write (e.g. `TakeOrder` /
      `Cleansia.Core.AppServices/Features/Orders/TakeOrder.cs`), a `Dispute` write (CreateDispute /
      add-message — SEC-DSP path), a membership/loyalty write (`UserMembership`), and a saved-address
      write (`SavedAddress`) — each with a cross-tenant case (AC1) and, where the path is
      user-owned, a cross-user case (AC2).
- [ ] **AC4** — Each test authenticates through the real host pipeline (issues/attaches a JWT with the
      `tenant_id` claim per `TenantProvider.TenantClaimType`) rather than stubbing `ITenantProvider`,
      so the claim→filter wiring is exercised, and the in-tenant happy path for the same endpoint
      returns success (proving the rejection is the boundary, not a broken endpoint).
- [ ] **AC5** — Tests are written **test-first** per `knowledge/testing.md`: each rejection case is
      red before the assertion logic and the status log records "red → green"; every case asserts on
      the `BusinessErrorMessage` constant / HTTP status, never a hardcoded string (anti-patterns list).

## Implementation notes
- **Built TEST-FIRST** per `knowledge/testing.md` (integration: "spin the Web host, authenticate as
  the relevant role, hit the route, assert the HTTP status + body; cover the auth/ownership rejection
  explicitly"). Use the builders in `Cleansia.TestUtilities`; do not hand-roll entity graphs inline.
- **Governing ADR:** ADR-0001 (authorization) defines the policy/ownership model these tests assert
  against. Read it before encoding expectations.
- **Tenant mechanics to assert against (cite):** filter composition lives in
  `CleansiaDbContext.cs:108-174` — note the single-tenant escape (`currentTenantId == null &&
  e.TenantId == null`, lines 149-156); tests must set a **non-null** `tenant_id` claim on both
  tenants so the multi-tenant `tenantMatch` branch (line 159) is the one under test, not the
  single-tenant fallback. The claim type is `TenantProvider.TenantClaimType` ("tenant_id",
  `TenantProvider.cs:8`).
- **Serialization cluster:** TC-9 does NOT belong to a shared-file serialization cluster in
  `TICKET-MAP.md` — it adds new test files under `Cleansia.IntegrationTests` and edits no production
  source, so it can run concurrently with non-test tickets. The related authz fixes (BSP-1 →
  IDA-SEC-04 → SEC-DSP-01/02 → SEC-EMP-01) are the serialized `PolicyBuilder.cs` cluster; this ticket
  only *reads* their behavior. Per the reviewer-per-developer invariant, run a reviewer instance
  concurrently on this ticket.
- Cover at least one endpoint per host that owns tenant-scoped writes (Partner :5000, Admin :5001,
  Mobile :5002, Customer :5003) so the host-level auth registration (ADR-0001 D4) is exercised, not
  just one API.

### Security verdict (2026-06-13) — PASS-WITH-NOTES

Adversarial review of the cross-tenant + cross-user WRITE-path boundary tests. Verified the tests
would CATCH a broken isolation implementation, not merely pass against shipped code.

**Both directions proven, both fixtures NON-NULL distinct tenant claims.** `TestJwtFactory` emits a
real signed `tenant_id` claim; both fixtures use `tenant-A`/`tenant-B` (non-null, distinct), so the
multi-tenant `tenantMatch` branch (`CleansiaDbContext.cs:218`) is under test, never the single-tenant
null/null escape (`:213-215`). Confirmed the role on every token matches the endpoint `[Permission]`,
so the caller clears the policy layer and the rejection comes from the tenant filter / ownership gate
(not a 403 short-circuit). `HttpAssert.RejectedAsync` lands on the 400 branch and asserts the
`BusinessErrorMessage` constant.

**Non-vacuity — the load-bearing property.** Each cross-TENANT test sets the foreign-tenant caller's
`sub` to the GENUINE owner, so the ownership gate alone would NOT reject — the tenant filter is the
SOLE rejection mechanism. Traced each path: if the global query filter were removed, every
cross-tenant test flips to 200 + mutated row (SavedAddress label changes; dispute message appended;
membership resolved → Stripe; order taken). The tests assert the rejection constant AND the unchanged
row, so each genuinely catches a removed filter. Cross-USER non-vacuity confirmed on the genuine
resource-by-id paths: Ac11 SavedAddress (same tenant, owner's id) rejects solely via
`BeOwnedByCallerAsync`; Ac14 StartOrder rejects solely via `EmployeeIsAssignedToOrderAsync`. Removing
those ownership checks flips the tests.

**Per-host coverage.** Customer host (SavedAddress, Dispute add-message, Membership) and Mobile
partner host (Order Take/Start) — the 4th host the ticket requires beyond Ac9's Partner-host case.
Partner/Admin cross-tenant writes are locked by the pre-existing Ac9 (not re-proven — correct dedup).

**Verification achieved:** built `Cleansia.HostTests.csproj` (`-p:BuildProjectReferences=false`,
0 warn/0 err) and RAN against real Testcontainers Postgres: the 4 new classes 13/13 green; full
HostTests suite 51/51 green (no Ac1–Ac10 regression from the additive `DomainSeed`/`HttpAssert`
changes). Zero production-source edits confirmed.

**Notes (not blocking):**
- The Ac13 cross-USER membership case (`Cross_user_same_tenant_cancel...`) is degenerate: the outsider
  has no membership, so `GetActiveForUserAsync(outsiderId)` returns null regardless of the tenant
  filter OR ownership — this leg would pass even with all isolation removed. The endpoint resolves
  purely from the subject (no resource-by-id), so a true cross-user IDOR is structurally
  impossible here; the developer documented this honestly in the class summary. The membership TENANT
  boundary remains non-vacuously proven by the cross-tenant leg, so isolation is still locked. Accept.
- `HttpAssert` tokenization (the `;`-joined multi-rule match) only matches the expected constant as an
  exact token of the production error value — it cannot mask a wrong rejection code. Acceptable.
- Production finding `StartOrder.cs:137` (handler NPE on the legitimate happy path) is correctly
  reported, not patched — out of scope per the ticket. The Mobile success leg via TakeOrder is a sound
  substitute; the cross-user StartOrder REJECTION is reached before that handler line, so it is exercised.
