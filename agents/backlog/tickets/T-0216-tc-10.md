## Context
Source: **TC-10** (Wave 4) — `agents/backlog/TICKET-MAP.md` Wave 4 line ("TC-10 (fiscal-mode
characterization)") + must-cover item **#4** in `agents/knowledge/testing.md`:

> **Fiscal enforcement modes** — `None` / `AsyncBackground` / `BlockingOnline` route correctly per
> `CountryConfiguration`; **customer completion is never blocked** by fiscal registration; failed
> registrations are retried.

Fiscal-mode selection is untested today and is legally load-bearing: it decides whether a receipt is
delivered immediately, delivered before the authority signs, or **held** until the authority signs.
Per `agents/knowledge/testing.md` "When you're changing existing untested code" and the strict-TDD
rule for "fiscal-mode selection", we pin the **current** observable behavior of the mode branches as a
**characterization** net so the Wave-1/2 fiscal durability work (F2-FULL, F6, FISCAL-RECON) can be
proven behavior-preserving rather than silently breaking a branch.

Units under characterization (no production code is modified by this ticket):
- The per-country mode source: `CountryConfiguration.FiscalEnforcementMode`
  (`src/Cleansia.Core.Domain/Configuration/CountryConfiguration.cs:71`, default `None`; values in
  `src/Cleansia.Core.Fiscal.Abstractions/FiscalEnforcementMode.cs`).
- The mode branch in receipt generation: `ReceiptService.HandleFiscalAsync`
  (`src/Cleansia.Core.AppServices/Services/ReceiptService.cs:98-161`) and its call site
  (`ReceiptService.cs:60-89`).
- The mode-gated email hold in the consumer: `GenerateReceiptFunction.Run`
  (`src/Cleansia.Functions/Functions/GenerateReceiptFunction.cs:79-99`,
  `ResolveEnforcementModeAsync` `:113-125`).
- The mode-gated retry/release: `FiscalRetryService.ProcessDueRetriesAsync`
  (`src/Cleansia.Core.AppServices/Services/FiscalRetryService.cs:59-77`,
  `ResolveEnforcementModeAsync` `:96-108`).

## Acceptance criteria
- [ ] **AC1 — `None` skips fiscal entirely.** Given a receipt whose order resolves to a country with
  `FiscalEnforcementMode.None` (or no `CountryConfiguration` row, i.e. the default), When
  `GenerateReceiptAsync` runs, Then `IFiscalServiceResolver.Resolve` is **never** called, no
  `FiscalReceiptRequest` is built, and the returned `OrderReceipt.FiscalCode` is `null`.
  (`ReceiptService.cs:62-73,107-110`)
- [ ] **AC2 — `AsyncBackground` registers once and never blocks delivery on success.** Given a country
  config of `AsyncBackground` and a fiscal service that returns `IsRegistered=true` with a `FiscalCode`,
  When `GenerateReceiptAsync` runs, Then `RegisterReceiptAsync` is called exactly once,
  `receipt.SetFiscalData(...)` is applied (`FiscalCode`/`FiscalRegisteredAt`/`FiscalProviderKey`
  populated), and control returns normally so the caller can deliver. (`ReceiptService.cs:130-140`)
- [ ] **AC3 — `AsyncBackground` lenient on failure (delivery still proceeds).** Given `AsyncBackground`
  and a fiscal service that returns `IsRequired=true, IsRegistered=false` (or throws), When
  `GenerateReceiptAsync` runs, Then `receipt.MarkFiscalRegistrationFailed(...)` is recorded,
  `FiscalCode` stays `null`, **no exception propagates** out of `GenerateReceiptAsync`, and the receipt
  is returned (the customer is not blocked). (`ReceiptService.cs:141-160`)
- [ ] **AC4 — `BlockingOnline` still attempts once and tolerates authority failure.** Given
  `BlockingOnline` and a fiscal service that throws/returns not-registered, When
  `GenerateReceiptAsync` runs, Then exactly one `RegisterReceiptAsync` attempt is made, the failure is
  marked, no exception escapes, and `FiscalCode` is `null`. (`ReceiptService.cs:130-160`)
- [ ] **AC5 — Consumer holds the email for blocking modes when unsigned.** Given
  `GenerateReceiptFunction.Run` for an order whose country is `BlockingOnline` (and
  `BlockingWithOfflineCache`) where the generated receipt has `FiscalCode == null`, When the function
  runs, Then `IEmailService.SendOrderReceiptEmailAsync` is **not** called, `receipt.MarkEmailSent` is
  not invoked, `unitOfWork.CommitAsync` is still called, and the function returns without throwing.
  (`GenerateReceiptFunction.cs:79-90`)
- [ ] **AC6 — Consumer sends immediately for non-blocking / signed.** Given the same function for a
  `None`/`AsyncBackground` country, OR a `BlockingOnline` country where `FiscalCode != null`, When it
  runs, Then `SendOrderReceiptEmailAsync` is called exactly once and `receipt.MarkEmailSent(messageId)`
  is recorded before commit. (`GenerateReceiptFunction.cs:80-99`)
- [ ] **AC7 — Retry releases the held email only for blocking modes.** Given
  `FiscalRetryService.ProcessDueRetriesAsync` over a due receipt where
  `RetryFiscalRegistrationAsync` returns `true` and `receipt.EmailSent == false`: when the order's
  country is `BlockingOnline`/`BlockingWithOfflineCache`, Then the held email is sent and
  `MarkEmailSent` is recorded; when the country is `None`/`AsyncBackground`, Then **no** release email
  is sent (it was already delivered up front). (`FiscalRetryService.cs:63-77`)
- [ ] **AC8 — Mode resolution falls back to `None`.** Given an order with a null
  `CustomerAddress.CountryId`, OR a country with no `CountryConfiguration` row, When
  `ResolveEnforcementModeAsync` runs (in both `GenerateReceiptFunction` and `FiscalRetryService`),
  Then it returns `FiscalEnforcementMode.None`. (`GenerateReceiptFunction.cs:117-124`,
  `FiscalRetryService.cs:100-107`)
- [ ] **AC9 — Tests are characterization, green, and test-first.** All cases live in `Cleansia.Tests`
  (and `Cleansia.IntegrationTests` only if a unit harness cannot exercise the function), use
  `Cleansia.TestUtilities` builders for the order/receipt/country-config graphs (no inline graphs),
  assert against the `BusinessErrorMessage`/enum **constants** (never hardcoded strings), run green
  against the **current** code, and the status log records "red → green". No production fiscal code is
  modified by this ticket.
