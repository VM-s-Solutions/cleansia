## Context
The customer app silently leaks raw machine keys to users on real failures. The HTTP error
interceptor resolves **every** server error through the `api.${errorKey}` path
(`libs/core/services/src/lib/interceptors/http-error.interceptor.ts:29,41`, where `errorKey` is the
`BusinessErrorMessage` dot-value, e.g. `order.cancellation_window_closed`), but the customer app's
`api` block contains only three subtrees â€” `common`, `address`, `service_area`
(`apps/cleansia.app/src/assets/i18n/en.json:1302-1316`). Any customer-reachable 4xx outside those
shows literal text like `api.order.cancellation_window_closed`. The order-cancellation flow (a money
path) is the worst hit.

This ticket closes three audit findings on one axis (error-contract parity, distinct from the
already-clean locale-key parity):
- **EP-1** â€” customer app has only 7 `api.*` keys; ~all interceptor-surfaced customer errors render as raw keys.
- **EP-2** â€” 54 backend keys have no translation anywhere; the customer-reachable subset is the part this ticket owns.
- **DA-7** â€” no `errors.*`/`api.*` entries exist for any dispute/address/file backend code in any of the 5 customer locales.

Findings: `agents/backlog/audits/AUDIT-2026-06-01-findings.json` (EP-1 lines 640-653, EP-2 lines
654-667, DA-7 lines 2446-2460); narrative in `agents/backlog/audits/AUDIT-2026-06-01-slice-reports.md`
(EP-1/EP-2 lines 722-744, DA-7 lines 2100-2103). The governing rule is `agents/knowledge/conventions.md:52`
("Every backend error key has a matching frontend translation in all 5 locales").
