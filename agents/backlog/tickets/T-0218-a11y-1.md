## Context
The blindspots audit names **accessibility as "entirely absent across the 3 Angular apps"**
(`agents/backlog/audits/AUDIT-2026-06-01-blindspots.md` §B, line 56) and prioritizes it as
follow-up pass **§D item 8**: "Accessibility pass — the `cleansia-*` library + the order wizard
across all 3 apps" (line 81). This ticket closes the highest-impact holes on the two surfaces the
audit calls out: the shared `cleansia-*` component primitives (reused by all three apps) and the
customer order wizard (the conversion-critical, keyboard-hostile path).

Grounded gaps in real code:
- **Clickable `<div>`/non-button elements with no keyboard affordance or ARIA state** in the wizard:
  stepper items (`order-wizard.component.html:29-45`), service/package selection cards
  (`:119-147`, `:218-251`), saved-address radio rows (`:335-364`), payment-method cards
  (`:546-586`), and the mobile price-bar header (`:732-745`) all carry `(click)` but have no
  `role`, `tabindex`, keyboard handler, or `aria-pressed`/`aria-selected`/`aria-current`. They are
  unreachable and unoperable by keyboard and unannounced by screen readers.
- **Form labels not programmatically associated** with their inputs — contact fields use bare
  `<label>` next to `pInputText` with no `for`/`id` pairing (`:267-322`), and the saved-address
  label field (`:415-422`).
- **Inline field errors not linked** to their inputs — `<small class="…__field-error">` is rendered
  (`:277-279` etc.) but inputs carry no `aria-invalid` / `aria-describedby`, so the error text is
  never announced.
- **Icon-only buttons with no accessible name** — the room/bathroom counter +/- buttons
  (`:165-203`) and the shared `cleansia-button` icon-only mode
  (`cleansia-button.component.html`) render an icon with no `aria-label`. The convention already
  exists in the library: `cleansia-scroll-top` is invoked with `[ariaLabel]` (`:805`), so the fix
  is to extend the same pattern.

## Acceptance criteria
- [ ] **AC1 — keyboard operability of click-only controls.** Given a keyboard-only user in the
  order wizard, When they Tab through step 0–3, Then every interactive element that currently has
  `(click)` only — stepper items, service cards, package cards, saved-address rows, payment cards,
  mobile price header — is focusable (`tabindex`/native control), shows a visible focus ring, and
  activates on **Enter and Space** with the same effect as a mouse click.
- [ ] **AC2 — ARIA state on selectable controls.** Given a screen reader, When focus lands on a
  selection card / payment card / saved-address row / stepper item, Then its selected/current state
  is exposed (`aria-pressed`, `aria-selected`, or `aria-current="step"` as appropriate) and its
  accessible name is the visible text — verified by the spec asserting the rendered attributes.
- [ ] **AC3 — label association.** Given the contact step (`:267-322`) and the save-address label
  field (`:415-422`), When rendered, Then every `<label>` is tied to its control via `for`/`id`
  (or wrapping), so clicking the label focuses the input and the input has an accessible name.
- [ ] **AC4 — error announcement.** Given a wizard field fails validation, When the error `<small>`
  renders, Then the input gets `aria-invalid="true"` and `aria-describedby` pointing at the error
  element's `id`, so the message is announced. Given the field becomes valid, Then `aria-invalid`
  clears.
- [ ] **AC5 — icon-only buttons have names.** Given the room/bathroom counter buttons (`:165-203`)
  and any icon-only `cleansia-button`, When rendered, Then each exposes an `aria-label` (sourced
  from a translation key present in all 5 locales). `cleansia-button` gains an `ariaLabel` input and
  emits `aria-label` on the underlying `p-button` when icon-only — mirroring the existing
  `cleansia-scroll-top [ariaLabel]` convention.
- [ ] **AC6 — tests prove the holes are closed.** Jest component specs assert the above on the
  wizard step components and on the touched `cleansia-*` primitives (`cleansia-button` at minimum):
  focusability, Enter/Space activation, the ARIA attributes, label/error association. Specs were
  written **before** the fix and fail against current markup (red→green per `knowledge/testing.md`).

## Out of scope
- A full WCAG 2.1 AA audit of all 3 apps end to end — this pass is scoped by §D8 to the shared
  `cleansia-*` library and the order wizard only. Other features get their own a11y tickets later.
- Visual/color-contrast theme changes and the email-template contrast work (that is BUG-22).
- Admin/partner feature pages beyond the shared primitives they consume.
- Any backend, DTO, or i18n-namespace restructuring. New aria-label strings are added under
  existing namespaces; no `nswag-regen` and no migration.
- Automated axe-core CI integration (propose as a follow-up, do not build here).

## Implementation notes
- **Built TEST-FIRST per `agents/knowledge/testing.md`** — UI is "pragmatic test-alongside", but the
  observable a11y contract here (focusability, Enter/Space activation, ARIA attributes, label/error
  wiring) is assertable logic: write the Jest specs first, confirm they fail against current markup
  for the right reason, then make them pass. The status log must show red→green and each AC must map
  to a case.
- **No governing ADR.** The three accepted ADRs (0001 authorization, 0002 outbox, 0003 rate-limiting)
  do not touch frontend a11y; `adrs: []`. Follow `agents/knowledge/patterns-frontend.md` +
  `conventions.md` — reuse existing `cleansia-*` primitives, never raw controls, OnPush + signals,
  every new string via `TranslatePipe`/`TranslateService` with keys added to **all 5 locales**
  (`apps/<app>/src/assets/i18n/{en,cs,sk,uk,ru}.json`).
- **Serialization cluster:** `cleansia-button.component.{html,ts}` is a shared library primitive
  reused across all 3 apps and many features — this ticket is the sole concurrent editor of the
  touched `cleansia-*` primitives and the order-wizard files for this wave. TICKET-MAP.md lists no
  named cluster for these files; the PM must NOT spawn another frontend ticket editing
  `libs/shared/components/**` or `libs/cleansia-customer-features/order-wizard/**` in parallel —
  serialize any such ticket after this one to avoid a shared-file collision.
- **Files to touch (known):**
  - `libs/cleansia-customer-features/order-wizard/src/lib/order-wizard/order-wizard.component.{html,ts}`
  - `libs/cleansia-customer-features/order-wizard/src/lib/order-wizard/components/wizard-summary-step.component.{html,ts}`
  - `libs/shared/components/src/lib/cleansia-button/cleansia-button.component.{html,ts}`
  - i18n JSON for the customer app (new `aria.*`-style keys) in all 5 locales.
- **Convention to mirror:** `cleansia-scroll-top [ariaLabel]` (`order-wizard.component.html:805`) is
  the existing accessible-name pattern — extend `cleansia-button` the same way rather than inventing
  a new mechanism. Prefer converting click-only `<div>`s to native `<button type="button">` where
  layout allows (cleanest a11y) over bolting `role`/`tabindex`/keydown onto a `<div>`.
- **No owner-only steps:** `manual_steps: []`, `security_touching: false`. Flag none.
